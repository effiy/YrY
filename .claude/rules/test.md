# Test Rules

## 测试先行（Gate A）

- Gate A 未过不得编码（单行 CSS 可跳过）
- Gate A 验证：测试方案 + 原型
- 每个 AC 必须有对应测试方法

## 分级标准

| 等级 | 含义 | 处置 |
|------|------|------|
| P0 | 阻塞发布：功能缺失、安全漏洞、数据丢失 | 必须修，清零前进 |
| P1 | 建议修复：边界情况、UX 退化、性能回退 | 评估后决定，记录不阻塞 |
| P2 | 可选优化：命名、注释、重构 | 记录不阻塞 |

## 测试类型

| 类型 | 阶段 | 内容 |
|------|------|------|
| 功能测试 | Gate A | AC 逐条验证，happy path |
| 边界测试 | Gate A | 输入边界、状态边界、空态 |
| 异常测试 | Gate B | 错误输入、网络异常、超时 |
| 回归测试 | Gate B | 影响链下游全覆盖 |

## Gate B 验证

1. 环境快照 → 2. 静态预检 → 3. 对齐检查 → 4. 单次执行
- >2 轮修复未通过 → H7 阻断
- 验证报告含实际命令 + 输出（A 级证据）

## fix 模式

仅对修改的函数/模块写测试，验证仅冒烟验证修改点。

## 覆盖要求

- §1.1 每个故事至少一条主操作流
- 每个用户操作至少一条交互路径
- 涉及 UI 的故事：视图状态覆盖（正常/加载/空/错误/禁用）
