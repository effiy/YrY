# Code Rules

## 最少代码

- 只解决当前问题，不写"万一用到"的功能
- 不创建单次使用抽象（一个函数只被调用一次 → 内联）
- 不处理不可能发生的错误（内部信任边界内）
- 200 行能写成 50 行就重写
- 自问：「高级工程师看了会说过度设计吗？」→ 会就简化

## 精确修改

- 只动必须动的文件，匹配现有风格
- 不动相邻代码、注释、格式
- 引入的 orphan（import/变量/函数/类型）必须清除
- 每行改动可追溯到用户请求

## 模块原则

- 优先级：复用现有 > 修改现有 > 新增
- 接口变更前证明所有调用方已覆盖
- 删除/重命名公共接口前全项目搜索 + 二级传递
- 不创建设计文档外的文件

## 分支管理

- 功能分支必须从 main/master 创建
- 分支命名：`feat/<story-name>`（代码）或 `docs/<story-name>`（文档）
- 未从主分支创建 → 阻断，重建分支

## 禁止项

- 预留结构（"以后可能用到"的参数、接口、抽象）
- 无来源的模块名/接口/路径（必须可追溯到 §3 Design）
- 硬编码第三方 URL 无 integrity hash
- 提交 secrets/tokens/credentials
