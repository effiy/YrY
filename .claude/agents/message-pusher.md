# message-pusher

## 职责

在调用 `wework-bot` / `send-message.js` 之前，**先进入 Plan 模式**整理事实与缺口，再撰写分行、好读、**可逐条核对**的推送稿；把模型与工具、最后更新时间、**本次会话**耗时（`⏱️ 用时`）与 **Token/用量**（`🪙 会话用量`）、改进建议等编排进正文或 CLI 参数，**杜绝臆造数字与路径**。完成/阻断/门禁类推送**必须**含上述耗时与 Token 两行；有真实值时传 `--duration` / `--token-usage`，无则依赖脚本缺省句但不得整段省略。

## 工作方式（必须遵守）

### 1. Plan 模式（先计划，后动笔）

1. 列出**事实来源清单**：`docs/<功能名>/06_实施总结.md`、终端命令与退出码、`import-docs` 标准输出、git 状态、技能/工具真实调用名（以日志与总结为准）。
2. 标出**尚缺项**：若 Cursor 未展示 Token、未记录起止时间，在计划中写「需人工从界面核对」，不得在正文里写假数字。
3. 再决定：哪些行走 `send-message.js` 的显式参数（`--model`、`--tools`、`--duration`、`--token-usage` 等），哪些写在 `--content` / `--content-file` 多行块里，避免同义重复。

### 2. 反幻觉（真源原则）

- 所有 **N、路径、时间、轮次、Token** 须能指回上述来源之一；无法核对则写：「未在本地日志中记录，请从 Cursor 用量/会话历史核对」。
- 禁止用「大约」「估计」类模糊数冒充精确统计。
- 改进建议**只写可执行、与本次已发生事实相关**的条目（如：某门禁缺证据、某步可脚本化），不写泛泛而谈。

### 3. 格式与可读性

- 正文**真实换行**：每行一条 `emoji 标签：值`；**禁止**在最终 payload 中保留字面上的 `\n` 反斜杠加 n（应使用多行文件或经脚本 `normalizeEscapedNewlines` 处理，见 `send-message.js`）。
- 语气温和、具体：先结论，再影响，再证据与下一步。
- 描述行 `📝 描述：...` 仍须 ≤100 字；`💡 改进建议` 建议 ≤280 字，分号或序号分隔短句。

### 4. 输出物（交给执行侧）

- 推荐输出两样之一：（1）完整的 `--content-file` 多行正文草案 + 建议附带的参数表；（2）可直接粘贴的 `send-message.js` 命令片段（含 `API_X_TOKEN` 等占位，由人本地替换）。
- 显式说明：`🤖 模型` / `🧰 工具` / `🕒 最后更新` 若未手写进正文，由脚本在 `--model` / `--tools` / `--updated-at` 下自动补行。

## 必答问题（被调用时必须回答）

1. 本次推送对应流程与结论类型是什么（完成 / 阻断 / 门禁 / 阶段）？证据路径各是什么？
2. 模型名、主要工具/调用链的**可核对来源**是哪里（环境变量、总结表、还是「未提供」）？
3. 本次会话**耗时**与 **Token/用量** 各是什么？来源是 Cursor/总结/还是「须从客户端核对」？（二者为推送**必填**维度）
4. 改进建议 2～4 条分别对应本次哪些**已出现**的问题或缺口？

## 约束

- 不代替用户执行 `send-message.js`（无 token 时只交付文案与参数说明）。
- 不编造 import-docs 的创建/覆盖数；须与上一步 `import-docs` 输出一致。
- 与 `wework-bot/SKILL.md` 中「生动总结格式规范」和 `rules/message-contract.md` 一致；新增字段 `🪙 会话用量`、`💡 改进建议` 由 `send-message.js` 的 `--token-usage`、`--improvement-hints` 注入时可单独成行。
