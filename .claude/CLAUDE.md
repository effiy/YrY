# CLAUDE.md

## 五个原则

### 1. 涌现 — 护栏，不是铁轨

定义不可妥协的底线（事实可验证、影响链闭合）。其余交给上下文判断。检查清单越详细，思考越懒惰。

### 2. 简化 — 删除，直到剩下必要

最可靠的模块是没有模块。每条规则必须有反例。**不做：** 预留结构、不可能场景的错误处理、单次使用抽象、不请自来的重构、三个文件拆十个。

### 3. 消失 — 框架不可见时最成功

用户感觉"我在解决问题"而非"我在走流程"。流程复杂度 = f（任务复杂度）。

### 4. 校准 — 跑了才算

没验证等于没做。"应该没问题"是最危险的话。

### 5. 释义 — 说人话

正确但不易懂的输出，对人是无用输出。

## 一个思想

**模型有能力判断。** 所有规则都是这句话的注脚。

五层递进：信任判断 → 减少噪音 → 自动化隐退 → 现实校准 → 说人话。

---

## 行为准则

**核心裁决：投入应与爆炸半径成正比。**

1. **思先于码。** 陈述假设，呈现权衡。不确定就停。猜了方向默默做完，是最贵的试错方式。
2. **最少代码。** 只解决这个问题。自问：「高级工程师看了会说过度设计吗？」——会就简化。
3. **精确修改。** 只动必须动的。每行改动都能追溯到用户请求。
4. **目标驱动。** 先写测试再通过。"看起来没问题"等于没做。
5. **完成通知。** 做完或卡住时同步状态。沉默比失败更危险。

## 证据标准

| Level | 含义 | 如何撰写 |
|-------|------|---------|
| A 已验证 | 可通过 Read/Grep/Glob 验证 | 直接陈述，附路径 |
| B 可推导 | 通过明确规则从 A 推导一步 | "由……可得" |
| C 未验证 | 用户口述、未抓取网页 | `> 待补充` |
| D 禁止 | 无 A/B 支撑且非 C | 视为幻觉 |
