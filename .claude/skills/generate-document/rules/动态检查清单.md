# 动态检查清单规范

## 文档概述

- 动态检查清单是全文档交付的**验证总表**，用于把需求任务中的主要操作场景、设计文档中的实现细节，映射为可执行的验证项。
- **上游文档**：需求任务（`docs/02_需求任务/`）、设计文档（`docs/03_设计文档/`）
- **下游文档**：项目报告
- **生成方式**：**规范驱动**。严禁基于 `templates/动态检查清单.md` 或任何历史模板反推内容；每一个场景、检查项、验证工具都必须可追溯到上游文档或代码。

## 生成前置条件（P0 硬门槛）

以下任一未满足，**必须中止生成** 并向用户要求补齐材料，不得凭经验补全：

1. `docs/02_需求任务/<功能名>.md` 存在，且含"主要操作场景"章节，且至少有 1 个完整场景（含前置条件、操作步骤、预期结果）。
2. `docs/03_设计文档/<功能名>.md` 存在，且含"主要操作场景实现"章节，且每个场景都能在其中找到对应实现说明。
3. 设计文档中引用的关键代码路径 **真实存在于仓库**（通过文件搜索校验）。

## 事实-来源映射（强制）

生成前在内部维护下列映射，并在写作时逐条核对；不在映射中的内容 **不得** 出现在文档里。

| 事实类型 | 来源约束 |
|---------|---------|
| 场景名 / 前置条件 / 操作步骤 / 预期结果 | 必须取自需求任务对应章节 |
| 关联模块 / 关键代码路径 / 验证要点 | 必须取自设计文档"主要操作场景实现"或真实文件 |
| 验证工具 / 技能 | 必须由 `find-skills` 返回；找不到则标注"未找到合适技能，建议人工复核"，**不得** 臆造技能名 |
| 优先级（P0/P1/P2） | 必须依据本规范的"优先级判定"小节 |

## 文档结构（严格遵循）

### 1. 文档头部（必须包含）

```markdown
# {功能名称} - 验证检查清单

> **文档版本**: v1.0 | **最后更新**: {YYYY-MM-DD} | **维护者**: {大模型名称} | **工具**: {Claude Code / Cursor}
>
> **关联文档**: [需求文档](../../01_需求文档/{文件名}.md) | [需求任务](../../02_需求任务/{文件名}.md) | [设计文档](../../03_设计文档/{文件名}.md) | [使用文档](../../04_使用文档/{文件名}.md)
>

[通用检查](#通用检查) | [主要操作场景验证](#主要操作场景验证) | [功能实现检查](#功能实现检查) | [代码质量检查](#代码质量检查) | [测试检查](#测试检查) | [检查总结](#检查总结)

---
```

### 2. 通用检查（必须）

固定四行，不得增删：

| 检查项 | 优先级 | 状态 | 备注 |
|--------|--------|------|------|
| 文档标题格式正确 | P0 | ⏳ | |
| 所有关联文档存在且链接有效 | P0 | ⏳ | |
| 代码仓库中相关文件已创建/更新 | P0 | ⏳ | |
| 项目可正常构建 | P0 | ⏳ | |

### 3. 主要操作场景验证（必须，动态生成）

**每个** 需求任务中的主要操作场景生成一节；场景数 = 需求任务场景数，不得增减。

每节结构：

```markdown
### 场景：{场景名称}

**关联需求任务场景**：[{场景名称}](../../02_需求任务/{文件名}.md#{场景锚点})

**关联设计文档实现**：[{场景实现}](../../03_设计文档/{文件名}.md#{实现锚点})

**验证工具建议**（由 find-skills 映射得出，只列出真实存在的）：
- 🏃 {技能名} — {适用原因}
- 📋 {工具名} — {适用原因}

#### 前置条件验证
| 检查项 | 优先级 | 状态 | 验证方法 | 备注 |
...

#### 操作步骤验证
...

#### 预期结果验证
...

#### 验证关注点检查
...
```

**约束**：
- 表格中的"检查项"必须是对上游原文的可验证转述，禁止新增原文没有的条件 / 步骤 / 结果。
- "验证方法"需具体（如"点击 #submit 按钮后 .toast 显示成功"），禁止出现"手动检查"这类与场景无关的泛泛表述。
- 若上游某信息缺失，写入 `> 待补充（原因：未在 <上游文档名>#<章节> 中找到）`，不编造。

### 4. 功能实现检查（按需）

仅当设计文档明确列出功能点时才包含。子章节：
- **核心功能检查**（P0 为主）
- **边界情况检查**（P1 为主）
- **错误处理检查**（P1/P2）

每条检查项必须可对应到设计文档的具体实现段落。

### 5. 代码质量检查（必须）

固定使用以下骨架，"验证方法"列必须引用真实存在的技能/工具；若本仓库未安装对应技能，写"未找到对应技能，建议人工复核"：

| 检查项 | 优先级 | 状态 | 验证方法 | 备注 |
|--------|--------|------|----------|------|
| 代码风格符合项目规范 | P1 | ⏳ | {code-review 技能 或 人工复核} | |
| 函数/变量命名清晰 | P1 | ⏳ | 人工复核 | |
| 无明显性能问题 | P2 | ⏳ | 性能分析工具 | |
| 无安全隐患 | P0 | ⏳ | {security-reviewer 代理建议 或 人工复核} | |

### 6. 测试检查（必须）

固定四行：

| 检查项 | 优先级 | 状态 | 验证方法 | 备注 |
|--------|--------|------|----------|------|
| 单元测试覆盖核心功能 | P1 | ⏳ | 检查测试文件 | |
| E2E 测试覆盖主要操作场景 | P0 | ⏳ | Playwright 运行 | |
| 所有 P0 测试通过 | P0 | ⏳ | 运行测试套件 | |
| 测试报告完整 | P1 | ⏳ | 检查测试输出 | |

### 7. 检查总结（必须）

必须包含以下三小节，初值为 0：

- **总体进度**（表格：类别 / 总项数 / 已完成 / 通过率）
- **待完成项**（复选框列表）
- **结论**（初始为 `⏳ 检查未开始`）

## 优先级判定

- **P0**：涉及主流程可用性、数据一致性、安全、构建是否成功
- **P1**：影响体验或可维护性但不阻断主流程（边界、样式、错误提示）
- **P2**：可选优化（性能微调、可访问性加强）

## 验证工具映射规则

由 `find-skills` 决定，禁止硬编码。推荐映射（仅作提示，最终以 find-skills 返回为准）：

| 场景特征 | 优先技能 | 备选 |
|---------|---------|------|
| UI / 用户流程 | `e2e-testing` | Playwright / 浏览器 DevTools |
| 代码实现 | `code-review` | 人工复核 |
| 安全 / 鉴权 / 敏感数据 | `find-agents` → `security-reviewer` | 人工复核 |
| 构建 / 集成 / 打包 | `verification-loop` | CI 日志 |

## 保存位置

`docs/<功能名>/05_动态检查清单.md`

## 质量检查清单

### P0 - 必须通过

- **文档头部完整**：包含版本、日期、维护者、关联文档
- **必需章节齐全**：通用检查 / 主要操作场景验证 / 代码质量检查 / 测试检查 / 检查总结
- **场景完备**：场景数与需求任务一致，每个场景含前置条件 / 操作步骤 / 预期结果三张表
- **来源可追溯**：每个场景都有需求任务与设计文档的锚点链接
- **代码路径真实**：所引用的文件路径在仓库中真实存在
- **无虚构技能**：验证工具必须由 find-skills 返回或降级为"人工复核"

### P1 - 应该通过

- **验证方法具体**：每条检查项的验证方法可执行、与场景相关
- **优先级合理**：P0/P1/P2 的分配符合本规范"优先级判定"
- **无断链**：所有锚点链接可解析

### P2 - 可以有

- **图标使用一致**：🏃📋🔍🟢🟡🔴 等图标统一
- **检查总结进度表**：有初始 0% 表格，便于后续填写
