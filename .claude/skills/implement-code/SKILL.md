> ## ⚠️ 强制最终步骤——每次调用完成前必须执行，禁止省略
>
> 无论代码实施是否完成（正常完成或任意阶段阻断），在生成 `06_实施总结.md` 后都必须**按顺序**完成以下四步，才算本次调用交付完成。
> **被阻断时同样必须执行，不得以"还没完成"为由跳过。**
>
> **步骤 C、D 为对 `import-docs`、`wework-bot` 两个技能的必调用**：须先完整阅读 `.claude/skills/import-docs/SKILL.md` 与 `.claude/skills/wework-bot/SKILL.md`，再按其流程执行（含标准命令与通知模板）；禁止只跑脚本不读技能、或省略任一技能即结束本次调用。
>
> **步骤 A：动态检查清单最终完成门禁（结束前唯一可放行的关卡）**
> 不论模型、工具、阶段是否声称"全部完成"，结束本次调用前必须按以下 **A1 → A2 → A3 顺序**完整执行，缺一不可：
>
> - **A1（先回写最终状态）**：按 `rules/artifact-contracts.md §3.1-3.3` 与 `rules/verification-gate.md §9.7` 把 `docs/<功能名>/05_动态检查清单.md` 全部应完成检查项的状态列、备注列与底部"检查总结"刷新到最终值（阶段 2 临时 `🏃` 必须升级为阶段 6 的 `✅` 或阻断）。仅口头声明"已通过"不算回写。
> - **A2（再复查全量完成）**：重新读取该文件并按 `artifact-contracts.md §3.4` 与 `verification-gate.md §9.8` 复查：
>   - P0 检查项必须全部为 `✅`，且备注包含阶段 6 验证证据、日期或报告引用
>   - "检查总结"的总体进度、待完成项与结论必须与表格状态完全一致
>   - 不得存在 `⏳`、`🏃`、`❌` 或未说明原因的 `⚠️` 的应完成项
>   - 必须把复查统计（P0/P1/P2 总数、已完成数、未完成数、需人工确认数）写入 `06_实施总结.md` **第 5 节「验证门禁结果归档」§5.5「动态检查清单最终完成复查」**（章节编号以 `rules/process-summary.md` 为准）
> - **A3（不通过即阻断）**：A1 或 A2 任一项不满足，**禁止**走正常完成路径——必须立即把 `06_实施总结.md` 切换为阻断版（标注"阶段 7/8 阻断 — 动态检查清单未全量完成"）、回写阻断状态、按本节步骤 D 走"门禁异常"通知模板，再结束。
>
> ⛔ **未按 A1 → A2 → A3 顺序执行、或 A2 任意一条未达成时仍发送"完成（成功）"通知，一律视为违反 skill 契约。**
>
> **步骤 B：回写功能文档状态**
> 读取 `.claude/skills/implement-code/rules/artifact-contracts.md`，按"功能文档状态回写契约"更新同一功能目录下相关文件：
> - `01_需求文档.md`、`02_需求任务.md`、`03_设计文档.md`、`04_使用文档.md`、`07_项目报告.md`：新增或更新 `## 实施状态` 小节
> - `05_动态检查清单.md`：更新状态列、备注列和"检查总结"
> - `06_实施总结.md`：记录状态回写结果
>
> **步骤 C：调用 import-docs 技能（文档同步）**
> **必须**按 `import-docs` 技能执行：读取 `.claude/skills/import-docs/SKILL.md`，再执行 `docs` 标准导入：
> ```bash
> node .claude/skills/import-docs/scripts/import-docs.js --dir docs --exts md
> ```
> - `docs` 不存在时跳过，并记录为 `☁️ 文档同步：docs 不存在，跳过导入`
> - 失败不阻断交付，但必须记录失败文件数和错误摘要
>
> **步骤 D：调用 wework-bot 技能（完成通知）**
> **必须**按 `wework-bot` 技能执行：读取 `.claude/skills/wework-bot/SKILL.md`，选择对应模板发送完成通知，并把步骤 C 的真实同步结果写入 `☁️ 文档同步` 行；通知**必须**含本次会话 `⏱️ 用时` 与 `🪙 会话用量`（`--duration` / `--token-usage` 或脚本缺省说明）。**须尽量补齐** `🤝 AI 调用`、`🔗 调用链`、`📐 实施总结图表`（`06_实施总结` §1/§2 节点与参与者数）、`🧩 MCP 明细`、`🧾 待办与风险`（P1/P2 条数，见报告 §9）、`🗂️ 状态回写`（`01/02/.../07` 是否已更新）等，与总结内「AI 调用记录」表一致（可用 `send-message.js` 的 `--diagram-summary`、`--mcp-breakdown`、`--backlog`、`--status-rewrite` 注入上述行）。
> - 正常完成 → 使用 "implement-code 完成（成功）" 模板（🚀✅）
> - 任意阶段阻断 → 使用 "implement-code 阻断" 模板（⛔🛠️）
> - 门禁未执行、被跳过、证据缺失或降级未记录 → 使用 "门禁异常" 模板（⛔）
> - 每条信息独占一行，使用 `━━━` 分隔线，数字填写实际值（禁止发含 `<占位符>` 的消息），并包含模型、工具、最后更新时间（精确到秒）；完成类推送在信息可统计时**不得**省略 `📁 测试路径`、`🧫 测试统计`、MCP/图表/待办行
>
> **⛔ 完成以上四步后才算本次 implement-code 调用结束。**

## 定位

`implement-code` 是 YiWeb 的代码实施主编排器。

它只负责 4 件事：

1. 判断是否满足进入实施的前提
2. 串起 8 个阶段的推进顺序
3. 在每个阶段设置进入 / 退出门禁
4. 确保每次调用都在 `docs/` 下生成实施总结或兜底运行记录

阶段细则、产物模板、回写格式、报告结构不在本文件重复定义，统一下沉到 `rules/`。

## 何时使用

- 已有 `docs/<功能名>/` 文档集
- 用户明确要求开始实施代码
- 需要按阶段门禁推进，而不是直接“边写边试”

## 何时不使用

- 还没有 `02_需求任务.md`、`03_设计文档.md`、`05_动态检查清单.md`
- 用户仍在做需求澄清、设计讨论或文档生成
- 只是做零碎代码修补，不需要完整实施流程

## 全局原则

1. **文档驱动**：所有实施决策都必须能回溯到文档或代码来源。
2. **测试先行**：先生成原型页和测试骨架，再进入代码实施。
3. **门禁优先**：阶段 2 和阶段 6 的 P0 门禁未通过时，禁止继续推进。
4. **Mock 仅限测试**：生产代码禁止引入条件性 mock 或测试桩。
5. **全项目影响链分析**：代码变更前后都要在项目范围内核对上游依赖、反向依赖、传递依赖、导出链、注册链、测试、文档、配置和外部依赖影响。
6. **测试产物路径唯一**：凡由本流程生成的 **Playwright spec、原型页、fixtures、截图、快照、trace、下载、测试辅助脚本** 等，**必须**落在仓库根目录 `tests/` 下（见 `rules/e2e-testing.md`、`rules/artifact-contracts.md §2`）；禁止写入 `src/`、`docs/`、项目根或其它目录作为最终落点。
7. **必须生成实施总结**：正常完成和中途阻断都必须落地 `06_实施总结.md`；总结须含 Mermaid **AI 调用流程图**与 **AI 调用时序图**（覆盖实际调用的 skills / agents / MCP / 工具等），见 `rules/process-summary.md`。
8. **每次调用必须落盘**：若 `{功能名}` 或文档集路径无法定位，必须写入 `docs/99_agent-runs/<YYYYMMDD-HHMMSS>_implement-code.md`，记录阻断原因和恢复建议。
9. **门禁失效即阻断**：任何应执行门禁未执行、被跳过、缺少证据、降级未记录或只口头声明通过，均视为 P0 阻断，必须推送 wework-bot 门禁异常消息。
10. **动态检查清单全完成才可结束**（结束前的唯一关卡）：本次调用结束前**必须**按"先回写最终状态 → 再复查全部完成 → 不通过即阻断"的顺序处理 `docs/<功能名>/05_动态检查清单.md`：
   - 先按 `rules/artifact-contracts.md §3.1-3.3` 与 `rules/verification-gate.md §9.7` 把所有应完成项的状态列、备注列、底部"检查总结"回写到最终值
   - 再按 `artifact-contracts.md §3.4` 与 `verification-gate.md §9.8` 复查全部应完成项是否已为最终通过状态
   - 复查未通过、回写未执行、证据缺失或仅口头声明完成，一律禁止走正常完成路径，必须改走门禁异常 / 阻断流程后再结束
11. **证据与不确定性（反幻觉）**：`../../shared/evidence-and-uncertainty.md` 为共享规范；`06_实施总结.md` 中「自我改进」与「可验证下一步」须引用真实过程产物（§5/§7/§8 等），**禁止**为凑满而写未发生的失败、未调用的技能或未存在的路径。
12. **场景—检查项覆盖预检（覆盖率）**：阶段 0 按 `rules/orchestration.md §3.4` 完成 `02` 与 `05` 的 P0 覆盖核对表后再进入阶段 1；缺口须留痕或先补文档，禁止静默跳步。
13. **结束必调两技能**：每次 `implement-code` 调用结束前必须**按技能**完成 `import-docs`（先读 `SKILL.md`，再执行 `docs` 标准导入）与 `wework-bot`（先读 `SKILL.md`，再发完成/阻断/门禁异常通知）；顺序与字段要求以两技能正文及本文「强制最终步骤」步骤 C、D、阶段 8 为准，禁止省略。

## 输入前提

- `{功能名}` -> `docs/<功能名>/`
- `{文档集路径}` -> 默认同上

若无法从用户请求或现有目录中确定 `{功能名}`，不得空跑或只口头询问；必须先生成兜底运行记录：

```text
docs/99_agent-runs/<YYYYMMDD-HHMMSS>_implement-code.md
```

最低文档要求：

| 文件 | 级别 | 用途 |
|------|------|------|
| `02_需求任务.md` | P0 | 用户故事、场景、前置条件 |
| `03_设计文档.md` | P0 | 模块、接口、实现约束 |

| 阶段 | 名称 | 目标 | 主要规则 |

| 1 | 测试先行 | 生成测试原型页与 Playwright 骨架 | `rules/e2e-testing.md` + `rules/test-page.md` |

## 阶段编排

### 阶段 0：文档驱动

- 目标：解析 `{功能名}`，完成文档预检、Grounding、分派与探针检查。
- 进入条件：用户明确要求开始实施，且目标功能目录可定位。
- 规则入口：`rules/orchestration.md`
- 退出门禁：P0 文档齐全，必要工具有可用方案。

### 阶段 1：测试先行

- 目标：为每个场景生成测试原型页和 Playwright 骨架。
- 进入条件：阶段 0 完成。
- 规则入口：`rules/e2e-testing.md`、`rules/test-page.md`、`rules/artifact-contracts.md`
- 退出门禁：测试产物、场景锚点、`data-testid` 覆盖率完整。

### 阶段 2：动态检查门禁

- 目标：在原型页面上用 Playwright-MCP 验证全部 P0 项。
- 进入条件：阶段 1 完成。
- 规则入口：`rules/verification-gate.md`、`rules/artifact-contracts.md`
- 退出门禁：P0 全部通过；否则按门禁规范阻断。

### 阶段 3：模块预检

- 目标：在写代码前完成环境预检和全项目影响链闭合分析，确认设计文档中的改动点、依赖方向、处置方式与真实代码一致。
- 进入条件：阶段 2 通过。
- 规则入口：`rules/orchestration.md`、`rules/code-implementation.md`
- 退出门禁：预检清单无阻断项。

### 阶段 4：写项目代码

- 目标：按设计文档顺序逐模块实施并立即验证。
- 进入条件：阶段 3 完成。
- 规则入口：`rules/code-implementation.md`
- 退出门禁：模块实现完成，逐模块验证结果已记录。

### 阶段 5：代码评审

- 目标：在冒烟测试前清掉所有 P0 代码问题。
- 进入条件：阶段 4 完成。
- 规则入口：`rules/e2e-testing.md`、`rules/code-implementation.md`
- 退出门禁：无 P0 问题。

### 阶段 6：冒烟测试

- 目标：在真实功能页面上验收全部 P0 项。
- 进入条件：阶段 5 完成。
- 规则入口：`rules/verification-gate.md`、`rules/artifact-contracts.md`
- 退出门禁：
  1. 真实页面全部 P0 通过
  2. 已按 `verification-gate.md §9.7` 把对应检查项**最终状态**（含备注与"检查总结"）回写到 `05_动态检查清单.md`，阶段 2 临时 `🏃` 已升级为 `✅` 或阻断
  3. 任一条不满足按门禁规范阻断，禁止进入阶段 7

### 阶段 7：过程总结

- 目标：生成完整或阻断版 `06_实施总结.md`，并更新同一功能目录内其他文档的实施状态信息。
- 进入条件：阶段 6 通过，或任意前置阶段阻断。
- 规则入口：`rules/process-summary.md`、`rules/artifact-contracts.md`
- 退出门禁（按顺序全部满足才放行）：
  1. `06_实施总结.md` 写入完成
  2. 已按 `artifact-contracts.md §4` 完成 `01/02/03/04/05/07` 的功能文档状态回写并在总结中记录
  3. **动态检查清单最终完成门禁通过**（按 `artifact-contracts.md §3.4` 与 `verification-gate.md §9.8` 复查），复查结果已写入总结 **§5.5「动态检查清单最终完成复查」**（见 `rules/process-summary.md` 章节结构）
  4. 任一条不满足，必须改走阻断版总结、门禁异常通知，禁止进入阶段 8 的"完成（成功）"路径

### 阶段 8：文档同步 + 完成通知（每次必须执行，不得省略）

- 目标：通知团队并同步最新文档到 YiAi。
- 进入条件：阶段 7 完成（无论正常完成还是阻断）。
- 执行步骤：
  1. **动态检查清单最终更新与确认**（必须按子顺序，缺一即阻断）：
     - 1.1 **先回写最终状态**：按 `verification-gate.md §9.7` 与 `artifact-contracts.md §3.1-3.3` 把 `05_动态检查清单.md` 全部应完成项的状态列、备注列、"检查总结"刷到最终值
     - 1.2 **再复查全量完成**：按 `artifact-contracts.md §3.4` 与 `verification-gate.md §9.8` 复查（P0 全部 `✅` + 备注含验证证据 + 无 `⏳/🏃/❌` + `⚠️` 已说明原因 + "检查总结"与表格一致）
     - 1.3 **结果写入总结**：把复查统计与最终结论写入 `06_实施总结.md` **§5.5**
     - 1.4 **不满足即降级**：任一项不达成，按门禁异常路径阻断，**禁止**继续走"完成（成功）"通知
  2. **状态回写确认**：确认阶段 7 已按 `artifact-contracts.md §功能文档状态回写契约` 更新 `01/02/03/04/05/07` 的实施状态，并在 `06_实施总结.md` 记录回写结果。
  3. **import-docs（必调技能）**：**必须**调用 `import-docs` 技能：先读 `import-docs/SKILL.md`，再执行 `docs` 标准导入（见 `import-docs/SKILL.md §docs 标准导入`）；目录不存在时记录跳过；失败不阻断交付但须保留失败文件数和错误摘要。
  4. **wework-bot（必调技能）**：**必须**调用 `wework-bot` 技能：先读 `wework-bot/SKILL.md`，**严格按照** `wework-bot/SKILL.md §生动总结格式规范` 中的对应模板发送完成通知，并把 import-docs 的真实结果写入 `☁️ 文档同步` 行；**须含** `⏱️ 用时` 与 `🪙 会话用量`（`--duration` / `--token-usage` 或脚本缺省句）：
     - 正常完成 → 使用 "implement-code 完成（成功）" 模板（🚀✅）
     - 任意阶段阻断 → 使用 "implement-code 阻断" 模板（⛔🛠️）
     - 门禁失效 → 使用 "门禁异常" 模板，写明缺失证据、门禁阶段和恢复动作
     - 每条信息独占一行，使用 `━━━` 分隔线，数字填写实际结果，并包含 `🤖 模型`、`🧰 工具`、`🕒 最后更新`
- 退出门禁（按顺序全部满足才可结束本次调用）：
  1. 步骤 1 的"动态检查清单最终更新与确认"已完整执行（先回写、再复查、结果入总结）
  2. 步骤 2 的"状态回写"已完成且记录入总结
  3. 步骤 3 的 `import-docs` 已执行或记录跳过原因（含真实数字）
  4. 步骤 4 的 `wework-bot` 通知已发送，模板与本次结束类型一致（成功 / 阻断 / 门禁异常），含**会话耗时、Token/用量**、模型、工具、秒级最后更新时间
  5. 任一同步或通知失败必须先把原因、脱敏路由、模型、工具、秒级时间写回 `06_实施总结.md` 或 `docs/99_agent-runs/` 后再结束
  6. **若动态检查清单最终完成门禁未通过却已发送"完成（成功）"模板，视为违反契约：必须立即追加发送"门禁异常"通知并在总结中标注修正记录**

## 停止条件

以下情况必须停止当前推进：

- `{功能名}` 或文档集路径无法定位
- P0 文档缺失
- 必要 MCP 不可用且无降级方案
- 阶段 2 达到修复上限仍未通过
- 阶段 6 达到修复上限仍未通过
- 任意阶段门禁未执行、被跳过、缺少证据、降级未记录或只口头声明通过
- 所有模块阻断且无法继续

停止时必须按顺序执行（**禁止跳过任何一步**）：

1. 记录阻断原因
2. 汇总已产生产物
3. 若功能目录可定位，生成 `docs/<功能名>/06_实施总结.md`
4. 若功能目录不可定位，生成 `docs/99_agent-runs/<YYYYMMDD-HHMMSS>_implement-code.md`
5. 若功能目录可定位，按 `artifact-contracts.md §功能文档状态回写契约` 回写阻断状态到 `01/02/03/04/05/07`，并在 `06_实施总结.md` 记录回写结果
6. **执行 import-docs 文档同步**（先于通知执行，获取真实数字）：
   ```bash
   node .claude/skills/import-docs/scripts/import-docs.js --dir docs --exts md
   ```
   `docs` 不存在时跳过，记录"目录不存在"。
7. **⚠️ 必须发送 wework-bot 阻断 / 门禁异常通知**（无论何种阻断原因，无论阶段多早，**一律发送**）：
   读取 `.claude/skills/wework-bot/SKILL.md`，使用 "implement-code 阻断" 或 "门禁异常" 模板发送通知，包含阻断原因、门禁证据、P0 统计、上一步的真实同步结果、模型、工具和最后更新时间（精确到秒）。
   > ⛔ **此步骤为阻断流程的最后一步，不得省略。省略即违反 skill 契约。**
