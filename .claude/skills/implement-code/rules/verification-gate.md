# 验证门禁规范

> 本规范约束 implement-code 技能在**阶段 2（动态检查门禁）**和**阶段 6（冒烟测试）**的决策逻辑与执行要求。  
> **核心意图**：没有通过门禁的项目，禁止进入代码实施阶段。门禁是不可跳过的强制关卡。

---

## 1. 门禁原则（P0 — 不可违反）

| 编号 | 原则 |
|------|------|
| G0-1 | P0 检查项有任意一项未通过，**禁止进入阶段 3（模块预检）**。没有例外。 |
| G0-2 | 门禁失败时，**必须** 修复测试页面（阶段 2），不得修改动态检查清单以"降级"P0 项。 |
| G0-3 | 自修复上限为 **2 轮**；第 3 次仍失败时，停止并输出阻断报告，等待人工介入。 |
| G0-4 | 门禁结果 **必须** 完整保存，供阶段 7 过程总结引用。 |
| G0-5 | "人工确认"类检查项 **不计入** P0 通过/失败判定，但必须在报告中列出。 |
| G0-6 | 门禁未执行、被跳过、缺少报告证据、降级未记录或只口头声明通过，均视为**门禁失效**，必须停止并发送 wework-bot 门禁异常通知。 |
| G0-7 | 门禁失败、门禁失效和阻断报告生成失败都必须推送消息；通知发送失败时必须把失败原因写入 `06_实施总结.md` 或 `docs/99_agent-runs/` 兜底记录。 |
| G0-8 | 阶段 1 / 2 / 6 退出前必须执行 `artifact-contracts.md §2.2 测试路径门禁扫描命令`：任一命令命中非空即视为 P0 失败，必须先迁移或清理；扫描命令、命中清单、处置结果与最终结论必须原样写入 `06_实施总结.md` 的「验证门禁结果归档 § 测试路径门禁」小节，并在通知的 `📁 测试路径` 行写明结论。 |

## 2. 检查项分类与验证方式

### 2.1 P0 — 必须通过（Playwright-MCP 自动验证）

| 检查类型 | MCP 验证方式 | 通过条件 |
|---------|------------|---------|
| UI 元素存在 | `browser_snapshot` 后语义比对 + `browser_evaluate` 执行 `toBeVisible()` | 元素可见，无控制台 JS 错误 |
| UI 元素可操作 | `browser_snapshot` 确认未禁用 + `browser_evaluate` 执行 `toBeEnabled()` | 元素未禁用、未被遮挡 |
| 操作步骤可执行 | `browser_click / browser_fill` 无报错 | MCP 工具调用成功，无超时 |
| 桩行为触发 | `browser_click` 后 `browser_snapshot` 比对状态变更 | 对应结果区域可见 / 文本更新 |
| API Mock 生效 | `browser_evaluate` 检查 `page.route` 是否拦截请求 | mock 响应被正确返回 |
| data-testid 覆盖 | `browser_evaluate` 枚举检查清单操作项的 testid | 无遗漏 testid |

### 2.2 P1 — 建议通过（自动 + 人工）

| 检查类型 | 验证方式 |
|---------|---------|
| 错误状态展示 | 触发错误条件，确认 `error-msg` 可见 |
| 边界值处理 | 输入极端值，确认无 JS 错误 |
| 可访问性 | ARIA 属性存在且语义正确 |

### 2.3 P2 — 可选（人工）

| 检查类型 | 说明 |
|---------|------|
| 视觉一致性 | 与设计稿或使用文档截图对比 |
| 响应式布局 | 在不同视口宽度下的表现 |
| 动画/过渡效果 | 是否符合交互设计说明 |

## 3. 验证执行流程（Playwright-MCP 驱动）

阶段 2 的所有 P0 自动验证 **必须使用 Playwright-MCP 工具**执行交互式验证，流程如下：

```
对每个场景（按动态检查清单顺序）：
  1. browser_navigate  → 导航至原型页面（本地 dev server 或 file://）
  2. browser_snapshot  → 确认初始状态与前置条件一致（DOM 快照比对）
  3. （若场景涉及 API）→ 确认 .spec.ts 中 page.route mock 已声明
  4. 按操作步骤顺序执行 MCP 工具：
       browser_click / browser_fill / browser_select_option
  5. browser_snapshot  → 截取操作后状态，与预期结果语义比对
  6. browser_evaluate  → 执行断言逻辑（对应检查清单预期结果）
  7. 记录结果：✅ 通过 / ❌ 未通过（含 MCP 操作记录 + 截图描述）
  8. 本轮结束后统计 P0 通过率，并保存门禁证据路径
  9. 若证据路径、工具序列或 P0 统计缺失，按"门禁失效"处理并停止
```

**降级策略**：若 Playwright-MCP 不可用，降级为 `npx playwright test tests/e2e/<功能名>/ --reporter=list`，但必须在验证报告中注明"Playwright-MCP 不可用，降级为命令行执行"。

**失效判定**：若既没有 Playwright-MCP 操作记录，也没有命令行降级执行记录，或降级原因未写入报告，禁止判定为通过；必须发送 `wework-bot` 的"门禁异常"通知，说明门禁未生效、缺失证据和恢复动作。

## 4. 修复循环约束

```
轮次 1：
  失败项 → 分析根因（元素缺失 / testid 错误 / 桩行为未触发）
         → 仅修复测试原型页面（阶段 2 产物）
         → 不修改检查清单内容
         → 重新运行阶段 2

轮次 2：
  失败项 → 同上
         → 若根因指向需求歧义（检查清单与设计文档矛盾），
           在报告中标注"文档冲突"，等待人工裁决后继续

轮次 3（上限）：
  输出阻断报告（见第 5 节），停止执行
  发送 wework-bot 门禁异常 / implement-code 阻断通知
  等待人工确认修复方向后，从阶段 2 重新开始
```

## 7. 原型验证状态回写（阶段 2）

阶段 2 门禁通过后，**必须**将原型页面验证结果回写到 `docs/<功能名>/05_动态检查清单.md`。

### 7.1 状态更新规则

| 验收结果 | 检查项优先级 | 状态列更新 | 备注列追加 |
|---------|------------|----------|----------|
| P0 通过 | P0 | 🏃（原型验证通过） | `原型验证通过 <YYYY-MM-DD>` |
| P0 未通过 | P0 | ❌ | 失败原因摘要 |
| P1/P2 通过 | P1/P2 | 🏃 | `原型验证通过 <YYYY-MM-DD>` |
| P1/P2 未通过 | P1/P2 | ⏳ 或 ⚠️ | 未通过原因摘要 |

### 7.2 回写时机

- 阶段 2 门禁通过后、解锁阶段 3 之前，一次性回写所有场景的原型验证结果
- 不采用逐场景回写（阶段 2 是整轮验收，通过/阻断决策后才回写）

### 7.3 状态标记含义

| 状态 | 含义 | 更新阶段 |
|------|------|---------|
| ⏳ | 未验证 | 初始状态 |
| 🏃 | 原型验证通过 | 阶段 2 |
| ✅ | 冒烟测试通过 | 阶段 6 |
| ❌ | 验证未通过 | 阶段 2 或 6 |
| ⚠️ | 需人工确认 | 阶段 2 或 6 |

### 7.4 阶段 2 回写禁止事项

- ❌ 在阶段 2 未实际执行 Playwright-MCP 验证的情况下更新状态
- ❌ 将原型验证结果（🏃）误标为最终通过（✅）— ✅ 仅由阶段 6 冒烟测试确认
- ❌ 为"通过"门禁修改动态检查清单的预期结果
- ❌ 跳过阶段 2 回写直接进入阶段 3

## 8. 禁止事项

- ❌ 将 P0 检查项降级为 P1 以"通过"门禁
- ❌ 在门禁未通过时直接跳到代码实施
- ❌ 通过修改动态检查清单的"预期结果"来让断言通过
- ❌ 将"人工确认"项标记为自动通过
- ❌ 修复轮次超过 2 次后仍继续尝试（必须停止并输出阻断报告）

## 9. 冒烟测试门禁（阶段 6）

> 阶段 2 验证测试原型页面（桩实现），本节约束**阶段 6（冒烟测试）**对真实项目代码的验证门禁。
> **核心意图**：代码写完后必须对真实页面执行与阶段 2 同等的 P0 验证，确保实施未引入回归。

### 9.1 代码验证门禁原则

| 编号 | 原则 |
|------|------|
| CG0-1 | 冒烟测试 P0 有任意一项未通过，**禁止进入阶段 7**。没有例外。 |
| CG0-2 | 验证失败时，**必须** 完成根因分析 + 充分预检后才能修复，**禁止**不分析原因直接修改代码重试。 |
| CG0-3 | 自修复上限为 **1 轮**（根因分析 → 预检 → 修复 → 重验证）；第 2 次仍失败时，停止并输出代码验证阻断报告，等待人工介入。 |
| CG0-4 | 修复必须**精确**：只修改根因定位到的代码，不做无关改动。 |
| CG0-5 | 重验证时必须重新运行**全部** P0 检查项（防回归），不能只验证修复项。 |
| CG0-6 | 代码验证门禁结果 **必须** 完整保存，供阶段 7 过程总结引用。 |
| CG0-7 | 阶段 6 未执行真实页面验证、缺少门禁报告、未回写动态检查清单或只验证修复项，均视为**代码验证门禁失效**，必须停止并发送 wework-bot 门禁异常通知。 |
| CG0-8 | 所有任务完成后，必须复查 `05_动态检查清单.md` 全部应完成项的最终状态；清单未全量完成或"检查总结"未同步更新时，禁止结束流程。 |
| CG0-9 | 阶段 6 退出前必须再次执行 `artifact-contracts.md §2.2 测试路径门禁扫描命令`，确认冒烟测试期间未在 `tests/` 外新增任何测试产物（包括 Playwright-MCP 默认输出、trace、HAR、临时下载）；命中视为 P0 阻断。 |

### 9.2 验证执行方式

与阶段 2 使用相同的 Playwright-MCP 驱动流程，关键差异：

| 对比项 | 阶段 2（动态检查门禁） | 阶段 6（冒烟测试） |
|-------|-----------------|-------------------|
| 验证对象 | 测试原型页面（HTML 桩） | 真实功能页面（Vue 组件） |
| 导航目标 | `file://` 或原型页面 URL | `http://localhost:<port>/<功能路由>` |
| API Mock | `page.route` 注入 mock | 使用真实 API 或测试环境后端；若后端不可用则允许在 `.spec.ts` 中 mock |
| 验证范围 | 动态检查清单 P0 项 | P0 项 + 组件注册 + 路由跳转 + Store 状态持久化等集成行为 |
| 修复上限 | 2 轮 | 1 轮 |

### 9.3 根因分析分类（代码验证专属）

代码验证失败时，根因分析必须从以下分类中选择：

| 根因分类 | 典型表现 | 预检要点 |
|---------|---------|---------|
| 组件未渲染 | `browser_snapshot` 中找不到组件元素 | 检查 `src/components/index.js` 注册、import 路径、组件名拼写 |
| Store 状态未更新 | 操作后页面无响应或数据不刷新 | 检查工厂函数、`reactive` 包装、`src/stores/index.js` 注册 |
| 路由配置错误 | 导航失败或显示空白页 | 检查 `src/router/index.js` 路径、守卫、懒加载配置 |
| data-testid 缺失/不匹配 | `browser_click` 找不到元素 | 对照测试原型页面确认 testid 一致性 |
| 异步时序问题 | 操作后状态未及时更新 | 检查 `await` 缺失、`nextTick` 未调用、Promise 未处理 |
| API 调用异常 | 接口返回错误或无响应 | 检查接口地址、请求体结构、错误处理逻辑 |
| CSS/布局遮挡 | 元素存在但不可见/不可操作 | 检查 `z-index`、`display`、`overflow`、`position` |
| 环境问题 | dev server 未启动或编译错误 | 检查 Node 版本、依赖安装、环境变量、编译输出 |

### 9.4 自修复预检清单（修复前必须全部确认）

- [ ] 已读取失败项涉及的所有源文件（组件 / Store / 路由）
- [ ] 已确认修复不会引入新的 P0 问题
- [ ] 已确认修复与设计文档约束一致
- [ ] 已确认修复与编码规范（Store 工厂模式、组件全局注册等）一致
- [ ] 若修改涉及多个文件，已确认修改的完整性和一致性
- [ ] 已确认 dev server 正在运行且无编译错误

### 9.7 动态检查清单状态回写

代码验证门禁验收完成后，**必须** 将验收结果回写到 `docs/<功能名>/05_动态检查清单.md`：

**状态更新规则**：

| 验收结果 | 检查项优先级 | 状态列更新 | 备注列追加 |
|---------|------------|----------|----------|
| P0 通过 | P0 | ✅ | `Playwright-MCP 验证 <YYYY-MM-DD>` |
| P0 未通过 | P0 | ❌ | 失败原因摘要 |
| P1/P2 通过 | P1/P2 | ✅ | `Playwright-MCP 验证 <YYYY-MM-DD>` 或验证方法描述 |
| P1/P2 未通过 | P1/P2 | ⚠️ | 未通过原因摘要 |

**回写时机**：
- 逐场景验收时，每完成一个场景的 P0 验收后立即回写该场景对应检查项的状态
- 自修复后重新验证时，同步更新受影响检查项的状态
- 全部场景验收完成后，回写"通用检查"、"功能实现检查"、"代码质量检查"、"测试检查"中涉及验证的检查项

**检查总结同步更新**：

回写检查项状态后，必须同步更新动态检查清单底部的"检查总结"章节：

- **总体进度表**：更新"已完成"列和"通过率"列
- **待完成项**：已完成的 `[ ]` 改为 `[x]`
- **结论**：
  - 全部 P0 通过 → `✅ 所有 P0 检查项已通过（Playwright-MCP 门禁验收 <YYYY-MM-DD>）`
  - 有 P0 未通过 → `❌ 存在 P0 检查项未通过（见上文 ❌ 标记项）`

### 9.8 最终完成门禁（结束前的唯一关卡）

> **核心意图**：阶段 6 通过 ≠ 可以结束。本节是 implement-code 结束前的最后一道强制门禁；任何分支（含正常完成、阻断、降级）走到"准备结束"前都必须经过本门禁。

#### 9.8.1 执行顺序（不可重排，缺一即门禁失效）

**步骤 1：先把 `05_动态检查清单.md` 回写到最终状态**

- 按 §9.7 与 `artifact-contracts.md §3.1-3.3` 的状态映射规则，把每条应完成检查项的**状态列**、**备注列**、底部**"检查总结"**刷到本次实施的最终值
- 阶段 2 写入的临时 `🏃` 状态必须基于阶段 6 验证结果升级为 `✅`，未通过的项必须改为 `❌` 或 `⚠️` 并写明原因
- 任何"打算稍后再回写"、"留到通知里说"或"先发送通知再补回写"的做法均视为门禁失效
- 回写完成后必须**再次读取**该文件，确认磁盘内容已被刷新（防止幻觉式确认）

**步骤 2：再执行最终完成复查**

依次校验下列条件，全部满足才允许进入"正常结束"路径：

1. 统计 P0 / P1 / P2 检查项的总数、已完成数、未完成数、需人工确认数
2. P0 检查项全部为 `✅`，且备注列包含阶段 6 验证证据（日期、报告引用、Playwright-MCP 工具序列摘要等）
3. 不存在 `⏳`、`🏃`、`❌`，也不存在未说明原因 / 负责人 / 后续动作的 `⚠️`
4. 阶段 2 的 `🏃` 临时状态已在阶段 6 后升级为最终 `✅`（没有"残留 🏃"）
5. "检查总结"的总体进度、待完成项、结论与表格状态完全一致

**步骤 3：把复查结果写入 `06_实施总结.md` 第 5 节「验证门禁结果归档」之 §5.5「动态检查清单最终完成复查」**

- 必须以 §9.8.2 的复查表格形式落盘
- 必须在完成通知的 `🧪 P0 检查项` 行（或 `🔍` 类相关行）体现 P0 已完成数 / 总数与最终结论

#### 9.8.2 复查结果表格（写入 `06_实施总结.md` §5.5，文档章节结构以 `process-summary.md` 为准）

#### 9.8.3 门禁不通过的处置（强制阻断分支）

步骤 1 未完整回写，或步骤 2 任意条件未满足时，**立即**按下列顺序处理，禁止继续走"完成（成功）"路径：

1. 把 `06_实施总结.md` 切换为阻断版，标注为 `⛔ 阶段 7/8 阻断 — 动态检查清单未全量完成`
2. 在阻断版总结中明确列出未完成检查项 / 缺失证据 / 不一致的"检查总结"字段
3. 按 `artifact-contracts.md §4` 把 `01/02/03/04/05/07` 的实施状态回写为 `⛔ 已阻断` 并写明恢复起点
4. 执行 `import-docs` 标准导入，记录真实结果
5. 调用 `wework-bot` 的"门禁异常"模板发送通知（**禁止**使用"完成（成功）"模板），通知正文必须包含未完成项数量、未达成的复查条件、恢复动作、模型、工具、秒级最后更新时间
6. 通知发送后才允许结束本次调用

#### 9.8.4 禁止事项

- ❌ 跳过步骤 1 直接执行步骤 2（"反正都是 ✅，不用回写"）
- ❌ 步骤 1 只回写状态列、不更新备注列或"检查总结"
- ❌ 复查时只看 P0 不看 P1/P2 的"应完成项"状态
- ❌ 复查未通过仍发送"完成（成功）"通知
- ❌ 把"复查不通过"作为 P2 问题写在"未解决问题"中而不阻断结束
- ❌ 在 `06_实施总结.md` 之外的任何地方"口头复查通过"

### 9.9 代码验证门禁禁止事项

- ❌ 修改检查项的优先级以绕过门禁
