✅ 好案例 "首修任务"引导设计

📁 docs/故事任务面板/yry-arch/场景-3-新人上手.md : 上手流程
以"环境搭建 → 了解项目 → 首修任务 → 提交变更"为主线，给新人一个有终点的学习路径。首修任务（如修改一个文档、添加一个测试用例）让新人在 30 分钟内完成第一个贡献。
💡 为什么好：给新人一个具体可完成的小目标——这比"读完所有文档"有效得多。完成首修即建立了对系统的感性认知和操作信心。

✅ 好案例 目录导览与架构图呼应

📁 docs/故事任务面板/yry-arch/场景-3-新人上手.md : 目录导览
按 skills/ → agents/ → rules/ → docs/ → tests/ 层次导览项目目录，与场景 1 架构图的 3 层 Tier 结构一致。新人在导览后自然过渡到架构文档的深入阅读。
💡 为什么好：目录结构反映了架构设计——让新人先看目录，再看架构图，物理结构和逻辑结构形成双重认知锚点。

❌ 坏案例 无预期输出验证

📁 docs/故事任务面板/yry-arch/场景-3-新人上手.md : 操作步骤
关键操作如 rui-story init、rui-claude sync 仅给出命令，未说明运行成功后的预期终端输出。新人无法判断操作是否成功——如果命令静默失败（如 token 未配置），新人可能在 30 分钟后才发现系统根本没跑起来。
💡 为什么不好：无声失败是新人的头号杀手。每个操作步骤都应包含"如果成功，你应该看到..."的预期输出截图或文字，让新人能自我验证每一步。

❌ 坏案例 缺少故障排查 FAQ

📁 docs/故事任务面板/yry-arch/场景-3-新人上手.md : 全局
整个文档假设环境搭建一帆风顺。现实中新人会遇到的 node 版本不对、权限不足、企微 Webhook 未配置、API token 缺失等问题，没有任何故障排查指引。
💡 为什么不好：文档的"快乐路径"假设在新人场景下最脆弱——新人恰恰是最容易遇到环境问题的人群。没有 FAQ 意味着新人卡住后只能中断上手流程去搜索解决方案。
P1 · 立即

为关键操作增加预期输出

在每步操作后增加"预期输出"框（终端截图或文字），让新人能自我验证。参考 formulas.md 中"效果示意"的写法。

P1 · 立即

增加故障排查 FAQ

收集过去 3 个月新人实际遇到的阻碍（≥5 项），编写症状→原因→修复的排查步骤。预计 1 小时。

P2 · 本迭代

增加"20 分钟快速体验"路径

设计可复制粘贴的命令序列，让新人在理解架构前先看到系统跑起来。

P3 · 未来

补充多平台环境搭建说明

增加 Linux (Ubuntu) 和 Windows (WSL2) 的差异化说明。
