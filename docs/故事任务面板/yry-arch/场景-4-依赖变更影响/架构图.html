⚡ ⚡ 场景-4-依赖变更影响 · 架构图

依赖变更时的全局影响分析与修复路径

📐 模块拓扑

输入 / 触发

核心模块（如 rules/code-pipeline.md）发生变更

核心执行

变更检测：识别变更类型和影响面 影响分析：追踪依赖链和消费者 级联更新：按影响优先级逐级刷新 回归验证：全量自检通过

门禁 / 审查

security 审计变更是否引入新风险 code-reviewer 验证级联更新完整性

产出 / 交付

影响分析报告 + 级联更新 PR

🔄 数据流

输入→ 处理→ 验证→ 产出

变更提交 → 影响分析 → 级联更新 → 自检 → review → 合并

📋 关键决策点

• T1 变更（措辞/格式）：跳过架构分析，直接刷新
• T2 变更（增删/接口）：裁剪分析，刷新目标和下游
• T3 变更（边界/重构）：完整重跑影响分析 + 架构设计
