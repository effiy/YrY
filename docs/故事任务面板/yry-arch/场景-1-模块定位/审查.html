✅ 好案例 场景文档结构与公式严格对齐

📁 docs/故事任务面板/yry-arch/场景-1-模块定位.md : 整体结构
场景 .md 文件严格遵循 formulas.md 定义的公式：版本元数据 → 导航 → 概述(角色·目标·优先级) → 主要价值 → 图谱定位 → §0 技术评审 → §1 测试设计 → §2 实施报告 → §3 测试报告 → §4 自改进。与场景-2/3/4 结构完全一致。
💡 为什么好：结构一致性是文档基线的基础——当所有场景遵循相同公式时，交叉引用和自动化校验变得可行。读者切换场景时认知负荷为零。

❌ 坏案例 图谱定位表占位文本未回填

📁 docs/故事任务面板/yry-arch/场景-1-模块定位.md : §0 图谱定位表
图谱定位表中结构层和内容层节点标记为"文档生成阶段填充"，这些字段在文档已生成完毕后仍未更新。导致知识图谱的可追溯性断裂——从图谱节点无法反向定位到具体的 .html 产物文件。
💡 为什么不好：占位文本在生成后不更新，违反了"验现实"铁律。下游工具（如知识图谱的完整性校验）会因为这些占位文本而误判文档不完整。正确做法是生成后立即回填实际文件路径。
| 图层 | 本场景节点 | 上游 | 下游 |
|------|-----------|------|------|
| 结构层 | —（文档生成阶段填充） | ← 应填: 架构图.html | ← 应填: 知识图谱.html |
| 内容层 | —（文档生成阶段填充） | ← 应填: 测试面板.html | ← 应填: 演示.html |
P1 · 立即

补全图谱定位表的结构层和内容层节点

将场景 .md 中图谱定位表的占位文本替换为实际文档路径（架构图.html、知识图谱.html、测试面板.html、演示.html），确保知识图谱可追溯。预计 5 分钟。

P2 · 本迭代

在知识图谱中标注模块依赖权重

为模块间关系增加 weight 字段（强依赖/弱依赖/可选），使依赖变更影响分析有精确的量化依据。建议先在 knowledge-graph.md 增加 weight schema。

P2 · 本迭代

增加模块演化历史记录

在场景文档中增加 version_history 小节，记录 Agent 数量从 9→8、规则从 10→8 的变化及原因。避免架构考古时无法追溯演化路径。

P3 · 未来

架构图增加交互式模块筛选

在演示页面增加可交互的架构图（支持按 Tier 筛选、按模块名搜索高亮），参考 D3.js 或 Mermaid 的交互能力。提升大型架构图的可用性。

P3 · 未来

补充模块间交互箭头

当前架构图主要展示包含关系，Agent 间的委派关系（pm→planner→coder→tester）不可见。增加调用箭头可使角色协作关系一目了然。
