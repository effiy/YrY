### 主要价值

- 🔍 模块快速定位 — 通过模块地图在数秒内找到任意模块的入口和上下游关系
- 📊 影响分析可视化 — 变更前通过依赖矩阵一览受影响的全部模块
- 🛡️ 架构漂移感知 — 项目结构变更后可主动检测地图与实际的不一致
- 🤖 诊断基线可消费 — 自改进 Agent 可直接读取模块地图作为 D0/D5 诊断参照

## §2 场景详述

### 场景 A: 新人快速定位模块

| 字段 | 内容 |
|------|------|
| 角色 | 新加入项目的开发者 |
| 触发条件 | 需要了解某个模块（如 rui-import）的职责、入口和上下游 |
| 核心目标 | 在 5 分钟内找到 rui-import 的入口文件、依赖的模块、以及哪些模块依赖它 |

```mermaid
flowchart TD
    START["打开模块地图文档"]:::action --> LAYER["定位到技能层表格"]:::action
    LAYER --> FIND["找到 rui-import 行"]:::action
    FIND --> READ["查看入口文件列<br/>查看核心依赖列<br/>查看下游消费者列"]:::action
    READ --> CHECK{"信息足够理解模块?"}
    CHECK -->|"是"| DONE["完成定位<br/>知道入口·依赖·消费者"]:::done
    CHECK -->|"否"| DIG["查看依赖矩阵<br/>追踪上下游链条"]:::action
    DIG --> DONE
    DONE --> DEEP["如需深入<br/>点击入口文件链接阅读源码"]:::action

    classDef action fill:#e8e8ff,stroke:#8888cc
    classDef done fill:#ccffcc,stroke:#00cc00
```

| # | 步骤 | 输入 | 系统响应 | 异常分支 |
|---|------|------|---------|---------|
| 1 | 打开模块地图 | 文档路径 | 显示四层拓扑总览 | 文档不存在 → 提示"模块地图尚未生成，执行 /rui doc --from-code module-map" |
| 2 | 按模块名查找 | 模块名 "rui-import" | 高亮匹配行，显示入口/依赖/消费者 | 模块名不匹配 → 显示全部模块列表供浏览 |
| 3 | 追踪依赖链 | 目标模块的依赖列表 | 在依赖矩阵中显示该模块的上下游关系 | 依赖矩阵未生成 → 仅显示表格中的依赖关系 |
| 4 | 点击入口文件链接 | 文件路径 | 跳转到对应源码文件 | 链接失效（文件已移动/删除）→ 标红提示"路径可能过时" |

### 场景 B: 跨模块变更影响分析

| 字段 | 内容 |
|------|------|
| 角色 | 项目维护者 |
| 触发条件 | 计划修改某个模块（如 rui-import 的 sync.mjs），想了解会影响哪些下游 |
| 核心目标 | 在做出变更前，获取完整的影响范围清单，评估变更风险 |

```mermaid
flowchart TD
    START["确定要修改的目标模块"]:::action --> TABLE["在模块地图中定位该模块"]:::action
    TABLE --> CONSUMERS["读取'下游消费者'列<br/>列出全部依赖此模块的模块"]:::action
    CONSUMERS --> MATRIX["交叉查看依赖矩阵<br/>确认二级影响链"]:::action
    MATRIX --> ASSESS{"影响范围评估"}
    ASSESS -->|"影响 ≤ 2 个模块"| SAFE["低风险变更<br/>通知下游负责人"]:::done
    ASSESS -->|"影响 > 2 个模块"| WARN["需仔细评估<br/>逐模块检查兼容性"]:::warn
    ASSESS -->|"空状态: 无下游消费者"| EMPTY["该模块无下游依赖<br/>自由变更"]:::done

    classDef action fill:#e8e8ff,stroke:#8888cc
    classDef done fill:#ccffcc,stroke:#00cc00
    classDef warn fill:#ffffcc,stroke:#ccaa00
```

| # | 步骤 | 输入 | 系统响应 | 异常分支 |
|---|------|------|---------|---------|
| 1 | 指定目标模块 | 模块名 | 显示该模块的入口/依赖/消费者 | 模块名不存在 → 列出相似模块名供选择 |
| 2 | 查看下游消费者 | — | 列出全部直接依赖此模块的模块 | 消费者列表为空 → 显示"该模块无下游依赖，可自由变更" |
| 3 | 追踪二级影响 | 下游模块列表 | 在依赖矩阵中标注受影响的下游模块的消费者（二级影响） | 依赖矩阵数据不完整 → 仅显示一级影响，标注"数据可能不完整" |
| 4 | 记录影响分析结论 | 影响范围 | 将分析结果写入变更记录 | 无下游影响 → 仅记录"无影响" |

### 场景 C: 模块地图一致性校验

| 字段 | 内容 |
|------|------|
| 角色 | 项目巡检者（或自改进 Agent） |
| 触发条件 | 检测到项目目录结构可能已变更（新增/删除/重命名模块），或定期巡检 |
| 核心目标 | 发现模块地图与实际项目结构的差异，获取修复建议 |

```mermaid
flowchart TD
    START["触发一致性校验"]:::action --> SCAN["扫描项目实际目录<br/>skills/ agents/ rules/ .memory/"]:::action
    SCAN --> COMPARE["对比模块地图中的模块清单<br/>与实际目录中的模块清单"]:::action
    COMPARE --> RESULT{"发现差异?"}
    RESULT -->|"新增模块"| NEW["记录：模块地图缺失 X<br/>建议补充映射"]:::warn
    RESULT -->|"模块已删除"| DEL["记录：模块地图含已删除的 Y<br/>建议移除或标注废弃"]:::warn
    RESULT -->|"路径变更"| MOVE["记录：模块 Z 路径变更<br/>旧路径→新路径"]:::warn
    RESULT -->|"空状态: 无差异"| OK["模块地图与实际结构一致"]:::done
    NEW --> REPORT["生成漂移报告<br/>含修复建议"]:::action
    DEL --> REPORT
    MOVE --> REPORT
    REPORT --> FIX["执行修复: /rui update module-map"]:::action
    FIX --> VERIFY["重新校验确认一致"]:::done

    classDef action fill:#e8e8ff,stroke:#8888cc
    classDef done fill:#ccffcc,stroke:#00cc00
    classDef warn fill:#ffffcc,stroke:#ccaa00
```

| # | 步骤 | 输入 | 系统响应 | 异常分支 |
|---|------|------|---------|---------|
| 1 | 触发校验 | 校验命令 | 扫描 skills/ agents/ rules/ 目录 | 目录不可访问 → 报告"无法读取目录"，跳过该层 |
| 2 | 模块清单对比 | 实际目录 vs 地图清单 | 逐项比对，标记新增/缺失/路径变更 | 地图中模块名与实际目录名大小写不一致 → 标记为"疑似匹配" |
| 3 | 输出漂移报告 | 对比结果 | 列出全部差异项，每项附修复建议 | 零差异 → 显示"模块地图与项目结构一致" |
| 4 | 执行修复 | 漂移报告 | 通过 /rui update module-map 更新模块地图 | 修复后仍有差异 → 人工介入确认是否为结构性变更 |

### 场景 D: 自改进 Agent 消费模块地图

| 字段 | 内容 |
|------|------|
| 角色 | self-improve Agent（自动化角色） |
| 触发条件 | /rui yry 全量扫描阶段，需要模块结构基线数据 |
| 核心目标 | 从模块地图提取结构化数据，为 D0（基线偏离）和 D5（依赖退化）诊断提供参照 |

```mermaid
flowchart TD
    START["/rui yry 全量扫描"]:::action --> READ["读取模块地图<br/>解析四层模块清单"]:::action
    READ --> D0["D0 基线偏离诊断<br/>模块地图覆盖范围 vs 实际结构"]:::action
    READ --> D5["D5 依赖退化诊断<br/>依赖矩阵中的循环依赖<br/>或依赖深度异常"]:::action
    D0 --> D0RES{"发现偏离?"}
    D0RES -->|"是"| D0FIX["生成改进提案<br/>更新模块地图"]:::warn
    D0RES -->|"空状态: 无偏离"| D0OK["D0 诊断通过"]:::done
    D5 --> D5RES{"发现退化?"}
    D5RES -->|"是"| D5FIX["生成改进提案<br/>建议重构依赖"]:::warn
    D5RES -->|"空状态: 无退化"| D5OK["D5 诊断通过"]:::done

    classDef action fill:#e8e8ff,stroke:#8888cc
    classDef done fill:#ccffcc,stroke:#00cc00
    classDef warn fill:#ffffcc,stroke:#ccaa00
```

| # | 步骤 | 输入 | 系统响应 | 异常分支 |
|---|------|------|---------|---------|
| 1 | 加载模块地图 | 模块地图文档 | 解析四层表格，构建模块清单和依赖关系数据结构 | 文档不存在 → D0/D5 诊断跳过，标注"模块地图缺失" |
| 2 | D0 基线对比 | 模块清单 + 实际目录扫描 | 比对各层模块数量、入口路径，标记偏离 | 某层目录不可读 → 该层跳过，记录 Level C |
| 3 | D5 依赖分析 | 依赖矩阵 | 检查循环依赖、依赖深度、孤儿模块 | 依赖矩阵数据不完整 → 仅对已有数据检查，标注范围 |
| 4 | 生成诊断结论 | 对比结果 | 将有偏离/退化的项转为改进提案 | 无偏离无退化 → 记录"模块地图健康" |
