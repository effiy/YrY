🗺️ 七步上手路径 建议严格按序

每步约 4 分钟，全程 ~30 分钟。每步有明确的入口文档和可验证的完成标志，可中断可恢复。

STEP 1 · 4 min 阅读项目入口
打开 CLAUDE.md，理解项目画像、四大铁律、领域语言指引。知道"项目是什么"和"不可妥协什么"。
→ 阅读项目级指令文档
✅ 完成标志：能用三句话向他人介绍本项目
STEP 2 · 5 min 了解系统全景
阅读 README.md，掌握六能力 · 八角色 · 管线十一阶段。理解"系统由什么组成"和"它们如何协作"。
→ 重点看 20 术语表和架构示意图
✅ 完成标志：能解释 6 能力 / 8 角色 / 11 阶段各自含义
STEP 3 · 4 min 学习角色拓扑
打开 agents/AGENT.md，理解 PM/Planner/Architect/Coder/Reviewer/Tester/Reporter/Security/Self-Improve 九类角色的边界和交接信号。
→ 重点看 ADR 段（架构决策记录）
✅ 完成标志：知道变更触发哪个角色、谁负责审查
STEP 4 · 5 min 掌握管线纪律
精读 rules/code-pipeline.md，理解分支隔离、Gate A/B、逐模块清零的工程纪律。知道"如何不破坏系统"。
→ 重点看 Gate 判定矩阵和"四不妥协"清单
✅ 完成标志：能列出 5 个核心门禁和对应的修复路径
STEP 5 · 3 min 熟悉核心技能
浏览 skills/rui/SKILL.md，了解编排器如何把需求转化为可执行管线。学习调用 /rui 命令族。
→ 跑一遍 node skills/rui/help.mjs 验证帮助输出
✅ 完成标志：能说出 8 条命令的用途
STEP 6 · 6 min 运行初始化 + 测试基线
执行 node tests/run.mjs，确认 7 组测试全部通过。基线即建立——后续变更都要保证不退化。
→ 若失败：检查 .claude/ 目录、Node 版本、依赖完整性
✅ 完成标志：测试输出 PASS/7 套件 · 0 失败
STEP 7 · 8 min 完成首个 PR
从 feat/<task> 分支开始 → 编写/修改代码 → 本地测试 → 文档同步 → 企微通知 → 合并。完整跑通一次"小变更 + 全管线"。
→ 推荐任务：修正一个 typo、新增一条测试用例
✅ 完成标志：PR 通过 Gate A/B + 同步 + 通知全部触发

💡 新人常见误区 避坑指南

⚠️
不要直接读代码，先读规约
新人常见错误是直接打开 .mjs 文件"看实现"，但本系统的核心是规约驱动的——先读 SKILL.md / AGENT.md / rules/*.md 理解"为什么这样设计"，再看代码才能事半功倍。
🚫
不要跳过 Gate A 直接进 Gate B
Gate A（设计/测试）未通过就提交 Gate B（实现/验证）会触发"管线跳跃"阻断，必须先有"设计可验证 + 测试覆盖断言"，再写实现。
✅
利用故事面板追踪进度
每个变更都有对应的故事卡片（位于 docs/故事任务面板/），开发过程中应同步更新状态、留痕决策、附上测试报告——这是"可追溯"的基础。

🐞 排错速查 高频问题

🔧
测试运行时缺少模块
症状：Cannot find module 'xxx'。修复：
npm install && node tests/run.mjs
⏱
Node 版本不匹配
症状：SyntaxError: Unexpected token 或 ESM 加载失败。本项目要求 Node ≥18。
node --version && nvm use 18
🔒
企微通知未触发
症状：WECHAT_WEBHOOK 环境变量缺失。检查项：
echo $WECHAT_WEBHOOK && ls .env*
🧭
不确定该改哪个文件
使用知识图谱反向查询：先看"我要改的功能属于哪个能力" → 再看"该能力由哪个模块实现"。
open 知识图谱.html
📐
架构图渲染异常
症状：Mermaid 图不显示。本系统使用暗色主题，详见 rules/mermaid-theme.md。
cat rules/mermaid-theme.md
🔄
修改后测试仍报旧错
症状：明明改过了，但测试还是报旧的错误。可能是缓存未刷新。
rm -rf node_modules/.cache && node tests/run.mjs
