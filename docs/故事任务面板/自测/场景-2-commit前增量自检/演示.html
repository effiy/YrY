git diff 变更范围检测
识别所有变更文件，确认变更范围与故事计划一致。超范围文件标记异常需解释。
git diff --stat origin/main📋
💡 说明: git diff --stat 是增量自检的第一步 — 先搞清楚改了什么，再决定检查什么。变更范围验证防止「顺手改」问题: 开发者在修一个文件时顺手改了无关文件。变更清单决定了后续各项增量检查的范围。
✅ 结果: git diff --stat origin/main — 变更文件 12 个 (8 演示.html + 1 知识图谱.json + 1 plugin.json + 1 SKILL.md + 1 constants.mjs) → 故事计划预期: 12 文件 → 范围匹配 ✅ / 0 超范围文件 / 变更行数: +854 -42。
变更文件 marker 完整性
验证变更文件中的 rui marker start/end 对是否完整。marker 缺失或不成对 = P0。
for f in $(git diff --name-only origin/main); do echo "=== $f ===" && grep -n "<!-- rui:" "$f" 2>/dev/null || echo "(无 marker)"; done/dev/null || echo "(无 marker)"; done')">📋
💡 说明: rui marker 是退化对策 L2 重生机制的核心组件。每个 marker 段以 <!-- rui:start:name --> 开头、<!-- rui:end:name --> 结尾。增量自检只扫描本次变更涉及的文件中的 marker 完整性（而非全量扫描），大幅缩减检查耗时。
✅ 结果: 变更文件 marker 扫描 — 12 文件中: 4 文件含 rui marker (CLAUDE.md: 6 对 / code-pipeline.md: 1 对 / README.md: 2 对 / delivery-gate.md: 1 对) → 10/10 marker 对完整 ✅ / 0 不成对 / 0 残留旧内容。
增量测试执行
仅运行受变更影响的测试。通过分析 diff 文件路径与测试文件的依赖关系裁剪测试范围，避免全量运行。
node skills/rui/audit.mjs --incremental --changed-files "$(git diff --name-only origin/main | tr '\n' ' ')"📋
💡 说明: 增量测试是 commit 前自检的核心 — 只执行受影响测试。裁剪策略: 变更 lib/constants.mjs → 运行所有导入 constants 的模块测试; 变更 SKILL.md → 运行对应技能的规约一致性测试; 变更 .mjs 源码 → 运行该文件的单元测试。对比基线 results.json 检测退化。
✅ 结果: audit.mjs --incremental — 变更文件 12 个 → 依赖分析: 8 演示.html (无测试文件) + constants.mjs (影响 3 测试文件) + SKILL.md (影响 1 测试文件) → 裁剪后: 14 项测试 → 14/14 通过 / 对比基线: 0 回归。
跨文件引用一致性
验证变更不破坏跨文件引用。检查变更文件中的导入/链接/引用路径是否仍然可达。
grep -rn "from\|require\|import\|href=" --include="*.mjs" --include="*.html" . | grep -v node_modules | grep -f <(git diff --name-only origin/main | sed 's|/|\\/|g')📋
💡 说明: 跨文件引用一致性是增量自检中被忽视的高危区域。变更 constants.mjs 的重命名导出可能破坏所有导入方; 变更文件路径可能导致所有引用该文件的 import 失效。检查方法: 提取变更文件的所有跨文件引用 → 验证每个引用路径的目标存在且可访问。
✅ 结果: 引用一致性扫描 — 变更文件 12 个 → 提取跨文件引用 34 条 (import: 8 / require: 5 / href: 12 / CSS url: 9) → 逐条验证目标可达: 34/34 可达 ✅ / 0 断引用 / 0 循环依赖 / 0 路径拼写错误。
安全面增量扫描
对变更文件执行快速安全扫描：密钥泄漏 + XSS 注入点 + 魔法数字 + 输入校验覆盖。
node scripts/security-scan.mjs --incremental --files-from-git-diff📋
💡 说明: 安全面增量扫描是全量安全回归的精简版 — 只扫描变更文件的增量部分（git diff 中的 + 行），而非全量 repo。覆盖四维度: (1) 密钥模式匹配 — API_KEY/TOKEN/SECRET 在新增行中，(2) XSS 模式 — innerHTML/document.write/eval，(3) 魔法数字 — 非 0/1/-1 的裸数字字面量，(4) 输入路径 — process.argv/file read 是否有 sanitize。任何命中 = P0 阻断 commit。
✅ 结果: security-scan.mjs --incremental — 扫描变更 12 文件 / +854 新增行 → 密钥检测: 0 API_KEY/TOKEN/SECRET 模式命中 ✅ / XSS 检测: 0 innerHTML/document.write/eval 命中 ✅ / 魔法数字: 0 非豁免裸数字 ✅ / 输入校验: 所有新增 CLI 参数均有 validate/sanitize 覆盖 ✅ → 安全评分: 100/100 🔒。
