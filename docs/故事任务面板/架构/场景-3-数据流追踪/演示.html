用户输入入口
用户通过 /rui 命令输入需求。pm Agent 首先解析需求，拆分为故事任务，委派给各子 PM。
/rui doc 用户需要登录功能📋
💡 说明: /rui 是所有管线的统一入口。pm 解析自然语言需求后，按「场景-步骤」公式拆分为可独立验证的场景卡片。拆分粒度决定后续 coder 的并行度 — 场景数越多，可并行度越高。
✅ 结果: pm 输出结构化故事卡片 — 故事名「用户登录功能」、标签「auth」、4 个场景拆分（登录界面 / 认证逻辑 / 会话管理 / 错误处理）。每个场景生成独立的任务文档到 docs/故事任务面板/。
文档生成阶段
pm 拆需求 → coder 补齐设计文档 → security 独立审计。产出故事任务.md + 场景文档 + 知识图谱.json。
/rui doc 用户登录 --from-code📋
💡 说明: doc 阶段是管线的「规约层」— 所有后续实现依赖此阶段的输出质量。--from-code 模式从现有源码反推文档（只读），适合存量项目建立文档基线。security 在此阶段注入约束而非事后审计（左移安全）。
✅ 结果: 生成 5 个交付物 — 故事任务.md (主文档) / 场景-N.md x4 (场景规约) / 知识图谱.json (节点+边) / 架构图.html (SVG 拓扑) / 计划清单.html (任务面板)。security Agent 注入安全约束到 coder 的任务列表。
代码实现阶段
分支隔离 → Gate A 测试先行 → 逐模块 P0 清零 → Gate B 验证 ≤ 2 轮。每个模块完成后 P0 必须清零才能进入下一个。
/rui code user-login📋
💡 说明: code 阶段是管线的「实现层」。分支隔离（feat/ 前缀）强制防止直接修改 main。Gate A 在编码前验证准入条件（测试先行、安全约束），Gate B 在交付前验证退出条件（≤2 轮修复）。清先于进铁律：P0 未清不得进入下一模块。
✅ 结果: feat/user-login 分支创建 → Gate A 通过 (测试先行+安全约束注入) → 4 个模块逐次清零 (M1:0 P0 → M2:1 P0→清零 → M3:0 P0 → M4:0 P0) → Gate B 1 轮通过 → merge 到 main。
自改进闭环
扫描所有故事 → D0-D7 诊断 → 选取最优改进项 → 实现 → 验证 → 版本升级。循环至收敛。
/rui yry --depth 3📋
💡 说明: 自改进闭环是管线的「反馈层」— 每轮扫描所有故事面板，运行 D0-D7 诊断（覆盖质量、覆盖盲区、文档漂移、测试缺口、安全回归、依赖老化、性能退化、体验降级），选取效果最优项实现。--depth 控制最大迭代轮数，收敛后自动退出。
✅ 结果: Round 1: 8 诊断 → 3 改进项 → 选取 #1 (测试覆盖率提升) → 实现 → 验证 68→72 测试 → Round 2: 5 诊断 → 2 改进项 → 选取 #1 (文档补全) → 实现 → 验证 → Round 3: 3 诊断 → 0 改进项 → 收敛退出。
交付收口
三步 Hook：追加通知日志 → rui-import 文档同步 → rui-bot 企微通知。每个 hook 独立可验证。
/rui version --up📋
💡 说明: 交付收口三步 hook 是管线的「交付层」。version --up 自动判定语义化版本号（major/minor/patch），更新 plugin.json/CLAUDE.md/README.md 三处版本号，然后顺序执行三个 hook。任一 hook 失败会回滚版本号变更，保证一致性。
✅ 结果: 版本 4.2.0 → 4.3.0: Hook1 通知日志追加 3 条新记录 → Hook2 rui-import 同步 12 个文件到远端 API → Hook3 rui-bot 企微通知发送成功 → commit + tag + push 一气呵成。
数据流可追溯
每阶段输入/输出有明确文件标记。git log + 通知日志 + 测试结果形成完整追溯链。
git log --oneline -5📋
💡 说明: 每条 commit 消息标注阶段类型（doc/code/merge/deliver/repair），git log 形成完整的「谁在什么阶段做了什么」追溯链。配合通知日志（docs/故事任务面板/通知日志.md）和测试结果（tests/results.json），三方交叉验证确保数据流不可篡改。
✅ 结果: 5 条提交记录完整追溯链 — e9d4ad0 (交付: 4.2→4.3) ← b926fc5 (merge: feat/yry-self-test) ← ad6aea1 (实现: 测试加强 68 项) ← 2d45b24 (交付: 4.1→4.2) ← 9d09af8 (自改进: 3-round repair)。
