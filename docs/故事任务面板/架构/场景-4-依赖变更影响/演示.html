6 步走完依赖变更影响

⏱ 总耗时 ~75min 📊 6/6 步骤 🎯 目标: 完成首次 PR
🎯
下一步: 阅读 CLAUDE.md 了解项目总纲
建议先从「项目画像 + 4 铁律」入手，建立全局心智模型。约 15 分钟完成。
了解项目全貌
阅读 README.md 了解系统全景和领域语言（20 个术语）。CLAUDE.md 定义四大铁律和不可妥协底线。
cat README.md CLAUDE.md📋
P0 阻断入门心智模型
💡 说明: 这两个文件是新人必读的「第一小时文档」。README 回答「这是什么」，CLAUDE.md 回答「怎么操作」。领域语言的 Avoid 列特别重要 — 列出了历史中曾使用但已废弃的术语别名，防止新旧概念混淆（退化对策 L2 重生机制）。
✅ 结果: README.md: YrY 全景图 + 20 个领域语言术语定义（含 Avoid 禁用别名列）+ 6 个技能入口 + 角色拓扑概览。CLAUDE.md: 3 基础信念 + 4 铁律 + 项目约束 + 6 不可妥协底线 + 4 层退化对策。
搭建开发环境 ~10 min
克隆仓库 → 配置 API_X_TOKEN 环境变量 → 运行 /rui init 初始化项目基线。
git clone && export API_X_TOKEN=*** && /rui init📋
P0 阻断环境密钥
💡 说明: API_X_TOKEN 必须通过环境变量传入，禁止硬编码在源码或配置文件中（不可妥协底线 #2）。init 命令是幂等的 — 多次运行不会产生重复或冲突，marker 段每次全量重生。首次运行建议在 clone 后立即执行。
✅ 结果: API_X_TOKEN 从环境变量注入（密钥不落盘验证通过）→ /rui init 六阶段全部通过 → detect(6技能+9Agent+8规则) → generate(12个基线文件) → verify(PASS)。所有 rui marker 段已生成。
从源码反推理解 ~5 min
如果项目已有代码但缺少文档，用 --from-code 从源码反推故事文档基线，全程只读不修改。
/rui doc --from-code📋
P1 推荐只读证据 B
💡 说明: --from-code 模式对新人理解存量代码库尤其有用 — 不需要手动写文档，系统从源码自动提取结构和关系，生成人类可读的故事文档。全过程只读（EACCES 检查），不会修改任何源文件。生成文档的证据级别为 Level B（源码路径 + grep 可验证），不如 Level A（运行验证）强但适合快速建立基线。
✅ 结果: 只读扫描 skills/ agents/ rules/ lib/ → 识别 6 技能 + 9 Agent + 8 规则 + 4 共享库 → 生成故事任务.md + 4 场景文档 + 知识图谱.json (含源码路径引用) → 证据 Level B（源码路径可追溯）。
完成首次修改 ~15 min
走完整管线：需求 → doc → code → test → deliver。分支隔离保护，Gate A/B 确保质量。
/rui 修复文档中的错别字📋
P1 推荐T1 变更端到端
💡 说明: 首次修改走完整管线体验全流程。T1 级变更（措辞/格式）自动跳过深度分析，流程更快（约 2 分钟 vs T3 的 15 分钟）。分支隔离策略确保 main 分支始终可交付，feat 分支的实验不会污染主分支。这是理解管线运作的最佳方式 — 做一个真实的小修改。
✅ 结果: pm 识别为 T1 措辞变更 → doc 生成场景卡片（影响范围: 仅 2 个 .md 文件）→ code 创建 feat/fix-typo 分支 → Gate A 通过 → 修改 2 处 → Gate B 1 轮通过 → deliver 合并到 main。
理解管线纪律 ~15 min
阅读 code-pipeline.md 理解分支隔离策略、Gate A/B 关卡、逐模块 P0 清零流程。
cat rules/code-pipeline.md📋
P0 阻断治理必读
💡 说明: 管线纪律是 YrY 最核心的治理机制。逐模块 P0 清零是「清先于进」铁律的直接体现 — P0 代表阻断级问题（安全漏洞/数据丢失/规约不一致），未清不得进入下个模块。不是鼓励完美主义（P1/P2 可以后续迭代），而是防止 P0 积累。
✅ 结果: code-pipeline.md 定义完整管线纪律 — 分支隔离（feat/ 前缀强制 + main 保护）→ Gate A 准入（测试先行 + 安全约束注入 + 分支验证）→ 逐模块 P0 清零（Modules: N→1，Module: M→1 子任务，当前模块 P0 未清 → 阻断下一模块）→ Gate B 交付（≤2 轮修复 + 交付 hook 三步）。
同步团队配置 ~1 min
rui-依赖变更影响 sync 从远端同步 .claude/ 配置，确保团队使用一致的 Agent 和规则。
/rui-依赖变更影响 sync📋
P2 收尾协作最后
💡 说明: sync 确保团队成员使用统一的 Claude Code 配置（权限规则、钩子、MCP 服务器等），避免「在我机器上能跑」问题。首次入职或配置更新后必须运行。
✅ 结果: 远端 .claude/ 配置 → 本地覆盖 — 同步 settings.json / hooks.json / MCP 服务器列表。diff 显示 3 个文件更新，0 冲突。
🎉 上手完成: 完成全部 6 步后，你已经具备独立开发能力。后续可深入 ① 技能源码阅读 ② Agent 行为观察 ③ 提交 P1/P2 改进任务。预计 2 周内可承担 T2 中等复杂度变更。

新人常见踩坑

⚠ 6 大类 🔧 含修复方案
🔐 P0: 硬编码密钥到源码 / 测试文件 HIGH

新人最常见的错误之一。把 API_X_TOKEN 直接写入测试 fixture 或示例代码，期望「反正不会跑」。但 src/ 与 test/ 都会被 grep 扫描，硬编码会触发 Gate A 安全约束检查导致阻断。

症状: /rui code 阶段报「Secret detected in source」阻断。

修复: 1) 立即删除硬编码 2) 用 process.env.API_X_TOKEN 读取 3) 在 .gitignore 加上 .env 4) 如果已提交，git filter-branch 清理历史
🌿 P0: 在 main 分支直接提交 HIGH

main 分支受分支隔离策略保护，所有修改必须通过 feat/ 前缀分支。直接在 main 上 commit 会被分支隔离 hook 拒绝。

症状: git commit 报「Branch protection: main is locked」；/rui code 自动创建 feat/ 分支。

修复: git checkout -b feat/your-feature → 在 feat 分支上提交 → 通过 Gate B → merge 到 main
📚 P1: 跳过阅读 CLAUDE.md 直接跑 /rui init MED

急于上手而忽略前置阅读。init 跑成功了，但不理解 4 铁律和 6 不可妥协底线，后续会在 Gate A/B 反复被阻断。

症状: 提交被反复拒绝，不理解原因；问「为什么要先写测试」。

修复: 回退 30 分钟，按顺序读 README.md → CLAUDE.md → AGENT.md。先建认知再动手，效率反而更高。
💥 P1: 一次提交做太多改动 MED

新人常犯 — 改 typo + 重构函数 + 新增功能放在一个 commit。违反「单一职责」原则，Code Review 困难，Gate B 易失败。

症状: PR 被要求拆分为多个；reviewer 反馈「这个改动太多无法 review」。

修复: 一个 commit 只做一件事。多个改动拆成多个 commit 或多个 PR。YrY 推荐「小步快跑」而非「大爆炸式提交」。
🔄 P1: 跳步 — 直接进入 code 阶段没有 doc 卡片 MED

新人有明确目标后倾向「直接动手」。但缺少场景卡片（doc 阶段产物）会导致 ① 影响范围不清 ② 测试不充分 ③ Gate B 失败。

症状: 改了 N 个文件但只有 1 个测试；reviewer 反馈「请补充场景卡片」。

修复: 严格走管线 — /rui 需求 → doc 自动生成场景卡片 → code 引用卡片 ID → test 覆盖卡片条目 → deliver。
🤷 P2: 卡住时硬撑不求助 LOW

新人卡在某个错误 30+ 分钟不求助是常见反模式。YrY 鼓励「及时同步」（sync 铁律）。

症状: 同样的错误重复尝试 → 疲惫 → 失去信心。

修复: 15 分钟未解决的错误 → 截图+复现步骤 → 在团队频道求助 或 运行 /rui status 让系统给建议。
⚠ 严重程度说明: HIGH = 触发 P0 阻断，需立即修复 · MED = 影响交付质量但不阻断 · LOW = 反模式但可后续改进。修复 HIGH 级错误后再继续后续步骤。
