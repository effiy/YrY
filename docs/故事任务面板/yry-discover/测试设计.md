- 🔍 扫描正确性验证 — 确认路由扫描适配器正确提取前端页面和后端路由

## §2 测试用例

### 2.1 正常用例

| ID | Given | When | Then | 关联 FP | 优先级 |
|----|-------|------|------|--------|--------|
| TC-N1 | Vue Router 项目（含 5 个路由） | 执行 scanFrontend() | 输出 5 个页面，每个含 name、routePath、componentFile | FP2 | P0 |
| TC-N2 | React Router 项目（含 3 个路由） | 执行 scanFrontend() | 输出 3 个页面，路径与组件正确对应 | FP2 | P0 |
| TC-N3 | Express 项目（含 4 个路由） | 执行 scanBackend() | 输出 4 个路由，每个含 method、path、handlerFile | FP3 | P0 |
| TC-N4 | Fastify 项目（含 2 个路由） | 执行 scanBackend() | 输出 2 个路由，METHOD 和 path 正确 | FP3 | P0 |
| TC-N5 | 存在 1 个缺失前端页面 | 执行故事目录创建 | 生成故事目录含 6 文档：故事任务 + 使用场景 + 技术评审 + 页面布局 + 数据流 + 交互流程 | FP5 | P0 |
| TC-N6 | 存在 1 个缺失后端路由 | 执行故事目录创建 | 生成故事目录含 6 文档：故事任务 + 使用场景 + 技术评审 + 数据库设计 + 数据流 + 模型设计 | FP6 | P0 |
| TC-N7 | fullstack 项目，前端页面和后端路由均发现 | 执行全栈关联 | 生成页面→路由关联表，每个页面至少关联 1 个路由 | FP7 | P0 |
| TC-N8 | 项目类型 frontend | detectType() 判定 | 返回 `frontend`，触发前端扫描 | FP1 | P0 |

### 2.2 边界用例

| ID | Given | When | Then | 关联 FP | 优先级 |
|----|-------|------|------|--------|--------|
| TC-B1 | 前端项目全部页面已有故事目录 | 执行缺失检测 | 返回空缺失列表，输出"全覆盖"报告 | FP4 | P1 |
| TC-B2 | 后端项目全部路由已有故事目录 | 执行缺失检测 | 返回空缺失列表，输出"全覆盖"报告 | FP4 | P1 |
| TC-B3 | 项目类型为 meta | detectType() 判定 | 返回 `meta`，跳过页面/路由发现，仅扫描已有故事 | FP1 | P1 |

### 2.3 异常用例

| ID | Given | When | Then | 关联 FP | 优先级 |
|----|-------|------|------|--------|--------|
| TC-E1 | 前端路由配置文件不存在或格式不支持 | 执行 scanFrontend() | 返回空数组，报告"路由格式不支持"，不阻断管线 | FP2 | P1 |
| TC-E2 | 后端 handler 文件被删除（源码不可读） | 为路由生成专属文档 | 文档标注 Level C + "待补充" + 缺失的源码路径，不阻断生成 | FP5, FP6 | P1 |

### 2.4 回归用例

| ID | Given | When | Then | 关联 FP | 优先级 |
|----|-------|------|------|--------|--------|
| TC-R1 | 项目新增一个页面路由后 | 重新执行 yry 全量扫描 | 新页面被检测到并创建故事目录，已有页面故事不变 | FP7, FP8 | P0 |
| TC-R2 | yry 管线升级后（如 §1 步骤调整） | 执行发现步骤 | 发现步骤仍正常执行，输出报告格式不变 | FP8 | P0 |
