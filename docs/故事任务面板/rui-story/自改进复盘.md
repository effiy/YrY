| 安全边界 | 强 — 三道防线 | 强 — 三道防线 | 强 — 纵深防御 | |
| 依赖方向 | 正确 — core → routes | 正确 — rui-story → import-docs | 正确 — 视图 → CDN → API | |

## §4 经验沉淀

| # | 经验 | 类别 | 来源阶段 | 适用范围 |
|---|------|------|---------|---------|
| 1 | CLI 技能优先使用远端 API 作为数据源，本地文件系统作为同步副本 | 模式 | 架构设计 | 依赖远端知识库的 CLI 技能 |
| 2 | 命令精简比功能堆砌更有价值 — 每次移除不必要命令都降低了维护成本 | 原则 | 迭代演进 | 所有 CLI 工具设计 |
| 3 | 双基线模型（问题空间 + 用户空间）使文档独立于实现，便于独立演进 | 模式 | 文档生成 | 所有 story 文档 |
| 4 | sync 委托模式 — 不自实现同步逻辑，完全委托 import-docs | 模式 | 架构设计 | 跨技能/跨服务协作 |
| 5 | 六状态判定用文件存在性而非状态机：确定性、无副作用、易测试 | 模式 | 设计 | 基于文件的进度追踪系统 |
| 6 | 文档版本可能领先于代码实现 — 改代码前先确认文档版本 | 反模式 | 需求解析 | 所有实现任务 |
| 7 | 同一故事目录下多项目文档应保持对称 — 内部一致性降低认知负担 | 原则 | 文档管理 | 多项目共享故事目录 |
| 8 | 文件系统 API 比数据库 API 更简单：目录不存在返回空列表，无需 try/catch | 模式 | 实现 | 文件系统查询类 API |
| 9 | 远端 API 集成模式：httpx 异步查询 + 文件下载，逐文件容错 | 模式 | 设计 | 外部 API 集成 |
