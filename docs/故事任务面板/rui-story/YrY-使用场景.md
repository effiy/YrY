### 主要价值

- 💻 以 CLI 用户视角描述故事面板的完整交互流程 — 从查看到同步，5 个场景全覆盖
- 🧪 每个场景覆盖正常路径、空状态和错误恢复 — 确保终端体验不遗漏
- 👁️ 进度一眼可见 — 在终端中通过文本表格快速判断项目健康度
- 🔗 双基线协同 — 每场景紧密映射 [YrY-故事任务](./YrY-故事任务.md) 的 Story# 和 FP#

## §2 场景详述

### 场景 1: 查看项目整体进度

| 角色 | 触发条件 | 核心目标 | 关联产品需求 |
|------|---------|---------|---------|
| CLI 用户 | 打开终端想快速了解项目中有哪些故事、各自处于什么阶段 | 在极短时间内看到按状态分类的故事计数和最近活动 | [Story 1](./YrY-故事任务.md) · FP1 · FP5 |

```mermaid
flowchart TD
    START["在终端输入 /rui-story"] --> API["查询远端 API sessions 集合"]
    API --> CHK{"有故事吗?"}
    CHK -->|"有"| CLASS["CLI 内存中按六状态判定并分类计数"]
    CLASS --> OUTPUT["输出状态统计表<br/>+ 最近活动故事列表"]
    CHK -->|"无"| EMPTY["输出空面板提示<br/>合计 0 个故事"]
    OUTPUT --> DONE["用户了解项目整体进度"]
    EMPTY --> DONE
```

| # | 步骤 | 输入 | 系统响应 | 异常分支 |
|---|------|------|---------|---------|
| 1 | 触发操作 | 在 Claude Code 中执行 `/rui-story` | 开始查询远端 API | 远端 API 不可达 → 降级提示"API 不可用" |
| 2 | 远程查询 | — | 查询 sessions 集合，筛选故事面板目录 | API_X_TOKEN 未设置 → 提示"请设置 API_X_TOKEN" |
| 3 | 状态判定 | — | 按六状态模型逐一判定每个故事 | 故事目录无任何文件 → 判定为"未开始" |
| 4 | 聚合输出 | — | 显示六状态计数汇总 + 最近修改的故事名称和时间 | 无任何故事 → 显示"最近活动：无" |

#### 六状态判定模型

```mermaid
flowchart TD
    START["检查 story file_path 列表"] --> Q1{"有 01-故事任务?"}
    Q1 -->|"否"| NS["not_started<br/>未开始"]:::s0
    Q1 -->|"是"| Q2{"有 02 + 05 + (03 或 04)?"}
    Q2 -->|"否"| DIP["docs_in_progress<br/>文档进行中"]:::s1
    Q2 -->|"是"| Q3{"有 06 或 07?"}
    Q3 -->|"否"| DD["docs_done<br/>文档完成"]:::s2
    Q3 -->|"是"| Q4{"有 08?"}
    Q4 -->|"否"| CIP["code_in_progress<br/>编码进行中"]:::s3
    Q4 -->|"是"| CD["code_done<br/>编码完成"]:::s4

    classDef s0 fill:#eceff1,stroke:#90a4ae;
    classDef s1 fill:#fff3e0,stroke:#e65100;
    classDef s2 fill:#e8f5e9,stroke:#388e3c;
    classDef s3 fill:#e3f2fd,stroke:#1565c0;
    classDef s4 fill:#c8e6c9,stroke:#2e7d32;
```

| 状态 | 判定规则（基于远端 file_path 存在性） | 含义 |
|------|------|------|
| `not_started` | 01-故事任务.md 不存在 | 故事尚未开始文档化 |
| `docs_in_progress` | 01 存在，但 02/05/(03或04) 不完整 | 文档基线正在建立 |
| `docs_done` | 文档基线齐全，06/07 不存在 | 文档阶段完成，待编码 |
| `code_in_progress` | 06 或 07 存在，08 不存在 | 编码进行中 |
| `code_done` | 08 存在，且未被阻断 | 编码完成 |
| `blocked` | .memory/rui-state.json 含 blocked=true | 被阻断 |

### 场景 2: 浏览所有故事详情

| 角色 | 触发条件 | 核心目标 | 关联产品需求 |
|------|---------|---------|---------|
| CLI 用户 | 需要查看每个故事的详细信息以决定工作优先级 | 在一个表格中看到所有故事的完整状态、文件数、类型和分支信息 | [Story 1](./YrY-故事任务.md) · FP2 · FP5 · FP6 |

```mermaid
flowchart TD
    START["在终端输入 /rui-story list"] --> API["查询远端 API sessions 集合"]
    API --> CHK{"有故事吗?"}
    CHK -->|"有"| COLLECT["CLI 内存中收集每个故事的：<br/>状态·文件数·最后修改时间<br/>·类型·关联分支"]
    COLLECT --> SORT["按最后修改时间<br/>降序排列"]
    SORT --> TABLE["输出六列表格"]
    CHK -->|"无"| EMPTY["输出空表格"]
    TABLE --> DONE["用户了解每个故事的完整信息"]
```

| # | 步骤 | 输入 | 系统响应 | 异常分支 |
|---|------|------|---------|---------|
| 1 | 触发操作 | `/rui-story list` | 开始查询远端 API | 远端 API 不可达 → 降级提示 |
| 2 | 收集信息 | — | 对每个故事收集：名称、状态、文件数量、最近修改时间、类型、关联分支 | API_X_TOKEN 未设置 → 提示设置 |
| 3 | 排序排列 | — | 按最近修改时间从新到旧排列 | 所有故事修改时间相同 → 按名称字母序排列 |
| 4 | 输出表格 | — | 六列表格，每行一个故事 | — |

### 场景 3: 查看单个故事详情

| 角色 | 触发条件 | 核心目标 | 关联产品需求 |
|------|---------|---------|---------|
| CLI 用户 | 需要深入了解某个特定故事的完整信息 | 看到该故事的所有文件、元数据、状态和关联分支的一体化详述卡 | [Story 2](./YrY-故事任务.md) · FP3 |

```mermaid
flowchart TD
    START["在终端输入 /rui-story show &lt;name&gt;"] --> PARSE["解析故事名称"]
    PARSE --> VALID{"名称格式合法?"}
    VALID -->|"否"| ERR1["提示：名称需为<br/>小写字母+连字符格式"]
    VALID -->|"是"| API["查询远端 API"]
    API --> FIND{"远端存在?"}
    FIND -->|"否"| ERR2["提示：该故事不存在"]
    FIND -->|"是"| ENUM["列出所有文件<br/>含大小和修改时间"]
    ENUM --> META["读取状态和类型信息"]
    META --> BRANCH["检查关联分支"]
    BRANCH --> CARD["输出详述卡"]
    CARD --> DONE["用户全面了解该故事"]
```

| # | 步骤 | 输入 | 系统响应 | 异常分支 |
|---|------|------|---------|---------|
| 1 | 触发操作 | 故事名称（kebab-case） | 校验名称格式 | 名称含大写字母或特殊字符 → 报错提示 kebab-case 格式要求 |
| 2 | 远程查询 | — | 查询远端 API 获取该故事的 sessions | 远端不存在该故事 → 报错提示"故事不存在" |
| 3 | 枚举文件 | — | 列出所有远端记录的文件，显示文件名、修改时间 | 无文件记录 → 文件清单显示"—" |
| 4 | 读取元数据 | — | 展示当前阶段、阻断原因（如有） | 无元数据 → 相关字段显示"—" |
| 5 | 检查分支 | — | 展示关联分支名称 | 无关联分支 → 显示"—" |

### 场景 4: 从远端同步文档

| 角色 | 触发条件 | 核心目标 | 关联产品需求 |
|------|---------|---------|---------|
| CLI 用户 | 需要从远端知识库获取最新的故事文档到本地 | 文档成功从远端同步到本地，或获知同步失败的具体原因 | [Story 3](./YrY-故事任务.md) · FP4 · R2 |

```mermaid
flowchart TD
    START["在终端输入 /rui-story sync [name]"] --> Q{"指定了故事名称?"}
    Q -->|"是"| VALID{"名称格式合法?"}
    VALID -->|"否"| ERR1["提示 kebab-case 格式要求"]
    VALID -->|"是"| ONE["委托 import-docs mode=pull<br/>从远端拉取该故事"]
    Q -->|"否"| LIST["展示可同步故事推荐<br/>列出可选故事"]
    ONE --> DELEGATE["子进程执行<br/>node skills/import-docs/sync.mjs"]
    LIST --> WAIT["等待用户选择"]
    WAIT --> DELEGATE
    DELEGATE --> RESULT{"同步结果?"}
    RESULT -->|"成功"| DONE["显示同步成功<br/>和同步的文件数量"]
    RESULT -->|"失败"| ERR["显示错误信息<br/>不吞没错误"]
```

| # | 步骤 | 输入 | 系统响应 | 异常分支 |
|---|------|------|---------|---------|
| 1 | 触发操作 | 可选的故事名称 | 判断是否有名称；有则校验后直接同步，无则展示推荐 | 名称格式非法 → 报错提示 |
| 2 | 推荐/执行 | — | 未指定名称时展示可同步故事列表并等待选择；指定名称时委托 import-docs 执行 | 指定名称不存在 → 报错提示 |
| 3 | 等待结果 | — | 子进程 stdout/stderr 输出同步进度 | 网络故障 → 显示连接错误，建议重试 |
| 4 | 确认结果 | — | 同步成功显示文件数；失败显示原因 | 部分文件同步失败 → 列出失败项 |
