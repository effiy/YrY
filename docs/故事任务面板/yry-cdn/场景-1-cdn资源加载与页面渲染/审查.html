✅ 好案例 CDN 加载链路架构图分层设计

📁 docs/故事任务面板/yry-cdn/场景-1-cdn资源加载与页面渲染/架构图.html : 64-123
SVG 架构图用 4 种色系（cyan / amber / emerald / violet）区分 Cat A 路径、Cat B 路径、共享资源加载和决策分支。从用户打开页面到最终可交互的 5 步流程清晰可见：① Google Fonts（仅 Cat A）→ ② shared.css → ③ Theme CSS 分支 → ④ shared.js → ⑤ 页面可交互。每步含资源文件名和功能描述，决策点用菱形表示页面类型分叉。
💡 为什么好：遵循了表达优先原则（图 → 结构化文本 → 表格）。用颜色编码区分执行路径，使 Cat A（等宽字体/深蓝底）和 Cat B（系统字体/深紫底）两个主题的加载差异一目了然。开发者 30 秒内可建立"页面分两种类型 → 加载路径不同 → 最终汇聚到 shared.js"的心智模型。

✅ 好案例 CSS 设计令牌系统 (14 Design Tokens)

📁 cdn/theme.css : 1-30 · 设计令牌定义段
theme.css 使用 :root 伪类定义 14 个 CSS 自定义属性作为设计令牌：颜色（7 个 — bg / bg-card / bg-flat / bg-raised / text / text2 / text3）、功能色（4 个 — accent / cyan / pass / fail）、间距（shadow / shadow-lg / radius）。所有组件样式通过 var(--yry-*) 引用这些令牌，实现单一来源的主题管理。
💡 为什么好：设计令牌是主题系统的基础抽象。14 个令牌覆盖颜色、阴影、圆角三个维度，theme-mono.css 和 theme.css 各自重定义这些令牌即可切换整个页面的视觉风格，无需修改任何组件 CSS。这为未来扩展更多主题（如高对比度主题、亮色主题）提供了清晰的扩展点。
/* 设计令牌（theme.css 中定义）*/
:root {
  --yry-bg: rgba(22,22,32,1);
  --yry-bg-card: linear-gradient(159deg, rgba(38,38,52,1) 0%, rgba(34,34,46,1) 100%);
  --yry-accent: #FFC107;
  --yry-cyan: #22d3ee;
  --yry-pass: #22c55e;
  --yry-fail: #ef4444;
  /* ... 共 14 个令牌 */}

❌ 坏案例 场景页面 CSS class 前缀未统一

📁 docs/故事任务面板/yry-cdn/场景-1-cdn资源加载与页面渲染/架构图.html : 39-63
架构图页面中同时存在 breadcrumb / bc-sep / bc-current（无 前缀）和 cross-nav / cross-link（有 前缀）两套命名体系。breadcrumb 组件在 shared.css 中定义为 .breadcrumb / .bc-sep / .bc-current，但页面内联 CSS 又重复定义了 .breadcrumb / .bc-sep / .bc-current 规则，导致样式冗余和潜在冲突。
💡 为什么不好：两套命名体系并存增加了维护成本 — 修改面包屑样式时需要同时更新 shared.css（yry- 前缀版）和页面内联 CSS（无前缀版）。更严重的是，未来如果 shared.css 中的 .breadcrumb 样式变更，使用 .breadcrumb 的页面不会自动受益。违背了"CDN 共享前端资源"的核心价值——一处定义，全局生效。
/* ❌ 页面内联 CSS — 重复定义了已有  前缀的样式 */
.breadcrumb { display: flex; ... }        /* 应为 .breadcrumb */
.breadcrumb .bc-sep { ... }              /* 应为 .bc-sep */
.breadcrumb .bc-current { ... }          /* 应为 .bc-current */
.cross-nav { display: flex; ... }        /* 应为 .cross-nav */

/* ✅ shared.css 已定义的正确版本 */
.breadcrumb { ... }
.bc-sep { ... }
.cross-nav { ... }
P1 · 立即

统一所有场景页面 CSS class 为 前缀

将 5 个场景 × 4 个页面类型（审查/测试/演示/计划）的 HTML 中所有 breadcrumb/cross-nav 等原生类名统一替换为 breadcrumb/yry-cross-nav 等 前缀版本。删除页面内联的重复 CSS 规则，完全依赖 shared.css 的 组件定义。预计每个页面 10 分钟，共 200 分钟。

P1 · 立即

补充 shared.js 公共 API 的 JSDoc 注释

为 window.YrY 暴露的 9 个公共方法（toast / copyCmd / switchPanel / toggleSuite / expandAllSuites / collapseAllSuites / fmtDur / esc / clipboardWrite）添加完整的 JSDoc @param @returns @example 注释。同时标注内部私有方法（_getOrCreateToastEl 等）的 @private 可见性。提升 JS API 的可追溯性从 75% 到 85%。

P2 · 本迭代

将 shared.css 中的硬编码值迁移到设计令牌

shared.css 中的 toolbar / progress-wrap / footer-note 等全局组件使用了硬编码颜色值（如 #475569 / #334155）。将这些值替换为 var(--yry-*) 令牌引用，使 Cat A/B 主题切换可以统一控制所有组件。同时将设计令牌从 14 个扩展到 18 个以覆盖缺失的组件场景。

P2 · 本迭代

shared.js IIFE 增加错误边界

当前 shared.js 的 IIFE 中，如果 DOM 未就绪或 CSS 变量未注入，API 调用会静默失败（如 toast 的 DOM 容器创建失败）。增加 try/catch 包裹 + console.warn 降级处理，确保 JS 异常不影响页面 HTML/CSS 的正常渲染。同时在 window.YrY 上暴露 .ready 状态标志供外部检测。

P3 · 未来

实现 CSS 按需加载与代码分割

当前每个页面全量加载 shared.css（5.9KB）+ theme.css（11.3KB）+ shared.js（5.1KB）共计约 22KB。可以按页面类型拆分：Cat A 页面仅加载 theme-mono.css（6.1KB），Cat B 页面仅加载 theme.css（11.3KB），减少约 40% 的 CSS 传输量。同时考虑将 shared.css 中不常用的动画（modalIn / stepIn）拆分为按需加载。
