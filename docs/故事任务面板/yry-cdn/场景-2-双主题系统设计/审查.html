✅ 好案例 设计令牌系统 (14 CSS 自定义属性)

📁 cdn/theme.css : :root 段 · 14 个 CSS 自定义属性
使用 :root 伪类定义 14 个 CSS 自定义属性：颜色 7 个（--yry-bg / --yry-bg-card / --yry-text 等）、功能色 4 个（--yry-accent / --yry-cyan / --yry-pass / --yry-fail）、间距 3 个（--yry-shadow / --yry-shadow-lg / --yry-radius）。所有组件通过 var() 引用令牌，实现"单一来源"的主题管理。
💡 为什么好：设计令牌是主题系统的核心抽象——将样式值从组件解耦，Cat A 和 Cat B 各自重定义 :root 变量即可切换整页视觉风格，无需修改任何组件 CSS。这为未来扩展更多主题（如高对比度主题）提供了清晰的扩展点。
/* 设计令牌架构 */
:root {                        /* ← 14 令牌定义层 */
  --yry-bg: rgba(22,22,32,1);
  --yry-accent: #FFC107;
}
.card {                    /* ← 组件层（仅引用令牌） */
  background: var(--yry-bg-card);
}

✅ 好案例 Cat A/B 主题物理隔离

📁 cdn/theme-mono.css + cdn/theme.css · 两个独立文件
两个主题以独立 CSS 文件实现物理隔离——Cat A 页面仅加载 theme-mono.css (6.1KB)，Cat B 页面仅加载 theme.css (11.3KB)。页面不会同时加载两个主题文件，避免样式冲突和冗余传输。
💡 为什么好：物理隔离是最强的隔离形式。相比 CSS class toggle，独立文件方案确保无两套样式同时生效的中间态，浏览器缓存更精准，每个主题可独立演进。
// Cat A: <link href="theme-mono.css">  (6.1KB)
// Cat B: <link href="theme.css">       (11.3KB)

❌ 坏案例 结构属性在双主题中重复定义

📁 cdn/theme.css + cdn/theme-mono.css · .card 段
yry-card 的 border-radius / padding / transition 等结构属性在两个主题文件中以相同的值重复定义。修改组件布局结构需要同时更新两个文件，增加维护成本和不一致风险。
💡 为什么不好：违反 DRY 原则。结构属性（布局/间距/动画）属于"组件骨架"，应在 shared.css 统一定义。主题文件应只覆盖"视觉皮肤"——颜色/字体/阴影。当前混在一起导致不必要的耦合。
/* 两文件均定义 border-radius: 12px → 应抽取到 shared.css */
.card { border-radius: 12px; padding: 20px 22px; }

❌ 坏案例 令牌→组件映射缺少文档

📁 缺失：令牌映射表（应存在于 theme.css 头部注释）
14 个设计令牌中，哪些令牌被哪些组件使用没有文档记录。开发者需 grep "var(--yry-" 才能了解令牌使用分布，修改令牌值时无法快速评估影响范围。
💡 为什么不好：缺少映射导致：① 修改令牌值时无法快速评估影响范围；② 新组件开发者不知道该使用哪个令牌，可能引入不必要的新令牌导致令牌膨胀。
/* ✅ 应有令牌映射注释：
 * --yry-bg → body, .container (页面背景)
 * --yry-bg-card → .card, .stat, .suite (卡片容器)
 * --yry-accent → .accent, .tab.on, .cross-link.on (强调色) */
