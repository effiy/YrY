## 概述

YrY 项目自动化测试套件——对技能脚本、Agent 规约、规则文件和配置文件建立可执行的自动化测试体系。覆盖核心脚本逻辑、规约结构完整性、配置合法性和 CLI 命令行为。

面向的用户角色：**系统维护者**（变更后快速回归）、**AI Agent**（验证产出正确性）、**CI 管线**（自动化门禁）。

**数据来源**：skills/*.mjs 脚本源码 + agents/*.md 规约结构 + rules/*.md 规则定义 + .claude-plugin/plugin.json 配置。

### 主要价值

- 🔍 **变更安全网** — 每次 skill 脚本修改后自动跑测试，防止回归
- 💬 **规约结构验证** — Agent/Rule 文件必填节完整性自动检查
- 🏷️ **配置合法性** — plugin.json schema + 版本一致性自动校验
- 🚫 **明确的不做清单** — 不测试 Agent 行为（需真实 API 调用）、不生成覆盖率报告阈值外的指标、不模拟完整管线，保持测试聚焦可验证事实

## §1 Story

### Story 1: 核心脚本逻辑测试

作为系统维护者，我想要对 skills/ 下所有 .mjs 脚本的导出函数进行单元测试，以便修改脚本后能快速验证行为正确性。优先级 P0。

#### 功能点与约束

| 类别 | 描述 | 输入/约束 | 输出/校验 | 错误/说明 | 优先级/级别 |
|------|------|----------|----------|----------|------------|
| 功能点 | branch-check.mjs 分支检查逻辑测试 | git 仓库状态 | 正确返回 0/1 退出码 | 非 git 环境时跳过 | P0 |
| 功能点 | recommend.mjs 项目扫描数据采集测试 | 项目目录路径 | JSON 输出含预期字段 | 目录不存在时报错 | P0 |
| 功能点 | help.mjs 帮助输出格式测试 | 各 skill 目录 | 输出含命令列表 | 脚本不存在时跳过 | P1 |
| 业务规则 | 每个 .mjs 脚本至少 1 条冒烟测试 | 脚本数量 | 冒烟测试数 ≥ 脚本数 | 纯库文件无 CLI 入口可豁免 | A |
| 数据约束 | 测试框架 | 枚举 | Node.js 内置 test runner (node:test) 或 vitest | 按项目生态选择 | — |

#### 成功标准

- 🎯 **核心脚本覆盖 100%** — 度量：已测试 .mjs 数 / 总 .mjs 数 · 目标：100% · P0
- ✅ **冒烟测试全部通过** — 度量：通过冒烟测试数 / 冒烟测试总数 · 目标：100% · P0

#### §1.1 User Operations

| 操作 | 触发条件 | 操作步骤 | 预期结果 |
|------|---------|---------|---------|
| 运行全量测试 | 代码变更后 | `node --test tests/` | 所有测试通过，0 失败 |
| 运行单文件测试 | 修改单个脚本 | `node --test tests/<name>.test.mjs` | 该脚本测试全部通过 |

#### 范围边界

**范围内**：.mjs 脚本函数测试、CLI 退出码测试、输出格式验证。**范围外**：集成测试（需真实 API）、性能测试。

### Story 2: 规约结构与配置验证

作为系统维护者，我想要自动化验证 Agent/Rule 文件结构和 plugin.json 配置，以便发现规约退化。优先级 P0。

#### 功能点与约束

| 类别 | 描述 | 输入/约束 | 输出/校验 | 错误/说明 | 优先级/级别 |
|------|------|----------|----------|----------|------------|
| 功能点 | Agent 文件必填节验证（铁律引用 + 职责边界 + 触发条件） | agents/*.md | 缺节列表 | 新增 agent 后同步更新检查项 | P0 |
| 功能点 | Rule 文件 frontmatter + paths 字段验证 | rules/*.md | 缺 frontmatter 列表 | 非标准 frontmatter 格式标注 | P0 |
| 功能点 | plugin.json schema 验证（name/version/description 必填） | .claude-plugin/plugin.json | schema 错误列表 | 字段类型错误标注 | P0 |
| 功能点 | SKILL.md 存在性验证（每 skill 目录必须有 SKILL.md） | skills/*/SKILL.md | 缺失列表 | 新增 skill 目录后同步检查 | P0 |
| 业务规则 | version 字段必须符合 semver | .claude-plugin/plugin.json | 正则 `^\d+\.\d+\.\d+$` | 格式不符时标注 | A |
| 数据约束 | 必填节列表 | 枚举 | 铁律引用 / 职责边界 / 触发条件 / 生效标志 | Agent 和 Rule 必填节不同 | — |

#### 成功标准

- 🎯 **规约完整性检查覆盖全部文件** — 度量：已检查文件数 / 总规约文件数 · 目标：100% · P0

#### §1.1 User Operations

| 操作 | 触发条件 | 操作步骤 | 预期结果 |
|------|---------|---------|---------|
| 运行规约验证 | init 或变更后 | `node --test tests/structure.test.mjs` | 全部文件通过必填节检查 |

#### 范围边界

**范围内**：文件结构完整性、配置合法性、版本格式。**范围外**：内容语义检查、文档质量评分。

## §2 范围边界

### 范围内（做什么）

| 条目 | 关联 FP# | 产品决策依据 |
|------|---------|-------------|
| .mjs 脚本单元测试 | Story1-FP1,FP2,FP3 | 脚本是唯一运行时代码，优先覆盖 |
| Agent/Rule 结构验证 | Story2-FP1,FP2 | 规约退化检测，与 init 验证互补 |
| plugin.json schema 验证 | Story2-FP3 | 版本一致性强制约束 |
| SKILL.md 存在性检查 | Story2-FP4 | 技能定义完整性基线 |

### 范围外（不做什么）

| 条目 | 排除原因 | 产品决策 |
|------|---------|---------|
| **Agent 行为集成测试** | 需要真实 Claude API 调用 | 成本高且不可重复，由 Gate B 手动验证 |
| **性能基准测试** | 非当前阶段需求 | 未来可追加 |
| **E2E 管线测试** | 管线依赖外部 API 和 git 状态 | 隔离到 CI 环境再考虑 |
| **覆盖率报告** | meta 项目代码量小 | 覆盖以冒烟测试完整性为准 |

## §3 AC

| Given | When | Then |
|-------|------|------|
| 所有 .mjs 脚本就绪 | 执行 `node --test tests/` | 全部测试通过，无失败 |
| Agent 文件存在 | 执行结构验证测试 | 所有 agent 必填节完整 |
| plugin.json 已更新 | 执行配置验证 | version 格式正确，必填字段齐全 |
| 新增 skill 目录 | 执行 SKILL.md 存在性检查 | 新目录检测到缺失并报告 |

## §4 风险与假设

### 风险

- ⚠️ **测试依赖 git 状态** — 可能性：M · 影响：M · 缓解：mock git 命令或检测 git 环境后跳过
- ⚠️ **Node.js 版本兼容性** — 可能性：L · 影响：M · 缓解：使用 Node 18+ 内置 test runner

### 假设

- 💡 **Node.js ≥ 18 可用** — node:test 为内置模块
- 💡 **Git 仓库可操作** — branch-check.mjs 测试需要 git 上下文
