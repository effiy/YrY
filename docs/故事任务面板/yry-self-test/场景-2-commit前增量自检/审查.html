✅ 好案例 git diff 驱动的增量聚焦设计

📁 docs/故事任务面板/yry-self-test/场景-2-commit前增量自检.md : §0
与全量检查不同，增量自检精准聚焦于 git diff --name-only 范围内的变更文件，避免了全量扫描的性能浪费。变更范围确定后仅对变更文件执行铁律检查。
💡 为什么好：增量聚焦是"惜注意"原则在自检体系中的落实——只检查变更的部分。在大型项目中，全量扫描成本不可接受，增量检查是唯一的可扩展方案。

✅ 好案例 分支隔离检测覆盖边界情况

📁 docs/故事任务面板/yry-self-test/场景-2-commit前增量自检.md : 分支隔离检测
检测覆盖了跨分支文件共享的边界情况（memory/、.claude/ 缓存），恰好是"分支隔离不可绕过"底线的具体执行点。检测逻辑与规则定义紧密对齐。
💡 为什么好：边界情况是安全检测中最容易遗漏的。本场景精准命中了"记忆/缓存文件跨分支共享"这一高频风险点，使分支隔离从原则变成可执行的检查。

❌ 坏案例 铁律违规定位缺少证据

📁 docs/故事任务面板/yry-self-test/场景-2-commit前增量自检.md : 铁律判定
铁律合规判定仅输出"通过/失败"——失败时不告知具体违规的文件和行号。例如"验先于称"违规时，用户需要知道哪个文件哪一行做了无证据的声称才能高效修复。
💡 为什么不好：没有证据的失败判定是不可操作的。"有铁律违规"的信息量几乎为零——用户需要在数百个文件中手动搜索违规点。精准定位到文件:行号才能让修复路径可操作。

❌ 坏案例 无 commit message 格式校验

📁 docs/故事任务面板/yry-self-test/场景-2-commit前增量自检.md : 全局
增量自检不检查 commit message 是否符合规范（是否含 Co-Authored-By、是否遵循 conventional commits）。在 YrY 的交付纪律中，commit message 是交付收口的一部分。
💡 为什么不好：commit message 是交付审计的基础。格式不一致会导致自动化工具无法解析变更历史。Co-Authored-By 缺失则违反项目自身规范。
P1 · 立即

增强铁律违规证据输出

失败时输出具体位置（文件:行号）和证据摘要。如"验先于称违规：docs/index.html:45 声称全部通过但无验证命令输出"。预计 1 小时。

P2 · 本迭代

增加 commit message 格式校验

验证是否包含 Co-Authored-By 签名（项目规范），警告级别。

P3 · 未来

增加 pre-commit hook 集成指南

说明如何将增量自检集成为 Claude Code hook 或 git pre-commit hook。
