✅ 好案例 零假阳性容忍设计

📁 docs/故事任务面板/yry-self-test/场景-1-init后全量自检.md : §0 判定逻辑
自检判定基于可复核的存在性证据（标记存在/目录存在/文档存在），不存在"可能通过"的模糊状态。每项检查的输出只有三种：✅ 通过 / ❌ 阻断 / ⚠️ 警告。
💡 为什么好：假阳性是自动化检查的头号杀手——它会侵蚀团队对检查体系的信任，导致"狼来了"效应。本场景用存在性证据替代概率判断，从根本上消除假阳性风险。

✅ 好案例 阻断/警告分级设计

📁 docs/故事任务面板/yry-self-test/场景-1-init后全量自检.md : 自检流程图
七项检查中五项为阻断（❌），两项为警告（⚠️）——架构基线和自检基线的缺失不阻断 init 但给出提醒。这平衡了"严格性"和"可用性"：核心骨架必须完整，扩展基线可以后续补齐。
💡 为什么好：全量阻断会让初始化流程过于脆弱——任何可选模块的缺失都会阻止用户启动。分级设计让必须有的和最好有的各得其所。

❌ 坏案例 缺失 memory/ 和 .claude/ 健康检查

📁 docs/故事任务面板/yry-self-test/场景-1-init后全量自检.md : 七项检查清单
当前七项检查未覆盖 memory/ 目录（执行记忆）和 .claude/ 配置（settings.json、workflows/）。这些是 AI Agent 行为一致性的关键基础设施，其损坏或缺失会导致 Agent 决策漂移而无人察觉。
💡 为什么不好：自检体系的盲区会变成系统的盲区。memory/ 和 .claude/ 对 Agent 行为的影响是隐性的——不像代码错误那样立即可见，但长期影响更深远。
