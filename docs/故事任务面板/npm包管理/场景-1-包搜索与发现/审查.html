✅ 优势 三层错误处理覆盖

📁 skills/rui-npm/rui-npm.mjs : 120-140
参数为空 → 搜索失败 → JSON 解析失败，每层有明确的用户提示和恢复路径。关键词为空时输出用法说明 + 示例命令，JSON 解析失败时输出"解析搜索结果失败"而非崩溃堆栈。
💡 为什么好：遵循"不沉默失败"原则。每层错误都是可操作的信息，用户不需要阅读源码就能理解发生了什么以及如何恢复。这是 CLI 工具可用性的标杆模式。
// 三层错误处理模式
if (!keyword) → 提示用法 + 示例
catch (search error) → 引导手动访问 npmjs.com
catch (JSON parse error) → 明确报错不崩溃

✅ 优势 降级设计不阻断用户流程

📁 skills/rui-npm/rui-npm.mjs : 128-131
网络不可达时降级输出手动搜索 URL（npmjs.com），不阻断用户流程。用户仍可通过浏览器访问 registry 完成包发现。
💡 为什么好：降级不是"放弃"，而是"换条路"。输出可点击的 URL 让用户在工具不可用时仍有替代路径。这是弹性的正确实现——不是 fallback to nothing，而是 fallback to a viable alternative。

✅ 优势 双格式输出设计

📁 skills/rui-npm/rui-npm.mjs : 148-151
默认表格输出（人类可读）+ --json 标志（机器可读），下游 CI/脚本可直接集成。排序在 JSON 解析后本地执行，无额外网络开销。
💡 为什么好：遵循"默认人类友好，flag 机器友好"的 CLI 设计惯例。表格 5 列（名称/描述/版本/下载量/更新时间）信息密度适中，JSON 模式保留原始数据完整性。

✅ 优势 按周下载量降序排列

📁 skills/rui-npm/rui-npm.mjs : 142-146
搜索结果按周下载量降序排列，用户优先看到主流包。排序在本地执行，不依赖 registry 返回顺序。
💡 为什么好：默认排序策略降低了用户的决策成本。面对 npm registry 上 200 万+ 包，下载量是社区信任的信号——优先展示被广泛使用的包，减少 typosquatting 风险。

🛡️ Low Risk 关键词注入防护

📁 skills/rui-npm/rui-npm.mjs
spawnSync 参数数组化传递，不经过 shell 解析。特殊字符（如 @、/、空格）不会导致命令注入。
🔒 缓解：参数以数组形式传递给 spawnSync，shell 模式为 false（默认值）。即使关键词含 `; rm -rf /` 也不会被解释为 shell 命令。

🛡️ Low Risk 恶意 registry 返回伪造数据

📁 skills/rui-npm/rui-npm.mjs
使用 npm 官方 registry；用户可通过 info 子命令二次确认包的元数据。搜索结果中展示下载量作为可信度信号。
🔒 缓解：npm 官方 registry 是可信源；info 命令可直接查询单个包的完整元数据，绕过搜索结果的缓存层。

🛡️ Low Risk 搜索关键词泄露敏感信息

📁 skills/rui-npm/rui-npm.mjs
仅查询公开 registry，不传输本地文件内容。搜索关键词仅发送到 npm registry API。
🔒 缓解：无本地文件泄露风险；search 子命令不读取任何本地文件。网络请求仅包含搜索关键词。

🛡️ Low Risk CDN 包名注入防护

📁 skills/rui-npm/rui-npm.mjs : 502-520
spawnSync 参数数组化传递，npm view 先验证包存在性再生成 URL。恶意包名不会导致命令注入。
🔒 缓解：参数以数组形式传递给 spawnSync；生成的 CDN URL 为 HTTPS 协议，用户可在浏览器中验证证书。

🛡️ Low Risk CDN URL 中间人攻击

📁 skills/rui-npm/rui-npm.mjs : 527-543
所有 CDN URL 使用 HTTPS 协议 (unpkg/jsDelivr/esm.sh)。不依赖外部 CDN API——URL 为字符串模板生成。
🔒 缓解：HTTPS URL 确保传输加密；URL 为本地字符串拼接生成，不向 CDN 服务发起额外 HTTP 请求验证。
