实现链 · 5 角色

主线
  • coder — RED→GREEN 实现
  • tester — Gate A RED 验证
  • reviewer — Gate B ≤ 2 轮
  • doc-writer — 文档同步
  • deliverer — 收口与通知
触发:任务卡就绪

闭环链 · 2 角色

横切
  • self-improve — 反馈聚合,驱动下一轮
  • security — 独立审计,无业务耦合
触发:任意阶段
ID门禁触发者通过条件等级失败升级路径
Gate A RED 验证 tester 失败用例先于实现,node tests/run.mjs 全绿 Hard 修复 → 重提 · 不允许跳过
Gate B 评审收敛 reviewer 反馈 ≤ 2 轮,无 P0 遗留 Medium 超限 → 升级 architect 介入
Gate C 安全审计 security 无明文凭据/注入/依赖漏洞 Hard 立即回滚 + P0 修复任务
Gate D 交付收口 deliverer 日志+同步+通知 三件套齐备 Soft 补齐缺失项后重提

通过策略

P0
  • 每道门禁独立判定,不互相抵消
  • Hard 阻断必须修复,Medium 可降级处理
  • 记录阻断原因,加入下一轮 self-improve 输入
独立可追溯

常见阻断

TOP 3
  • 补写测试 — 43% 的 Gate A 失败源自此
  • 评审分歧 — 32% 的 Gate B 超轮源自命名/边界
  • 凭据泄露 — 17% 的 Gate C 来自示例代码残留
P0 阻断复盘
M1 · 能定位任意模块 ≤ 30 min
完成标志:回答三个问题
某个模块在哪?它的上下游是谁?变更它会影响到谁?做到这一步说明已建立 全局认知地图。来源:README.md + docs/故事任务面板/架构/ 浏览。
M2 · 跑通 node tests/run.mjs ≤ 2 h
完成标志:7 组测试套件全绿
从零拉代码 → 装依赖 → 跑通测试,说明 环境就绪。可承担 D0 级别任务(文档修正、注释补全)。
M3 · 提交首个 PR ≤ 当日
完成标志:通过 1 轮 reviewer 评审
建议从 D0 级别任务 开始(改动 < 50 行,无接口变更)。首 PR 走 双 reviewer 强化。
M4 · 连续通过 3 道门禁 ≤ 3 天
完成标志:Gate A + B + C 全部 pass
具备独立完成 D1 级别任务 的能力(单文件新增 + 测试覆盖)。开始进入主模块开发循环。
M5 · 独立负责 1 个模块 ≤ 1 周
完成标志:能描述模块完整依赖图与变更影响
从 被动接任务 转向 主动发现改进点。可参与 FP 拆分、code review 反向给建议。
↻ 持续 · 反馈驱动自改进 持续
完成标志:每轮 self-improve 至少 1 条有效反馈
从 个人贡献者 转向 系统改进参与者。每完成一次变更,主动复盘至少 1 条改进点。

自检问题

每阶段
  • M1:能在 30 秒内找到任意能力模块的入口?
  • M2:测试失败的根因能在 5 分钟内定位?
  • M3:首个 PR 的 commit 信息能讲清 what+why?
  • M4:能列出当前任务涉及的所有上游模块?
  • M5:能独立拆解一个 FP 的子任务?

常见误区

避坑
  • ❌ 跳过 M1 直接看代码 → 迷失在细节
  • ❌ 提交前不跑测试 → Gate A 必失败
  • ❌ 评审反馈未理解就修复 → 反复打回
  • ❌ 交付缺日志/同步 → Gate D 阻断
