DevTools 6 步验证 CDN 加载链路

打开 Network 标签观察瀑布流
F12 → Network → 勾选 "Disable cache" → 刷新。观察 shared.css → theme.css → shared.js 加载时序。
预期：CSS 资源并行请求（HTTP/2 多路复用），shared.js 在 CSS 之后加载。首次 ~50-150ms，缓存命中 0ms（304 或 disk cache）。
验证 HTTP 状态码与 Content-Type
点击每个 CDN 资源 → Headers → 确认 Status 200，Content-Type 分别为 text/css 和 application/javascript。
curl -sI http://localhost:8080/cdn/shared.css | grep -E "HTTP|Content-Type"
# HTTP/1.1 200 OK
# Content-Type: text/css📋
同样验证 theme.css（text/css）和 shared.js（application/javascript）。
检查加载顺序正确性
Network 标签按 Start Time 排序。shared.css 最先，theme.css 紧随，shared.js 最后。CSS 并行不阻塞，JS 执行等 CSS 解析完成。
Cat B 链：shared.css → theme.css → shared.js（CSS 间无依赖，CSS 先于 JS 是 HTML 中 <link> 在 <script> 之前的自然结果）。
Console 验证 YrY 全局对象
Console 标签输入 YrY 回车，确认返回对象存在。展开查看 9 个 API 方法。
// 在 Console 中执行：
Object.keys(YrY).join(', ')
// → "toast, copyCmd, switchPanel, initSuiteToggle, expandAllSuites, collapseAllSuites, fmtDur, esc, clipboardWrite"

typeof YrY.toast  // → "function"📋
模拟资源阻断验证降级
Network → 右键 theme.css → "Block request URL" → 刷新。页面回退为浏览器默认样式但内容完整。验证后取消 Block。
再试 shared.js：阻断后 Toast / Tab 切换失效，但 CSS 和 HTML 不受影响。证明降级策略有效。
Lighthouse 性能审计
Lighthouse 标签 → "Analyze page load" → 查看 Performance 分数、FCP、LCP、TBT。
预期：Performance ≥ 95。CDN 资源总计 ~23KB（Cat B），不应触发 "Reduce unused CSS/JS" 警告。
