✅ 好案例 内联代码减少 60%+

📁 docs/故事任务面板/cdn/场景-1-*/*.html · 迁移前后对比
迁移前平均每页约 200 行 HTML（含 150 行内联 CSS），迁移后降至约 80 行（含 5-10 行场景特有样式 + CDN 引用）。55 个页面合计减少内联代码约 8,000 行，CSS 体积从 ~15KB/页降至 ~5KB/页（CDN 缓存共享）。
💡 为什么好：CDN 迁移的核心价值——"一处定义，全局生效"。修改 card 样式只需更新 shared.css 一处，55 个页面全部自动受益。相比迁移前需要逐页修改内联 CSS，维护效率提升 50 倍以上。
/* 迁移前：每页 150 行内联 CSS */
<style>.card{background:...;border:...;padding:...}...</style>

/* 迁移后：3 行 CDN 引用 */
<link rel="stylesheet" href="../../../../../cdn/shared.css">
<link rel="stylesheet" href="../../../../../cdn/theme.css">
<script src="../../../../../cdn/shared.js"></script>

✅ 好案例 统一 命名空间

📁 cdn/shared.css · 全局 class 命名规范
所有 CDN 组件使用 前缀（yry-breadcrumb / cross-nav / header / card / tabs 等），与其他库（Bootstrap / Tailwind）的 class 零冲突。命名空间隔离使得 CDN 样式可以安全地引入任何 HTML 页面而不影响该页面的其他样式。
💡 为什么好：命名空间是 CSS 可维护性的基础。yry- 前缀使得开发者一眼就能区分"这是 CDN 提供的组件"还是"页面特有的样式"。在大型项目中，命名空间可以防止不同团队/模块间的样式冲突。

✅ 好案例 Google Fonts 统一管理

📁 所有 Cat A 页面 · 共享同一个 Google Fonts 引用
JetBrains Mono 字体链接集中在 Cat A 页面（架构图/知识图谱），Cat B 页面使用系统字体栈。避免了 55 个页面各自声明字体链接导致的不一致和冗余请求。
💡 为什么好：统一字体管理确保品牌一致性——所有 Cat A 页面的代码块/数字使用相同的等宽字体，所有 Cat B 页面的正文使用相同的系统字体。迁移前有些页面使用 'SF Mono'，有些使用 'Fira Code'，现在全部统一。

❌ 坏案例 迁移过程缺少自动化校验

📁 缺失：页面迁移后自动化回归测试脚本
页面迁移后需要手动打开浏览器逐个验证渲染效果——55 个页面纯手动检查工作量大且易遗漏。缺少自动化工具如视觉回归测试（Percy/Chromatic）或至少一个 Puppeteer 脚本来截图对比迁移前后差异。
💡 为什么不好：没有自动化校验，任何 CDN 样式变更都可能意外影响已迁移页面，但不会被及时发现——直到用户反馈"页面样式坏了"才发现。自动化视觉回归测试是 CDN 共享样式方案的安全网。

open 测试面板.html → 检查样式 → 关闭
