💡 演示 1 · 6 步迁移路径演练

以 审查.html 页面为例，演示完整 6 步流程：

  1. 审计内联 — 找到 200 行内联 <style>，识别 .card/.header/.nav 三个重复模式
  2. 匹配组件 — .card → yry-item-card · .header → yry-scene-header · .nav → yry-cross-nav
  3. 类名映射 — 内联 .card h3 → property name · .card p → desc · .tag → tags
  4. 视觉对比 — 截图 diff 0 差异
  5. 删除冗余 — 移除 200 行内联 <style> · 保留 20 行页面专属
  6. 回归测试 — Lighthouse 性能不劣化 · axe-core 0 新增违规

✓ 结果：850 行 → 620 行 · 减少 27% · 内联消除 90%

💡 演示 3 · 灰度发布流程

将迁移后的 审查.html 推到生产：

# 1. 推到 staging (10% 内部用户)
git commit -m "feat(审查): 迁移到 CDN 组件"
git push origin staging

# 2. 观察 24h 无回归
./check-metrics.sh staging/审查.html
# - Lighthouse 性能 ≥ 迁移前
# - axe-core 0 新增违规
# - 控制台 0 error

# 3. 推到 canary (10% 流量)
./deploy.sh canary 审查.html

# 4. 48h 监控
./monitor.sh canary/审查.html
# - 错误率 ≤ 0.1%
# - 用户反馈无负向

# 5. ramp 50% → 100%
./deploy.sh ramp 50
sleep 86400  # 24h 监控
./deploy.sh full 100

💡 演示 4 · 视觉回归对比

用 Percy 做视觉回归：

# 截图迁移前
git checkout before-migration
npx percy snapshot 审查.html --base

# 截图迁移后
git checkout after-migration
npx percy snapshot 审查.html --review

# Percy 自动 diff 并生成报告
# 0 视觉差异 ✓ 通过

若 diff 不为 0，检查：组件 property 赋值是否完整 · 设计令牌是否生效 · 页面专属 CSS 是否丢失。
