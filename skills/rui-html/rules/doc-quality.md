## 文档等级

### 四级证据模型

| 等级 | 定义 | 标识 | 判定标准 | 处理 | 示例 |
|------|------|:---:|---------|------|------|
| **A** | 已验证 | ✅ | 可通过 Read/Grep/Glob/Bash 直接验证，附具体文件路径+行号 | 可直接引用，无需额外验证 | `src/auth.ts:42 — verifyToken() 函数签名` |
| **B** | 可推导 | ⚡ | 从 A 级证据经一步推导可得，推导链可追溯 | 可引用，标注推导链 | `从 package.json deps 推断使用 Express 4.x → 推断路由模式` |
| **C** | 未验证 | ⚠️ | 内容合理但缺乏可验证来源 | **必须显式标注**「待补充」 | `[待补充] 性能目标需产品确认` |
| **D** | 虚构 | 🚫 | 无任何来源的断言，或来源不可追溯 | **禁止出现**，发现即 P0 | `"系统性能提升 30%"`（无测量数据） |

### 证据等级升级路径

```
C (未验证) → 补充验证来源 → B (可推导) → 补充直接证据 → A (已验证)
D (虚构) → 删除或降级为 C → 补充验证 → B → A
```

### 证据标注格式

```markdown
<!-- 行内标注 -->
系统使用 JWT 进行认证。<!-- A: src/auth/jwt.ts:15-30 -->

<!-- 块级标注 -->
> **证据等级：B**
> 推导链：package.json → express@4.18.2 → 默认使用 connect 中间件栈 → 推断路由注册顺序
```

## 内容标准

### 必须包含

| 文档类型 | 必含元素 | 验证方式 | 缺失处置 |
|---------|---------|---------|---------|
| Agent .md | frontmatter + 工作循环 mermaid + 规则表 + Red Flags + 合理化速查表 | 逐节检查 | P0 — 补全 |
| Skill SKILL.md | 触发条件 + 输入/输出 + 使用示例 + 退出码 + 自循环 | `grep "## 触发" SKILL.md` | P0 — 补全 |
| Rule .md | 适用范围 + 门禁条件 + 反例 + 生效标志 | `grep "反例\|Anti-pattern" rule.md` | P0 — 补全 |
| 故事任务.md | FP# + AC + SC + 风险 + 知识图谱基线 | 每 FP# 对应 ≥1 AC | P0 — 补 AC |
| 场景 index.md | §0 mermaid 效果示意 + §1 测试设计 + §2 实施报告 + §3 测试报告 + §4 自改进 | 逐 § 检查 | P0 — 补章节 |

### 禁止出现

| 禁止项 | 为什么 | 替代做法 | 检测方式 |
|--------|--------|---------|---------|
| TBD / TODO | 不完整 = 不可执行 | 写完整或标注 Level C | `grep -r "TBD\|TODO"` |
| "类似 XXX" | 引用不可追溯 | 写独立完整描述 | 人工审查 |
| 无来源的性能声明 | Level D | 附测量数据或标注 Level C | `grep "性能提升\|速度加快"` |
| "应该" / "可能" | 不可证伪 | 用"必须"（P0）/ "建议"（P1）/ "可选"（P2） | `grep "应该\|可能"` |
| 过期截图 | 误导 | 标注截图日期或使用 mermaid（可文本 diff） | 检查图片文件 mtime |
| "显然" / "当然" | 跳过推理 | 写完整推理链 | 人工审查 |
| 无证据的"最佳实践" | 上下文无关 | 附具体场景和证据 | 人工审查 |

### 语言规范

| 规范 | 说明 | 正例 | 反例 |
|------|------|------|------|
| 确定性措辞 | 用"必须/建议/可选"替代"应该/可能" | "P0 项必须修复" | "P0 项应该修复" |
| 主动语态 | 主语明确，动作清晰 | "coder 实现模块" | "模块被实现" |
| 术语一致 | 使用领域语言定义，不混用别名 | "故事"（不用 task/ticket） | 混用"故事"和"任务" |
| 量化 | 用数字替代模糊描述 | "响应时间 < 2s" | "响应速度快" |

## 结构要求

### Agent 文件统一模版

```
---
name: <name>
description: <一句话>
tools: <Read/Grep/Glob/Bash/Edit/Write>
---

# <name> — <中文职责>

> <一句话角色定位>

[工作循环](#) · [触发](#) · [规则表](#) · [Red Flags](#) · [职责边界](#) · [生效标志](#)

## 工作循环       ← mermaid 流程图
## 触发           ← 谁/什么触发该 Agent
## 规则表         ← # | 规则 | 反例
## Red Flags      ← 暂停信号 + 合理化速查表
## 职责边界       ← 归我 / 不归我
## 生效标志       ← 达标标准 | 未达标处置
```

### Skill 文件统一模版

```
---
name: <skill-name>
description: <一句话>
user_invocable: true/false
lifecycle: default-pipeline | standalone
agents: (可选)
  required: [agent1, agent2]
  optional: [agent3]
---

# <skill-name>

> <一句话定位>

[触发](#) · [输入/输出](#) · [模式](#) · [约束](#) · [交付](#) · [自循环](#)

## 触发
## 输入/输出
## 模式（如有多种）
## 约束
## 交付（如是写入命令）
## 自循环
```

### Rule 文件统一模版

```
---
paths: (可选)
  - "glob pattern"
---

# <rule-name> — <中文主题>

> <一句话原则>

[适用范围](#) · [门禁](#) · [反例](#) · [生效标志](#)

## 适用范围
## 门禁条件
## 反例 / 常见误判
## 生效标志
```

## 质量评分

### DQI 文档质量指数

| 维度 | 权重 | 评估方法 | 满分条件 |
|------|:---:|---------|---------|
| **完整性** | 40% | 必含元素全部存在 | 模板字段全部非空 |
| **证据等级** | 30% | A/B/C/D 比例分布 | A+B ≥ 80%，D = 0 |
| **反模式缺失** | 20% | 无 TBD/TODO/模糊措辞 | grep 零命中 |
| **规约一致性** | 10% | frontmatter 字段完备 + 交接信号可验证 | 全部字段存在且有效 |

### 评分等级

| 等级 | 分数 | 含义 | 处置 |
|------|:---:|------|------|
| **A** | ≥ 90 | 文档质量优秀 | 无需处理 |
| **B** | ≥ 75 | 文档质量良好 | 关注 C 级证据比例 |
| **C** | ≥ 60 | 文档需要改进 | 制定改进计划 |
| **D** | < 60 | 文档质量不合格 | P0 — 立即修复 |

## 审查清单

### 全量审查 (12 项)

| # | 检查项 | 通过条件 | 工具 | 严重度 |
|---|--------|---------|------|:---:|
| 1 | 无 TBD/TODO | `grep -r "TBD\|TODO" docs/ skills/` 返回空 | grep | P0 |
| 2 | 无 D 级断言 | 所有陈述有来源标注或显式标注 Level C | 人工 | P0 |
| 3 | 版本行正确 | 文档版本与 plugin.json 一致 | grep | P0 |
| 4 | 交叉引用有效 | 所有内部链接可达 | grep 验证锚点 | P1 |
| 5 | Mermaid 图可渲染 | 图在本地预览中正常显示 | 人工 | P1 |
| 6 | Agent 模版完整 | 含 frontmatter + 工作循环 + 规则表 + Red Flags | 逐节检查 | P0 |
| 7 | Skill 模版完整 | 含 frontmatter + 触发 + 输入/输出 + 约束 + 自循环 | 逐节检查 | P0 |
| 8 | Rule 模版完整 | 含适用范围 + 门禁 + 反例 + 生效标志 | 逐节检查 | P0 |
| 9 | 无模糊措辞 | 无"应该/可能/显然/当然" | grep | P1 |
| 10 | 术语一致 | 使用领域语言定义，不混用别名 | 人工 | P1 |
| 11 | 证据标注格式 | A/B/C 标注格式正确 | grep | P1 |
| 12 | 语言规范 | 确定性措辞 + 主动语态 + 量化 | 人工 | P2 |

### 增量审查（文档变更时）

| # | 检查项 | 适用场景 |
|---|--------|---------|
| 1 | 变更章节无新增 TBD/TODO | 所有变更 |
| 2 | 新增内容有证据标注 | 所有变更 |
| 3 | 交叉引用未断裂 | 修改链接/锚点时 |
| 4 | 版本行已更新 | 所有变更 |

## 退化检测

### 退化信号

文档退化的三个核心信号，出现任一即触发审查：

| 信号 | 检测方式 | 严重度 | 处置 |
|------|---------|:---:|------|
| **版本漂移** | `grep` 版本号 vs plugin.json | P0 | 更新文档版本行 |
| **链接断裂** | 404 / 锚点不存在 | P1 | 修复或移除链接 |
| **内容过时** | 文档描述与 `git diff main` 不一致 | P0 | 运行 `rui-update` 刷新 |

### 退化检测矩阵

| 检测维度 | 频率 | 方法 | 阈值 |
|---------|------|------|------|
| 版本号一致性 | 每次健康检查 | `grep` 版本号 vs plugin.json | 不一致 = 告警 |
| 链接可达性 | 每周 | 逐链接检查 | 任一连结断裂 = 告警 |
| 内容新鲜度 | 每次 commit | 文档 mtime vs 源码 commit 时间 | 落后 > 7 天 = 告警 |
| 证据等级分布 | 每月 | A/B/C/D 比例统计 | D > 0 或 C > 30% = 告警 |
| 模版完整性 | 每次健康检查 | 逐文档检查必含元素 | 任一缺失 = 告警 |
| 术语一致性 | 每月 | 领域语言 Avoid 列检测 | 发现禁用别名 = 告警 |

### 退化恢复流程

```
1. 检测到退化信号 → 触发审查
2. 审查确认退化类型和范围
3. 自动修复（版本号/链接/格式）
4. 无法自动修复 → 生成改进提案 → 进入自改进管线
5. 修复后重新评分
```

## 自动化检查

### 检查脚本

```bash
# 全量文档质量检查
node lib/arch-check.mjs --doc-quality

# 单文件检查
node lib/arch-check.mjs --doc-quality --file skills/rui-code/SKILL.md

# 退化检测
node lib/arch-check.mjs --doc-quality --detect-degradation
```
