# 编排与阶段契约规范

> 本规范约束 `generate-document` 的阶段状态机、skill/agent 分派、通知要求与降级策略。
> **权威原则**：阶段编号、阶段名称、进入条件、退出条件以本文件和 `../SKILL.md` 为准。

---

## 1. 阶段状态机（权威定义）

| 阶段   | 名称                      | 目标                                                                                                                     | 解锁条件                                                                                                         |
| ------ | ------------------------- | ------------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------- |
| 阶段 1 | 解析 + 规范检索           | 解析功能名或 `init` 命令，调用 `spec-retriever` 获取适用规范                                                             | 功能名/init 可定位，规范列表已返回                                                                               |
| 阶段 2 | 上游 Grounding + 影响分析 | 读取上游文档，调用 `impact-analyst` 完成影响链闭合（init 命令：扫描项目代码和配置）                                      | 事实-来源映射表完整，影响链分析结果已写入文档                                                                    |
| 阶段 3 | 专家生成                  | 调用 `architect`、`planner` 等 agent 获取架构与实施方案（init 命令：推断项目架构模式）                                   | 模块划分、接口规范、实施策略已确认 / init 命令：项目架构模式已推断                                               |
| 阶段 4 | 逐文档生成 + 自检         | 按规范生成各文档；**凡产出 Mermaid 图源代码块，定稿前必须调用 `mermaid-expert` 审查并修复**；再调用 `code-reviewer` 审查设计文档，自检通过（init 命令：生成 8 个基础文件 + `docs/项目初始化/` 01-07） | 01-05, 07 文档已生成，含图源已通过 `mermaid-expert`，设计文档已通过 `code-reviewer`，P0 自检通过或已记录失败项 / init 命令：所有文件已生成，P0 自检完成 |
| 阶段 5 | 保存 + 知识策展           | 保存文档，调用 `knowledge-curator` 策展知识                                                                              | 文档已保存到 `docs/<功能名>/` 或项目根目录 / init 命令：所有文件已保存                                           |
| 阶段 6 | 文档同步与通知            | 同步 `docs` 并发送完成 / 含 P0 失败通知（**所有命令强制适用，包括 init**）                                               | `import-docs` 已执行并记录真实结果，`wework-bot` 已按本次结束类型（成功 / 含 P0 失败）模板发送或记录发送失败原因 |

## 2. 输入前提

### 2.1 默认输入

- `{功能名}`：对应 `docs/<功能名>/` 目录
- `{文档集路径}`：默认 `docs/<功能名>/`

若无法解析 `{功能名}`，必须先写入 `docs/99_agent-runs/<YYYYMMDD-HHMMSS>_generate-document.md`，记录原始请求、解析失败原因和建议恢复步骤，再向用户请求补充信息。

### 2.2 `from-weekly` 输入（从周报拆解多套全文档）

当用户调用 **`generate-document from-weekly <周报路径>`** 时：

- **前置**：周报文件必须存在；优先解析「四、后期规划与改进优先级总表」；详见 `../SKILL.md`「命令扩展：从周报拆解全文档（from-weekly）」。
- **映射表**：必须在 `docs/99_agent-runs/<YYYYMMDD-HHMMSS>_from-weekly.md` 落盘周报路径、`{功能名}` 列表与顺序。
- **阶段循环**：对每个 `{功能名}` 完整执行阶段 1～5；上一功能目录保存完毕后再进入下一功能名。
- **阶段 6**：全部功能处理完毕后 **仅执行一次** `import-docs`（同步整个 `docs/`）与 **一条** `wework-bot` 汇总通知（列出各功能目录与周报溯源）。

### 2.3 缺失处理

无法定位仓库或项目结构时：

1. 记录缺失内容。
2. 写入兜底运行记录。
3. 按阻断通知模板调用 `wework-bot`。
4. 提示用户补充信息。

### 2.4 一次执行到底（默认策略）

`generate-document` 默认应“一次执行到底”，减少人工干预：

- 除非命中 `../SKILL.md` 的「人工介入门槛」，否则不得频繁向用户追问确认；缺失信息用“待确认/待补充”标注并继续推进。
- 一旦命中人工介入门槛，必须先写入兜底运行记录，再**必须调用 `wework-bot`** 推送“需要人工介入/阻断”消息，提供原因、证据与恢复点。

## 3. 阶段进入 / 退出条件

### 阶段 1

- 进入：用户调用 `generate-document`
- 退出：功能名已解析，`spec-retriever` 已返回适用规范列表

### 阶段 2

- 进入：阶段 1 完成
- 退出：上游文档已读取，`impact-analyst` 已完成影响链闭合分析，结果已准备写入

### 阶段 3

- 进入：阶段 2 完成
- 退出：`architect`、`planner` 等 agent 已回答必答问题，结果已采纳

### 阶段 4

- 进入：阶段 3 完成
- 退出：01-05, 07 文档已生成；**凡含 Mermaid 图源代码块的文档均已通过 `mermaid-expert` 审查并写回修复结果**；设计文档已通过 `code-reviewer` 审查；自检已完成

### 阶段 5

- 进入：阶段 4 完成
- 退出：文档已保存，`knowledge-curator` 已完成知识策展

### 阶段 6

- 进入：阶段 5 完成，或任意前置阶段阻断
- 执行：
  1. 先运行 `import-docs` 标准导入并记录真实结果（创建 N、覆盖 N、失败 N）
     - init 命令：同步整个 `docs/` 目录（包括新生成的 8 个基础文件和 `docs/项目初始化/`）
     - 功能文档：同步 `docs/<功能名>/`
  2. 再调用 `wework-bot` 发送通知，通知模板必须与本次结束类型一致：
     - 成功：使用 `wework-bot/SKILL.md` 的「generate-document 完成（成功）」模板
       - init 命令：`📋 类型` 填「项目初始化」，`📎 证据` 列出主要生成的文件
     - 含 P0 失败：使用「generate-document 完成（含 P0 失败）」模板
     - 阻断：使用「implement-code 阻断」模板（适配 generate-document 场景）
- 退出（全部满足才放行）：
  1. `import-docs` 已执行或记录跳过原因（必须说明为什么跳过；若缺少系统环境变量 `API_X_TOKEN`，仅在该时提示）
  2. `wework-bot` 已按模板发送，或发送失败时已把 API 状态 / 错误摘要、脱敏机器人路由、模型、工具、最后更新时间（精确到秒）写回文档或 `docs/99_agent-runs/`
  3. 通知必须包含 `⏱️ 用时`、`🪙 会话用量`、`🤖 模型`、`🧰 工具`、`🕒 最后更新时间`（精确到秒）
  4. **init 命令强制**：若已执行 import-docs，`☁️ 文档同步` 必须使用真实结果，不得虚构；若未执行，仅记录原因，不要固定重复兜底文案

## 4. 阻断点

以下情况必须停止当前推进：

- `{功能名}` 无法解析
- 规范检索失败且无法降级
- 影响链分析发现未闭环的阻断性依赖
- agent 调用失败且无备用方案
- 文档生成 P0 失败且无法自修复

停止时必须：

1. 输出阻断原因。
2. 记录当前阶段和已产生产物。
3. 功能目录可定位时，保存已生成的文档；init 命令：保存已生成的部分文件。
4. 功能目录不可定位时，写入 `docs/99_agent-runs/<YYYYMMDD-HHMMSS>_generate-document.md`。
5. 执行 `import-docs` 标准导入；目录不存在、导入失败或部分失败都必须记录真实数字和错误摘要。
6. **必须调用 `wework-bot`** 发送阻断通知，正文必须包含阻断阶段、证据路径、P0 统计、模型、工具、最后更新时间（精确到秒）。
   - init 命令阻断时：`🔍 阻断阶段` 明确标注是 init 命令的哪一步，`🧭 恢复点` 说明从哪里重新开始
7. 通知发送失败时，把失败原因写回文档或 `docs/99_agent-runs/`，再结束流程。

## 5. wework-bot 通知强制要求

### 5.1 任何结束都必须通知

> **P0 强制约束**：`generate-document` 每次执行结束，**无论成功或失败**，**无论功能文档还是 init 命令**，都必须调用 `wework-bot` 发送通知，禁止省略。

依据：`../wework-bot/SKILL.md §157-160`

### 5.2 通知模板选择

| 结束类型             | 模板                                                     | 说明                                           |
| -------------------- | -------------------------------------------------------- | ---------------------------------------------- |
| 成功完成，无 P0 失败 | `wework-bot/SKILL.md §168-193`                           | 若已执行 import-docs，填写 `☁️ 文档同步` 真实数字 |
| 完成但含 P0 失败     | `wework-bot/SKILL.md §195-222`                           | 必须包含 `❌ 原因`、`🧭 恢复点`                |
| 中途阻断             | `wework-bot/SKILL.md §264-298`（适配 generate-document） | 必须包含 `🔍 阻断阶段`、`❌ 原因`、`🧭 恢复点` |

**init 命令特殊要求**：

- init 命令成功时 `📋 类型` 填「项目初始化」，`📎 证据` 列出主要生成的文件（`CLAUDE.md`、`README.md`、`docs/architecture.md`、`docs/项目初始化/` 等）

### 5.3 通知必填字段

所有通知**必须**包含：

| 字段              | 说明             | 来源                                   |
| ----------------- | ---------------- | -------------------------------------- |
| `🎯 结论`         | 一句话结论       | 本次执行结果                           |
| `📝 描述`         | 100 字以内描述   | 本次执行背景                           |
| `🤖 模型`         | 模型名称         | 自动检测或传入                         |
| `🧰 工具`         | 主要工具列表     | 本次使用的 skills/agents               |
| `🕒 最后更新时间` | 精确到秒         | 当前时间                               |
| `⏱️ 用时`         | 本次会话耗时     | 自动检测或传入；无则脚本注入可核对说明 |
| `🪙 会话用量`     | Token 或用量摘要 | 须可核对；无则脚本注入缺省说明         |
| `👉 下一步`       | 接收者下一步动作 | 明确可执行的步骤                       |

成功 / 含 P0 失败通知**还必须**包含：

| 字段          | 说明                                             |
| ------------- | ------------------------------------------------ |
| `☁️ 文档同步` | 仅在已执行 `import-docs` 时填写真实结果（创建 N、覆盖 N、失败 N） |
| `🌿 分支`     | git 分支（自动检测）                             |
| `🔖 提交`     | git 短提交号（自动检测）                         |

阻断通知**还必须**包含：

| 字段          | 说明                       |
| ------------- | -------------------------- |
| `🔍 阻断阶段` | 停止在哪个阶段             |
| `❌ 原因`     | 阻断原因摘要（≤2 条）      |
| `🧭 恢复点`   | 从哪里恢复，先做什么       |
| `📎 证据`     | 证据路径 / 命令 / 错误摘要 |

### 5.4 通知发送顺序

**严格遵守**：

1. 先执行 `import-docs`，获取真实的「创建 N、覆盖 N、失败 N」数字
   - init 命令：同步整个 `docs/` 目录
2. 再调用 `wework-bot`；若已执行 `import-docs`，把真实数字填入 `☁️ 文档同步` 字段

禁止在 `import-docs` 执行前发送通知，否则 `☁️ 文档同步` 字段无法填写真实值。

### 5.5 通知发送失败处理

若 `wework-bot` 调用失败：

1. 记录失败原因（API 状态码、错误摘要）
2. 记录脱敏机器人路由信息（使用哪个机器人 / webhook）
3. 记录通知发送尝试的模型、工具、最后更新时间（精确到秒）
4. 把以上信息写入 `docs/<功能名>/07_项目报告.md`（功能文档）或 `docs/项目初始化/07_项目报告.md`（init 命令）或 `docs/99_agent-runs/<YYYYMMDD-HHMMSS>_generate-document.md`
5. 结束流程（不重复尝试，避免重复通知）

## 6. Skill / Agent 分派

默认分派：

| 类型  | 名称                | 用途                     |
| ----- | ------------------- | ------------------------ |
| Skill | `find-skills`       | 声明和发现要用到的 skill |
| Skill | `find-agents`       | 发现和分派代理           |
| Skill | `import-docs`       | 阶段 6 文档同步          |
| Skill | `wework-bot`        | 阶段 6 完成 / 阻断通知   |
| Agent | `spec-retriever`    | 阶段 1 规范检索          |
| Agent | `impact-analyst`    | 阶段 2 影响链分析        |
| Agent | `architect`         | 阶段 3 架构方案          |
| Agent | `planner`           | 阶段 3 实施策略          |
| Agent | `docs-lookup`       | 文档歧义定位（按需）     |
| Agent | `mermaid-expert`    | 阶段 4 含 Mermaid 文档语法审查与修复（先于 `code-reviewer`） |
| Agent | `code-reviewer`     | 阶段 4 设计文档审查      |
| Agent | `quality-tracker`   | 阶段 4 质量统计          |
| Agent | `knowledge-curator` | 阶段 5 知识策展          |

## 7. 编排会话日志（强制）

与本编排相关的 **skill / agent / MCP / memory / shared** 交互，在对应步骤完成后必须追加写入 **`docs/周报/<YYYY-MM-DD>~<YYYY-MM-DD>/logs.md`**（Markdown）。

1. **触发时机**：每次完成对 `.claude` 内某一 skill、agent、MCP 的一轮调用（含读完记忆文件后的下游采纳），立即追加一条记录。
2. **工具**：`node .claude/scripts/log-orchestration.js`（参数见 `../SKILL.md`「编排会话日志」）。
3. **记录结构**：每条含 **操作场景** 与 **对话与交互摘要**；可选 **`--case good|bad`**、`--tags`、`--lesson` 标注 good/bad case（见 `docs/logs/CASE-STANDARD.md`）。禁止空占位。
4. **阻断与兜底**：即使中途阻断，对已发生的交互仍须补齐日志后再结束。

## 8. 关键节点记录（推荐）

阶段切换、门禁结论、对外通知结果等 **一眼可扫描** 的里程碑，推荐追加写入 **`docs/周报/<YYYY-MM-DD>~<YYYY-MM-DD>/key-notes.md`**（Markdown）：`node .claude/scripts/log-key-node.js`。与「编排会话日志」互补，**不可替代**上文第 7 节的强制要求。
