## 文档结构

> 格式原则：面向管理者，"大表格 + Mermaid 图"，约 3 大表 + 2~3 Mermaid。

### 1. 头部

版本信息 + 覆盖周期(自然周周一至周日) + 关联功能目录

**关联功能目录写法**：
- 有活跃功能目录时：列出 `docs/<功能名>/` 目录名（不强制加链接，禁止链接到不存在的路径）
- 无活跃功能目录时：写 `> 本周无活跃功能目录`，禁止虚构目录或生成死链

### 2. KPI 量化总表

按功能分行：交付完成率 / P0通过率 / 防幻觉率 / 修复轮次 / 规则覆盖率 / 维度综合(✅/🟡/❌+依据)

判定：交付≥80%✅ | P0≥90%✅ | 轮次≤2✅

### 3. 本周复盘

进展亮点(1~3条可验证事实) → 问题根因(现象→推断→证据路径) → 与上周对比(可选)

### 4. KPI→复盘→规划链路全景图（Mermaid flowchart）

KPI未达标 → 复盘根因 → 后期规划（因果链）

**语法约束**：
- 中文节点、含空格或特殊字符的节点必须用双引号包裹，如 `A["KPI未达标"]`
- 禁止在节点文本中使用全角括号 `（）`、全角冒号 `：` 等未转义字符

### 5. 后期规划与改进

**原则**：以需求为导向，参考业内标准但不套用复杂方法论。每项改进必须回答三个问题：解决什么问题？如何验证有效？不做会怎样？

#### 5.1 改进优先级总表

合并后期规划+系统自改进+项目自改进。每条记录包含：

| 字段 | 说明 |
|------|------|
| 类型 | 规划 / 系统 / 项目 |
| 问题来源 | 对应本周哪个薄弱KPI或根因 |
| 改进描述 | 具体、可执行的一句话 |
| 参考标准 | 业内同类问题的通行做法（如代码评审、自动化测试、配置即代码等），不写书名和理论 |
| 验证方式 | 可量化的验收标准 |
| 时间维度 | 下周 / 本月 / 季度 |
| 专业深度 | 技术架构 / 流程效率 / 质量保障 / 团队协作 |

**排序规则**：
1. 首条必须对应本周最薄弱的KPI
2. 同一优先级按"时间维度近者优先"
3. 禁止用象限图、矩阵图等可视化方式表达优先级

#### 5.2 工作流程标准化审视

每周强制回答以下问题（无则不写，禁止虚构）：

1. **重复劳动识别**：本周是否有超过2次的手动重复操作？能否脚本化或模板化？
2. **决策标准缺失**：是否有依赖个人经验的模糊决策点？能否沉淀为checklist或规则？
3. **信息孤岛**：是否有多个环节依赖口头传递或临时文档？能否统一信息源？
4. **反馈闭环**：本周发现的问题是否有明确的跟进Owner和验收节点？

#### 5.3 系统架构演进思考

仅当本周有架构相关交付时填写：

- 当前架构瓶颈是什么？（性能 / 可维护性 / 扩展性 / 安全）
- 下一个自然的演进节点在哪里？
- 演进的风险和回退方案是什么？

## 一次执行到底（强制）

周报生成必须单轮完成，不得因任何非阻断原因中断或降级为"稍后补充"。

- **缺失信息处理**：数据缺失写"待确认"、"本周无数据"、"未记录"，不得停下来等待用户补充
- **阻断门槛**：仅当命中 H1-H4 阻断条件（见 `orchestration.md §4`）时才允许中断；中断后仍必须执行步骤 6（import-docs → wework-bot）
- **禁止行为**：禁止说"请提供XX数据后再继续"、禁止将周报拆分为多轮生成、禁止因某个功能目录缺失而跳过整份周报
