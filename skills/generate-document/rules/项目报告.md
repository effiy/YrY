# 项目报告规范

> **生成约束**：本文档类型**禁用模板**。即使 `templates/` 下存在同名文件，也不作为生成输入。
> 所有内容必须来自真实来源（用户目标、上游文档、git diff / 文件对比、实际源码、验证日志），无真实来源则写"待补充（原因：...）"，**不得编造**。

## 文档概述

- 项目报告是功能交付后的交付存档与复盘文档，需要让读者快速理解"为什么改、改了什么、影响哪里、如何验证、还有什么风险"
- **上游文档**：需求文档、需求任务、设计文档、使用文档、动态检查清单、实施总结（若存在）
- **Grounding 来源**：用户目标、上游文档、`git diff`、`git status`、实际源文件读取、验证日志、审查结论
- **写作原则**：先结论后细节；概述面向项目干系人，变更对比面向开发维护者；所有结论必须能追溯来源

## 生成前置条件（P0 硬门槛）

以下任一未满足，**必须中止生成** 并向用户要求补齐：

1. 能够访问本次变更的文件列表（`git diff --name-status` / `git status --short` 或用户提供）。
2. 每个变更文件的"变更前"内容可读取（`git show HEAD:<path>` / `git diff` 上下文 / 原始内容已提供）；新增文件允许无变更前内容。
3. 至少存在一个可关联的上游文档（优先 `docs/<功能名>/03_设计文档.md`，其次为同目录需求任务、动态检查清单或实施总结）。
4. 能够区分本次报告范围内变更与工作区内无关变更；无法区分时必须在"报告范围"中显式声明不确定性。

## 文档结构（严格遵循）

```markdown

>
> **关联文档**: [需求文档](../01_需求文档/{文件名}.md) | [需求任务](../02_需求任务/{文件名}.md) | [设计文档](../03_设计文档/{文件名}.md) | [使用文档](../04_使用文档/{文件名}.md)
>

[交付摘要](#交付摘要) | [报告范围](#报告范围) | [变更概览](#变更概览) | [影响评估](#影响评估) | [验证结果](#验证结果) | [风险与遗留项](#风险与遗留项) | [变更文件列表](#变更文件列表) | [变更前后对比](#变更前后对比) | [变更汇总表](#变更汇总表)

```

### 2. 交付摘要

面向非实现读者，用 3-5 条要点说明本次交付结果：

- **交付目标**：来自用户请求、需求文档或实施总结，禁止凭空扩展目标
- **核心结果**：概括已完成的主要能力或文档/代码成果
- **变更规模**：新增 N 文件 / 修改 N 文件 / 删除 N 文件 / 重命名 N 文件
- **验证结论**：已执行验证、未执行验证及原因
- **当前状态**：已完成 / 部分完成 / 待补充，并说明依据

### 3. 报告范围

说明本报告纳入与排除的范围，避免把工作区无关变更误写入交付结论：

| 范围项 | 内容 | 来源 |
|--------|------|------|
| 纳入范围 | {本次报告覆盖的功能、文档或代码变更} | {用户请求 / 上游文档 / git diff} |
| 排除范围 | {明确不属于本次报告的变更；无则写"无"} | {git status / 用户说明} |
| 不确定项 | {无法判断归属的变更；无则写"无"} | {原因} |

### 4. 变更概览

从读者视角总结变化，避免只罗列文件：

| 变更域 | 变更前 | 变更后 | 价值 / 影响 | 来源 |
|--------|--------|--------|-------------|------|
| {功能 / 文档 / 配置 / 测试 / 工具链} | {旧状态} | {新状态} | {为什么重要} | {文件路径或上游文档} |

**约束**：
- "变更前 / 变更后"必须来自 diff、旧文件内容或上游文档，不得根据期望效果推断
- 若只能确认文件层面的变化，写文件层面的事实，不扩展成用户行为结论
- 每条概览必须至少关联一个来源

### 5. 影响评估

按影响面说明本次变更可能影响的模块、用户、数据和运维：

| 影响面 | 等级 | 影响说明 | 依据 | 处置建议 |
|--------|------|----------|------|----------|
| 用户体验 | 无 / 低 / 中 / 高 | {说明} | {来源} | {无需处理 / 建议验证 / 待补充} |
| 功能行为 | 无 / 低 / 中 / 高 | {说明} | {来源} | {无需处理 / 建议验证 / 待补充} |
| 数据 / 接口 | 无 / 低 / 中 / 高 | {说明} | {来源} | {无需处理 / 建议验证 / 待补充} |
| 构建 / 部署 | 无 / 低 / 中 / 高 | {说明} | {来源} | {无需处理 / 建议验证 / 待补充} |
| 文档 / 协作 | 无 / 低 / 中 / 高 | {说明} | {来源} | {无需处理 / 建议验证 / 待补充} |

**等级标准**：
- **高**：影响公共接口、Store、路由、持久化数据、构建配置、部署流程或跨模块契约
- **中**：影响单个功能模块行为、主要用户流程或重要文档契约
- **低**：局部 UI、样式、文案、注释、单一文档补充
- **无**：确认不涉及该影响面，仍需写明依据

### 6. 验证结果

记录真实执行过的验证，不得把"建议验证"写成"已通过"：

| 验证项 | 命令 / 方法 | 结果 | 证据 | 备注 |
|--------|-------------|------|------|------|
| {构建 / lint / 单测 / E2E / 人工检查 / 文档检查} | `{命令}` 或 {方法} | 通过 / 失败 / 未执行 | {日志摘要 / 文件路径 / 未执行原因} | {后续动作} |

**约束**：
- 未执行的验证必须写"未执行"并说明原因
- 失败的验证必须保留失败摘要，不得删除或弱化
- 若动态检查清单存在，应引用其中对应检查项

### 7. 风险与遗留项

列出仍需关注的问题，不能为了显得完成而省略风险：

| 类型 | 描述 | 严重度 | 负责人 / 后续动作 | 来源 |
|------|------|--------|------------------|------|
| 风险 / 遗留 / 待确认 | {具体问题} | 高 / 中 / 低 | {建议动作} | {来源} |

无风险时写：`未发现明确遗留风险（依据：已读取的 diff / 验证日志 / 上游文档范围）。`

### 8. 变更文件列表（必须，强制完整）

列出**所有**变更文件，不得遗漏、不得新增：

| 序号 | 文件路径 | 变更类型 | 变更域 | 说明 |
|------|---------|---------|--------|------|
| 1 | `src/...` | 新增 / 修改 / 删除 / 重命名 | 功能 / 文档 / 配置 / 测试 / 工具链 | {一句话说明改动目的} |

**变更类型取值**：新增 / 修改 / 删除 / 重命名

**来源约束**：文件路径必须取自 git diff、git status 或用户确认的清单；禁止根据设计文档"猜测"应该变更哪些文件。

### 9. 变更前后对比（必须，每个变更文件一节）

每节结构：

```markdown
#### {文件路径}

**变更类型**：{新增 / 修改 / 删除}

**变更前**：

```{语言}
{变更前的完整相关代码或"（新增文件，无变更前内容）"}
```

**变更后**：

```{语言}
{变更后的完整相关代码或"（已删除，无变更后内容）"}
```

**说明**：{一句话说明本次变更的原因与影响}
```

**约束**：
- "变更前"内容必须来自真实的旧版本，不得根据"变更后"反推
- 若变更文件行数超过 100 行，展示关键片段（函数/类级别），并注明"已截取关键段落，完整内容见文件"
- 新增文件："变更前"写 `（新增文件，无变更前内容）`
- 已删除文件："变更后"写 `（已删除，无变更后内容）`
- 文档类文件可展示关键章节差异；代码类文件优先展示改动函数、类、配置块或导出入口
- 只展示片段时必须说明截取依据，避免读者误以为是完整文件

### 10. 变更汇总表（必须）

| 文件路径 | 变更类型 | 变更域 | 影响评估 | 关键变更说明 | 验证覆盖 |
|---------|---------|--------|---------|------------|----------|
| `src/...` | 修改 | 功能 / 文档 / 配置 / 测试 / 工具链 | 无 / 低 / 中 / 高 | {关键变更说明} | 已验证 / 未验证 / 不适用 |

**影响评估标准**：
- **高**：影响公共接口 / Store / 路由 / 构建配置 / 部署流程 / 持久化数据

- **低**：样式调整 / 注释 / 单一文档
- **无**：纯格式、索引或无行为影响的元信息变更，并能说明依据

### 11. skills / agents / rules 自我改进总结（证据驱动，必须，放在文档最后）

> **目标**：让本次项目报告不仅“记录交付”，还要能“反向改进文档生产体系”，降低下次生成/复用的成本。
>
> **证据约束（强制）**：

## 质量检查清单

### P0 - 必须通过

- **文档头部完整**：含版本、日期、维护者、关联文档
- **报告范围明确**：纳入 / 排除 / 不确定项清楚，不能混入无关变更
- **交付摘要可追溯**：交付目标、状态、验证结论均有来源
- **变更文件列表完整**：与 git diff 或用户提供的清单完全一致
- **每文件有前后对比**：变更文件列表中的每个文件都有对应的"变更前后对比"节
- **变更前内容真实**：来自旧版本，非推断
- **代码路径真实**：文件路径在仓库中真实存在
- **变更汇总表完整**：所有变更文件都出现在汇总表中
- **验证结果真实**：已执行 / 未执行 / 失败的验证状态如实记录
- **自我改进章节存在且证据化**：文档最后包含 `## skills / agents / rules 自我改进（证据驱动）`，且每条结论都能回指到本报告或仓库路径/规则/checklist 的证据

### P1 - 应该通过

- **变更概览清晰**：能从功能、文档、配置、测试等变更域理解整体变化
- **影响评估完整**：覆盖用户体验、功能行为、数据 / 接口、构建 / 部署、文档 / 协作
- **风险与遗留项明确**：无风险时有依据，有风险时有后续动作
- **变更说明清晰**：每条变更有明确的"原因与影响"说明
- **影响评估合理**：无/低/中/高评估与实际改动范围匹配
- **关键代码截取得当**：超长文件只展示关键段落，并注明截取

### P2 - 可以有

- **变更概述数据准确**：新增/修改/删除文件数与列表一致
- **导航链接存在**：头部有章节跳转链接
- **读者分层友好**：摘要适合快速阅读，对比适合开发追溯
