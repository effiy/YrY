# 需求任务文档规范

> 上游：需求文档(01)；下游：设计文档。
> 核心职责：需求细化为场景与拆解，承载影响分析结论。

## 文档结构

### 1. 头部

标准头部 + 导航锚点（功能概述 | 功能分析 | 功能详情 | 验收标准 | 使用场景示例）。

### 2. 功能概述

100-200 字 + 3 核心价值（🎯⚡📖）

### 3. 功能分析（必须含 Mermaid）

4 种图表，每图下方须有说明文字：
- 功能分解图 `graph TB`
- 用户流程图 `flowchart TD`
- 功能流程图 `flowchart TD`
- 时序图 `sequenceDiagram`

节点色值：核心 `#ccffcc`，中性 `#e1f5ff`

### 4. 用户故事表格（必须从01提取）

完整提取，保持优先级图标（🔴🟡🟢）。**必须含主要操作场景描述**。

### 5. 主要操作场景定义（必须包含）

格式：场景名 → 场景描述 → 前置条件 → 操作步骤 → 预期结果 → 验证关注点 → 相关设计文档章节。

- P0 用户故事至少 2 个场景，P1 至少 1 个
- 场景须具体、可操作、可验证

### 6. 影响分析（必须包含）

> **强制执行**：按 `../../../shared/impact-analysis-contract.md` 全项目影响链闭合。

必须含四张子表：
1. 搜索词与改动点清单
2. 改动点影响链
3. 依赖闭合摘要
4. 未覆盖风险

末尾附改动范围汇总：直接修改 N / 验证兼容 N / 追踪传递 N / 需人工复核 N。

> 未命中搜索词标注"未找到引用"，不可省略该行。

### 7. 功能详情

按功能点分章节：功能说明、价值、痛点、收益。复杂功能可加 Mermaid。

### 8. 验收标准

P0(核心/不实现无法发布) / P1(重要/建议实现) / P2(锦上添花)

### 9. 使用场景示例

每个场景：背景、操作、结果，📋🎨 标识。
