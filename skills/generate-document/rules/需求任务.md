# 需求任务文档规范

## 文档概述

- 需求任务文档是对需求文档中用户故事的细化和功能分解，用于指导开发实施
- 必须从对应需求文档提取用户故事表格，保持需求与任务的一致性
- 上游文档：需求文档（位于 `docs/01_需求文档/`）
- 下游文档：设计文档

## 文档结构（严格遵循）

### 1. 文档头部（必须包含）

```markdown
# {功能名称}

> **文档版本**: v1.0 | **最后更新**: {日期} | **维护者**: {大模型名称} | **工具**: {Claude Code / Cursor}
>
> **关联文档**: [需求文档](../01_需求文档/{文件名}.md) | [设计文档](../03_设计文档/{文件名}.md) | [使用文档](../04_使用文档/{文件名}.md)
>

[功能概述](#功能概述) | [功能分析](#功能分析) | [功能详情](#功能详情) | [验收标准](#验收标准) | [使用场景示例](#使用场景示例)

---
```

### 2. 功能概述

- 简要说明功能的目标和范围，100-200字
- 列出 3 个核心价值点，使用 🎯⚡📖 图标

### 3. 功能分析（必须包含 Mermaid 图表）

必须包含 4 种 Mermaid 图表：

- **功能分解图**：`graph TB` - 展示功能模块和子功能
- **用户流程图**：`flowchart TD` - 展示用户操作流程
- **功能流程图**：`flowchart TD` - 展示系统功能处理流程
- **完整时序图**：`sequenceDiagram` - 展示组件间交互时序

每个图表下方必须添加明确的说明文字。

### 4. 用户故事表格（必须从需求文档提取）

- 完整提取自需求文档的用户故事表格
- 保持优先级图标（🔴🟡🟢）和格式不变
- 包含过程生成文档和产出智能文档列
- **必须包含主要操作场景描述**：在用户故事中明确列出主要操作场景，用于后续检查清单的动态生成

### 5. 主要操作场景定义（必须包含）

为每个用户故事明确列出主要操作场景，格式如下：

```markdown
#### 🎯 主要操作场景：{场景名称}

**场景描述**：{简要描述这个操作场景}

**前置条件**：{列出执行此操作前需要满足的条件}

**操作步骤**：
1. {步骤1}
2. {步骤2}
3. {步骤3}

**预期结果**：{描述操作完成后的预期结果}

**验证关注点**：{列出验证此场景需要检查的关键点}

**相关设计文档章节**：{指向设计文档中对应的实现章节}
```

主要操作场景的要求：
- 每个 P0 用户故事至少包含 2 个主要操作场景
- 每个 P1 用户故事至少包含 1 个主要操作场景
- 场景应覆盖核心用户路径和关键功能点
- 场景描述应具体、可操作、可验证

### 6. 影响分析（必须包含）

> **强制执行**：生成需求任务文档前，必须按 `../../../shared/document-contracts.md` 对整个项目执行完整影响分析。分析必须覆盖上游依赖、反向依赖、传递依赖、导出链、注册链、数据流、类型契约、样式、测试、文档、配置和外部依赖影响，避免改动点被其他引用或依赖时发生遗漏。

#### 执行步骤

0. **读取共享契约**：先读取 `../../../shared/document-contracts.md`，适用阶段、搜索范围、必查维度、输出格式、依赖闭合标准和 P0 门禁以该文件为准。
1. **确定核心标识符**：从需求文档中提取本次功能涉及的核心模块名、组件名、事件名、Store key、路由路径、CSS 类名/变量、公用工具函数名、配置项、依赖包名等关键标识符，形成“搜索词与改动点清单”。
2. **按契约全项目搜索**：对每个搜索词在整个仓库执行分类搜索，记录所有命中路径与行号；不得只搜索当前目录或 `src/`。搜索必须覆盖实现点、导出入口、注册入口、公共聚合入口、测试、文档、配置与外部依赖。
3. **追踪依赖链闭合**：对每个命中点继续检查其上游依赖、调用方、导出方、注册入口、消费方、测试和文档引用，直到影响链闭合或明确记录停止原因。
4. **排除无关结果**：排除 `node_modules/`、`dist/`、`*.lock`、文档自身等非业务文件；若排除的自动生成快照可能影响验收，必须写入未覆盖风险。
5. **标注处置方式**：同步修改、保持兼容、补充验证、人工复核、无需处理。

#### 搜索词与改动点清单格式

| 改动点 | 类型 | 搜索词 | 来源 | 备注 |
|--------|------|--------|------|------|
| `{模块1}` | component / store / route / event / css / config / dependency | `{标识符1}, {别名1}` | 需求文档 / 代码路径 | {为什么纳入分析} |

#### 改动点影响链格式

| 改动点 | 搜索词 | 命中文件 | 引用方式 | 影响层级 | 依赖方向 | 处置方式 | 闭合状态 | 说明 |
|--------|--------|----------|----------|----------|----------|----------|----------|------|

| 改动点 | 上游依赖是否核对 | 反向依赖是否核对 | 传递依赖是否闭合 | 测试 / 文档 / 配置是否覆盖 | 结论 |

| 风险来源 | 原因 | 影响 | 缓解方式 |

### 7. 功能详情

- 按功能点分章节
- 每个功能点包含：功能说明、价值、解决的痛点、收益
- 复杂功能点可添加 Mermaid 图表

### 8. 验收标准

- **P0 - 必须通过**：核心功能，不实现无法发布
- **P1 - 应该通过**：重要功能，建议实现
- **P2 - 可以有**：锦上添花，可选实现

### 9. 使用场景示例

- 每个场景包含：背景、操作、结果
- 使用 📋🎨 等图标标识不同场景

## Mermaid 图表规范

- 功能分解用 `graph TB`
- 用户流程用 `flowchart TD`
- 功能流程用 `flowchart TD`
- 时序图用 `sequenceDiagram`
- 节点色值：核心功能 #ccffcc，中性 #e1f5ff
- 每个图表下方添加明确的说明文字

## 保存位置

`docs/02_需求任务/{功能名称}.md`

## 质量检查清单

### P0 - 必须通过

- **用户故事表格**：完整提取自需求文档，包含优先级图标
- **主要操作场景**：每个P0用户故事至少有2个主要操作场景定义，每个P1用户故事至少有1个
- **Mermaid图表**：至少包含功能分解图、用户流程图、功能流程图
- **验收标准**：P0项清晰可测试
- **影响分析章节存在**：包含"影响分析"章节，且包含搜索词与改动点清单、改动点影响链、依赖闭合摘要和未覆盖风险
- **影响链基于实际搜索**：影响链中每条记录的文件路径和行号均来自实际搜索结果，**不得估算或凭记忆填写**
- **依赖关系已闭合**：对上游依赖、反向依赖、传递依赖、导出链、注册链、测试、文档、配置和外部依赖必须继续追踪
- **改动范围汇总存在**：包含直接修改文件数、间接影响文件数、传递影响文件数、需人工复核或阻断的风险说明

### P1 - 应该通过

- **影响链维度与处置方式标注完整**：每条影响链记录均有引用方式、影响层级、处置方式及说明
- **时序图**：包含完整的时序交互图
- **使用场景**：3-5个场景，采用"背景-操作-结果"格式
- **图表说明**：每个Mermaid图表都有明确的说明
- **操作场景完整性**：主要操作场景包含前置条件、操作步骤、预期结果、验证关注点

### P2 - 可以有

- **功能详情**：每个功能点都有详细说明
- **设计文档关联**：主要操作场景明确关联到设计文档的对应章节
