# 设计文档规范

> **生成约束**：本文档类型**禁用模板**。即使 `templates/设计文档.md` 存在，也不作为生成输入。
> 章节顺序、命名、图表要求严格来自本规范；所有模块/路径/接口/场景必须可追溯到上游文档或代码，
> 无来源的内容写"待补充（原因：未找到来源）"，**不得编造**。
>
> **动态适配约束**：设计文档的章节结构和图表类型必须从项目基础文件动态推断，不使用固定项目名或固定架构假设。

## 动态上下文读取（生成前必须执行）

生成设计文档前，必须读取以下项目基础文件作为动态上下文，用于适配章节结构和图表类型：

| 文件 | 读取目的 |
|------|---------|
| `CLAUDE.md` | 推断技术栈、架构约束 |
| `README.md` | 推断项目类型、快速开始 |
| `docs/architecture.md` | 推断架构约定、编码规范、架构模式 |
| `docs/auth.md` | 推断鉴权自检规则（涉及鉴权时使用） |
| `docs/security.md` | 推断安全自检规则（涉及安全时使用） |
| `docs/FAQ.md` | 推断常见故障排查（影响分析时参考） |

基于读取内容推断以下适配决策：

| 推断维度 | 推断信号 | 决策输出 |
|----------|---------|---------|
| 项目类型 | 技术栈关键词（Vue/React/Svelte→前端 SPA；Express/Koa/FastAPI/Django→后端服务；Tauri/Electron→桌面应用；浏览器扩展 manifest→扩展插件；React Native/Flutter→移动端） | 章节结构+图表类型组合 |
| 架构模式 | `docs/architecture.md` 中的核心架构模式章节 | 模块划分方式+接口规范格式 |
| 是否有数据层 | 代码中存在 ORM/Schema/Database/Migration | 是否需要 ER 图、DB 设计章节 |
| 是否有 API 层 | 代码中存在 Controller/Route/API/Endpoint | 是否需要 API 接口章节、时序图 |
| 是否有类继承 | 代码中存在 class/extends/interface/abstract | 是否需要类图 |
| 是否有鉴权 | `docs/auth.md` 存在且非空 | 是否需要鉴权流程章节 |
| 是否有安全 | `docs/security.md` 存在且非空 | 是否需要安全设计章节 |

## Mermaid 图类型适配决策

根据项目类型和架构模式推断需要的 Mermaid 图类型，不得仅使用固定图类型：

| 项目类型推断信号 | 推荐图类型 | 用途 |
|------------------|-----------|------|
| 有数据库/ORM/Schema 代码 | `erDiagram`（ER 图） | 展示实体关系和数据模型 |
| 有 API/Controller/Route 代码 | `sequenceDiagram`（时序图） | 展示请求/响应时序 |
| 有类继承/接口定义 | `classDiagram`（类图） | 展示类结构和继承关系 |
| 有组件/视图/页面代码 | `graph TB`（组件架构图） | 展示组件树和模块划分 |
| 有状态管理/事件流 | `flowchart TD`（流程图） | 展示状态流转和事件链 |
| 有微服务/消息队列 | `graph LR`（服务拓扑图） | 展示服务间通信拓扑 |

### 5. 影响分析（必须包含）

> **强制执行**：生成设计文档前，必须按 `../../../shared/document-contracts.md` 对整个项目执行完整影响分析。分析必须覆盖上游依赖、反向依赖、传递依赖、导出链、注册链、数据流、类型契约、样式、测试、文档、配置和外部依赖影响，避免改动点被其他引用或依赖时发生遗漏。

#### 执行步骤

0. **读取共享契约**：先读取 `../../../shared/document-contracts.md`，适用阶段、搜索范围、必查维度、输出格式、依赖闭合标准和 P0 门禁以该文件为准。
1. **确定核心标识符**：从需求任务中提取本次改动涉及的核心模块名、函数名、组件名、事件名、Store key、路由路径、CSS 类名/变量、公用工具函数名、配置项、依赖包名、测试路径和文档引用等，形成"搜索词与改动点清单"。
2. **按契约全项目搜索**：对每个搜索词在整个仓库执行分类搜索，记录所有命中路径与行号；不得只搜索当前目录或 `src/`。搜索必须覆盖实现点、导出入口、注册入口、公共聚合入口、测试、文档、配置与外部依赖。
3. **追踪依赖链闭合**：对每个命中点继续检查其上游依赖、调用方、导出方、注册入口、消费方、测试和文档引用，直到影响链闭合或明确记录停止原因。
4. **排除无关结果**：排除 `node_modules/`、`dist/`、`*.lock`、文档自身等非业务文件；若排除的自动生成快照可能影响验收，必须写入未覆盖风险。
5. **标注处置方式**：同步修改、保持兼容、补充验证、人工复核、无需处理。
