# 设计文档规范

> **禁用模板**。章节顺序、命名、图表须严格来自本规范；模块/路径/接口/场景必须可追溯上游或代码，无来源写"待补充（原因：未找到来源）"。
> 上游：需求任务(02)；下游：使用文档。

## 文档结构

### 1. 头部

标准头部 + 导航锚点（设计概述 | 架构设计 | 变更内容 | 实现细节 | 影响分析）。

### 2. 设计概述

100-200 字 + 3 设计原则（🎯⚡🔧）

### 3. 架构设计（必须含 Mermaid）

整体架构 `graph TB` + 说明 → 模块划分表(名称|职责|位置) → 核心流程 `flowchart TD` + 说明

### 4. 修复/变更内容（必须）

全新功能时标题改为「变更内容」。问题分析 → 方案(思路+文件清单+选择理由) → 前后对比

### 5. 影响分析（必须）

> **强制**：按 `../../../shared/impact-analysis-contract.md` 全项目影响链闭合。

四张子表同需求任务（搜索词/影响链/闭合摘要/未覆盖风险）+ 改动范围汇总。设计文档层面进一步深入到函数/组件/事件/Store/路由/CSS/配置。

> 未命中标注"未找到引用"，不可省略。

### 6. 实现细节（必须）

技术要点(做什么+怎么做+为什么) → 关键代码(50-100行+行内注释) → 依赖关系 → 测试考虑

### 7. 主要操作场景实现（必须）

每个需求任务场景：关联02场景链接 → 实现概述 → 模块与职责 → 关键代码路径 → 验证要点

### 8. 数据结构设计

数据流程图 `sequenceDiagram` + 说明
