## 检查项

### P0 - 必须通过

| 检查项 | 检查内容 | 验证方法 |
|---------|---------|---------|
| **文档头部完整** | 含版本、日期、维护者、关联文档、章节导航 | 对照 `rules/项目报告.md` 第 1 节 |
| **报告范围明确** | 纳入范围、排除范围、不确定项均有说明 | 对照规范第 3 节，结合 `git status` / 用户说明核验 |
| **交付摘要可追溯** | 交付目标、核心结果、变更规模、验证结论、当前状态均有来源 | 对照规范第 2 节，核验上游文档 / diff / 验证日志 |
| **变更文件列表完整** | 与真实变更文件清单一致，不遗漏、不新增 | 对照规范第 8 节，结合 `git diff --name-status` / 用户清单核验 |
| **前后对比完整** | 每个变更文件都有变更前后内容对比 | 对照规范第 9 节 |
| **变更前内容真实** | 变更前内容来自旧版本、diff 上下文或用户提供原文，不能根据变更后反推 | 对照规范第 9 节，核验 `git diff` / `git show` / 原始材料 |
| **路径真实有效** | 报告中引用的文件路径在仓库中真实存在 | 结合仓库文件核验 |
| **汇总表完整** | 每个变更文件都出现在汇总表中，且含验证覆盖状态 | 对照规范第 10 节 |
| **验证结果真实** | 已执行、未执行、失败的验证状态如实记录 | 对照规范第 6 节，核验命令输出或未执行原因 |
| **自我改进章节存在且证据化** | 文档最后包含 `## skills / agents / rules 自我改进（证据驱动）`，内容基于本次任务真实证据、可回指到路径/规则/检查项 | 对照规范第 11 节；抽查每条改进是否包含可追溯证据与验证方式 |

### P1 - 应该通过

| 检查项 | 检查内容 | 验证方法 |
|---------|---------|---------|
| **变更概览清晰** | 能从变更域理解整体变化，不只是文件罗列 | 对照规范第 4 节 |
| **影响评估完整** | 覆盖用户体验、功能行为、数据 / 接口、构建 / 部署、文档 / 协作 | 对照规范第 5 节 |
| **风险与遗留项明确** | 无风险时有依据，有风险时有后续动作 | 对照规范第 7 节 |
| **变更说明清晰** | 每条变更均说明原因与影响 | 对照规范第 8-10 节 |
| **影响评估合理** | 无 / 低 / 中 / 高 与实际改动范围一致 | 对照规范第 5、10 节 |
| **关键截取得当** | 超长文件只展示关键片段，并注明截取 | 结合文档内容检查 |

### P2 - 可以有

| 检查项 | 检查内容 | 验证方法 |
|---------|---------|---------|
| **读者分层友好** | 摘要适合快速阅读，对比适合开发追溯 | 主观检查 |
| **变更规模准确** | 新增 / 修改 / 删除 / 重命名数量与列表一致 | 结合变更文件列表核验 |
| **引用关系清晰** | 关联文档与章节导航便于追踪 | 结合文档内容检查 |
