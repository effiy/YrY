# 周报检查清单

> **关联规范**: [周报规范](../rules/周报.md) | [通用文档检查清单](./通用文档.md)

## P0 - 必须通过

- 文档头部完整（版本、日期、维护者、覆盖周期、关联功能目录）；无活跃功能目录时写"本周无活跃功能目录"
- KPI 数据来源自动化（`collect-weekly-kpi.js --with-logs` 已执行且输出被采用）
- KPI 总表覆盖范围（每个含 KPI 数据的功能目录各占一行；无数据时不虚构目录）
- KPI 量化与证据（五个维度有具体数值及路径）
- 维度综合有据（✅/🟡/❌ 及依据与 06/05 等来源一致）
- 本周复盘非空（含进展/亮点与问题根因）
- 链路全景图正确（KPI→复盘根因→后期规划，无 OKR 节点）
- 后期规划总表可执行（每项有类型标签、KPI 指标、验证方式、证据）
- 后期规划首条对应最薄弱 KPI
- 改进项包含"参考标准"字段且为业内通行做法而非理论书名
- 工作流程标准化四问已审视（无则写"无"，禁止跳过）
- 禁止出现象限图、矩阵图等可视化优先级表达
- 防幻觉（无虚构数据、案例、D 类陈述）

## P1 - 应该通过

- Git 统计与脚本输出一致
- 链路因果关系准确（薄弱 KPI 连到根因与规划首项）
- 后期规划优先级合理（首条对应最薄弱 KPI）
- 覆盖周期正确（自然周起止）
- 步骤 6 已执行（先 import-docs 再 wework-bot）
- execution memory 已写入（功能文档交付后）
- self-improve 已触发并输出改进提案（weekly 命令）
- doc-mermaid-expert 已调用
- 一次执行到底（无因非阻断原因中断、无"待补充"占位未推进、无拆分为多轮生成）

## P2 - 可以有

- 跨周对比
- 改进建议可执行性高
