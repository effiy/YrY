# generate-document

## 核心原则

1. **规范优先，模板为辅**：`rules/<类型>.md` 是唯一契约；`03_设计文档` 和 `05_动态检查清单` 禁用模板，必须规范驱动生成
2. **防幻觉**：技术事实必须可追溯到已读取的上游文档或代码；无据可依时写 `> 待补充（原因：…）`，禁止虚构
3. **Agent 实干**：关键步骤必须调用 agent，必须采纳其输出写入文档
4. **全文档交付**：每次调用在 `docs/<功能名>/` 下生成完整编号集（01-05, 07；06 由 implement-code 写入）
5. **默认自动决策**：尽量不打断用户，缺失信息写"待确认"并继续推进；所有 agent 必答问题自动作答，不确定时输出建议方案继续

## 何时使用

- `/generate-document init` — 生成或更新项目基础文件（见下方 init 命令章节）
- `/generate-document weekly` — 生成自然周周报，OKR/KPI 自我分析及改进规划（见下方 weekly 命令章节）
- `/generate-document <功能名>-描述` — 为某一功能在 `docs/<功能名>/` 下生成或更新完整文档集
- 不用于仅生成单份文档（应引导为全文档一次交付）

## init 命令

`/generate-document init` — 生成或更新项目基础文件，并在 `docs/项目初始化/` 下创建完整文档编号集（01-07，含 06_实施总结）。

### 检查范围

| 位置 | 必要文件 | 说明 |
|------|---------|------|
| 根目录 | `CLAUDE.md` | 项目行为准则 |
| 根目录 | `README.md` | 项目概述、技术栈与快速开始 |
| `docs/` | `architecture.md` | 项目架构约定 |
| `docs/` | `changelog.md` | 变更日志 |
| `docs/` | `devops.md` | 构建/部署/运维流程 |
| `docs/` | `FAQ.md` | 常见问题与解答 |
| `docs/` | `auth.md` | 认证/鉴权方案 |
| `docs/` | `security.md` | 安全策略与自检规则 |
| `docs/项目初始化/` | `01_需求文档.md` ~ `07_项目报告.md` | 项目初始化全文档集（含 06） |

### 执行流程

1. **规范检索**：根据文档类型和关键词检索适用的 `rules/`、`shared/`、`checklists/` 规范集合（Glob 定位 + Read 确认适用性），替代全量盲读
2. **扫描**：逐一检查 7 个基础文件和 `docs/项目初始化/` 目录是否存在，读取上游文档确认路径
3. **报告**：向用户列出缺失文件清单和用途说明
4. **Grounding**：读取项目代码结构，推断技术栈/架构/构建流程/鉴权方案
5. **生成基础文件**：对缺失文件按 `rules/项目基础文件.md` 规范生成内容；对已有文件检查与当前代码一致性，不一致则提出更新建议
6. **生成全文档集**：在 `docs/项目初始化/` 下按 5 步工作流生成 01-07（含 06_实施总结，init 命令是唯一例外，06 由 init 自行写入）
7. **自检**：按 `checklists/项目基础文件.md` + `checklists/<类型>.md` 执行质量检查，P0 全部通过才保存
8. **保存 + 知识策展 + 通知**：写入所有文件，调用 `knowledge-curator`，执行 `import-docs` + `wework-bot`

### 生成规则

详见 `rules/项目基础文件.md`。核心规则：
- **防幻觉**：所有内容必须来自实际代码扫描和文件读取，不得虚构路径或函数
- 不确定的内容标注"待补充（原因：…）"，不虚构
- 已有文件不自动覆盖，不一致时提出更新建议需用户确认
- **06_实施总结 由 init 写入**：init 命令是唯一例外，06 由 init 自行生成（记录项目初始化的实施过程和结果），其他场景 06 仍由 implement-code 写入

## weekly 命令

`/generate-document weekly` — 生成自然周周报，基于 `docs/` 下全文档集和 `.claude/` 系统状态进行 OKR/KPI 自我分析及改进规划。

`/generate-document weekly <YYYY-WNN>` — 生成指定 ISO 周的周报（默认当前周）。

### 周编号计算

- 使用 ISO 8601 周编号：`YYYY-WNN`
- 周一为每周起始日
- 示例：2026-04-28（周二）→ 2026-W17

### 产出路径

`docs/周报/<YYYY>-W<NN>.md`

### 执行流程

1. **周编号推断**：从当前日期（或用户指定的 `<YYYY-WNN>`）计算 ISO 周编号和覆盖周期（周一~周日日期范围）
2. **规范检索**：必须调用 `spec-retriever`，加载 `rules/周报.md` + `rules/通用文档.md` + `checklists/周报.md` + `checklists/通用文档.md` + 共享规范
3. **动态上下文读取**（按 `rules/周报.md` 严格执行）：
   - 读取项目基础文件（`CLAUDE.md`、`README.md`、`docs/architecture.md`、`docs/FAQ.md`、`docs/auth.md`、`docs/security.md`）
   - 扫描 `docs/` 下所有 `<功能名>/` 目录的全文档集（01-07）
   - 读取 `.claude/shared/evidence-and-uncertainty.md`
   - 读取 `.claude/agents/memory/` 下记忆文件
   - 获取本周 git 日志（`git log --since="<本周一>" --until="<本周日>" --oneline`）
4. **weekly-analyzer 推断**：必须调用 `weekly-analyzer`，先读取其记忆文件获取历史推断模式，自动作答 5 个必答问题（功能目录案例、完成度/质量、系统自改进、项目自改进、下五步优先级），直接采纳其 OKR/KPI/自改进候选数据
5. **周报生成**：按 `rules/周报.md` 文档结构（1.文档头部 → 2.OKR → 3.KPI复盘+下五步 → 4.系统自改进 → 5.项目自改进）逐章节生成，每条断言关联来源（文件路径或代码路径）
6. **自检**：读取 `checklists/周报.md` + `checklists/通用文档.md`，进入 plan 模式动态自检规划，P0 全部通过才保存；未通过最多自修复 1 轮
7. **必须调用 `quality-tracker`**：将本次 P0/P1/P2 统计追加到 `.claude/agents/memory/quality-tracker.md`
8. **保存 + 知识策展 + 通知**：
   - 保存 `docs/周报/<YYYY>-W<NN>.md`
   - 必须调用 `knowledge-curator` 策展可复用知识
   - 必须调用 `tool-chain-logger` 记录调用链路日志
   - 必须执行 `import-docs` 同步到 YiAi / YiDocs
   - 必须调用 `message-pusher` 基于本次真实执行数据动态生成推送文案
   - 必须执行 `wework-bot` 发送完成通知

## 全文档包含的类型

| 类型 | 模板 | 规范 | 文件名 | 依赖 | 生成方式 |
|------|------|------|--------|------|---------|
| 需求文档 | ✅ | ✅ | `01_需求文档.md` | - | 模板骨架 + 规范约束 |
| 需求任务 | ✅ | ✅ | `02_需求任务.md` | 需求文档 | 模板骨架 + 规范约束 |
| 设计文档 | ❌ | ✅ | `03_设计文档.md` | 需求任务 | **仅规范驱动** |
| 使用文档 | ❌ | ✅ | `04_使用文档.md` | 设计文档 | 规范驱动 |
| 动态检查清单 | ❌ | ✅ | `05_动态检查清单.md` | 需求任务+设计文档 | **仅规范驱动** |
| 实施总结 | — | — | `06_实施总结.md` | — | **仅 implement-code 写入**（init 命令除外，init 自行写入 06） |
| 项目报告 | ❌ | ✅ | `07_项目报告.md` | 设计文档 | 规范驱动 + 真实变更数据 |
| 周报 | ❌ | ✅ | `docs/周报/<YYYY>-W<NN>.md` | 全文档集+系统状态 | **仅规范驱动** + weekly-analyzer 推断 |

## 前置条件检查（运行前必须执行）

进入 5 步工作流之前，必须检查项目基础文件是否存在。缺失文件将影响文档生成的 Grounding 质量，**必须提醒用户补充**：

| 位置 | 必要文件 | 说明 |
|------|---------|------|
| 根目录 | `CLAUDE.md` | 项目行为准则 |
| 根目录 | `README.md` | 项目概述与快速开始 |
| `docs/` | `architecture.md` | 项目架构约定 |
| `docs/` | `changelog.md` | 变更日志 |
| `docs/` | `devops.md` | 构建/部署/运维流程 |
| `docs/` | `FAQ.md` | 常见问题与解答 |
| `docs/` | `auth.md` | 认证/鉴权方案 |
| `docs/` | `security.md` | 安全策略与自检规则 |

检查规则：
- 文件存在 → 读取作为 Grounding 上下文
- 文件不存在 → 向用户明确提示缺失文件名和用途，建议补充后再继续
- 用户选择跳过 → 在生成文档中标注"待补充（原因：项目缺少 `{filename}`）"，不阻断流程

## 5 步工作流

### 步骤 1：解析 + 规范检索

- 从用户输入提取 `{功能名}`，按默认决策矩阵自动补齐
- 根据文档类型和关键词检索适用的 `rules/`、`shared/`、`checklists/` 规范集合（Glob 定位 + Read 确认适用性），替代全量盲读
- 产出：功能名、规范集、上游文档清单

### 步骤 2：上游 Grounding + 影响分析

- 按依赖顺序读取前一阶段文档（01→02→03）；设计文档同时阅读相关源码
- **必须调用 `impact-analyst`**（仅需求任务和设计文档）：
  - 先读取 `../../shared/document-contracts.md` 影响分析契约
  - 先读取 `.claude/agents/memory/impact-analyst.md` 获取历史分析经验
  - 必答问题自动作答，执行全项目影响链闭合分析，产出四个部分：搜索词与改动点清单、改动点影响链、依赖闭合摘要、未覆盖风险
  - **影响链分析结果必须写入**需求任务第 6 章 / 设计文档第 5 章
- 产出：事实-来源映射表、影响链闭合

### 步骤 3：专家生成（自动决策，不打断用户）

- **必须调用 `architect`**（设计文档生成前）：
  - 先读取 `.claude/agents/memory/architect.md` 获取历史架构决策
  - 5 个必答问题由 architect 基于上游文档和代码自动作答，**无需用户确认**，直接采纳其输出
  - 其模块划分、接口规范、数据流方案**必须采纳到**设计文档架构设计章节
- **必须调用 `planner`**（设计文档生成前）：
  - 必答问题由 planner 基于需求任务自动作答，**无需用户确认**，直接采纳其输出
  - 其实施策略与风险清单**必须采纳到**设计文档
- architect 与 planner **可并行调用**
- 产出：设计方案、模块划分、接口规范

### 步骤 4：逐文档生成 + 动态自检

- 严格按 `rules/<类型>.md` 章节顺序产出每份文档
- 每条技术断言必须在文中关联来源（上游文档链接或代码路径）
- 无据可依的章节标注"待补充"
- 版本信息头部填写实际值（大模型名称、AI 工具名称）
- **设计文档生成后必须调用 `code-reviewer` 审查架构一致性**：
  - P0 问题必须修复后再保存
  - P1/P2 问题记录但不阻断
- **自检前必须进入 plan 模式进行动态自检规划**：
  - 读取 `checklists/<类型>.md`，根据当前任务内容逐项评估适用性
  - 识别本次任务特有的检查要点，动态调整检查清单（增/删/改优先级、标注不适用的通用项）
  - 产出动态自检计划后再执行正式自检
- 自检：加载调整后的检查清单，P0 全部通过才保存为通过状态；未通过最多自修复 1 轮，仍不通过也必须保存并标注
- P0/P1/P2 统计追加到 `.claude/agents/memory/` 对应记忆文件
- 产出：全文档编号集

### 步骤 5：保存 + 知识策展 + 通知

- 保存 `docs/<功能名>/` 下 01-05, 07（06 不创建）
- 将可复用知识追加到 `.claude/agents/memory/` 对应记忆文件
- 将本次调用链路日志和错误案例记录到记忆文件
- **必须执行 `import-docs`**：文档同步到 YiAi / YiDocs
- **必须调用 `message-pusher`**：基于本次真实执行数据动态生成推送文案（AI 调用链、起止时间、用量、改进建议、下一步），不得直接套用静态模板
- **必须执行 `wework-bot`**：采用 `message-pusher` 生成的文案发送完成通知

## Agent 调用契约

所有 agent 必答问题均自动作答、不打断用户。不确定时输出建议方案或"待确认"继续推进。

| Agent | 触发步骤 | 自动作答要点 | 采纳规则 |
|-------|---------|-------------|---------|
| `impact-analyst` | 2 | 自动执行全项目影响链闭合分析 | 影响链分析结果必须写入文档对应章节 |
| `architect` | 3 | 5 个必答问题基于上游文档+代码自动推断 | 模块划分、接口规范必须采纳到架构设计章节 |
| `planner` | 3 | 基于需求任务自动推断实施策略与风险 | 实施策略与风险清单必须采纳到设计文档 |
| `code-reviewer` | 4 | 自动审查架构一致性+文档结构 | P0 问题必须修复；P1/P2 记录不阻断 |
| `weekly-analyzer` | weekly | 扫描 docs/ 下全文档集推断 OKR/KPI 和自改进建议 | OKR/KPI/自改进候选数据必须采纳到周报对应章节 |
| `message-pusher` | 5 | 基于本次执行数据动态生成生动总结推送文案 | 推送文案必须采用其输出 |

## 人工介入门槛

仅当以下情况才打断用户：
- 无法解析功能名且无法生成合理默认
- P0 章节必须基于上游/代码，但仓库完全找不到来源且无法"待补充"降级
- 同一请求指向两个不同功能域且无法兼容

## 动态检查清单生成规则

1. 从 `02_需求任务.md` 抽取操作场景（场景名/前置条件/操作步骤/预期结果）
2. 从 `03_设计文档.md` 抽取实现细节（涉及模块/关键代码路径/验证要点）
3. 用 `find-skills` 为每个场景映射验证技能
4. 每条检查项必须标注来源（需求任务章节锚点/设计文档章节锚点/代码路径）

## 项目报告强制要求

1. 交付摘要（目标、结果、变更规模、验证结论）
2. 变更概览与影响评估
3. 验证结果（如实记录）
4. 变更文件列表
5. skills/agents/rules 证据驱动自我改进

## 自检规则

- P0 全部通过才保存为通过状态
- 最多自修复 1 轮
- 仍不通过也必须保存，顶部标注未通过项

## 防幻觉约束

- 设计文档/动态检查清单不得读取模板
- 章节顺序严格来自 `rules/<类型>.md`
- 事实-来源映射：内部维护 `事实 → {上游文档章节 | 代码路径:行}`
- 未知处理：P1/P2 章节无来源写"待补充"；P0 缺来源拒绝输出
- 代码路径必须实际存在于仓库
- Mermaid 图节点必须对应真实模块

## 支持文件结构

```
.claude/skills/generate-document/
├── SKILL.md                 # 本文件
├── README.md                # 快速开始
├── checklist.md             # 主检查清单入口
├── checklists/              # 各文档类型检查清单
├── rules/                   # 各文档类型规范（唯一契约）
└── templates/               # 模板（设计文档、动态检查清单不使用）
```
