# wework-bot 消息契约

## API 契约

默认请求：

```text
POST https://api.effiy.cn/wework/send-message
```

请求体：

```json
{
  "webhook_url": "<企业微信机器人 webhook>",
  "content": "<消息正文>"
}
```

认证：

- Header 使用 `X-Token`
- `X-Token` **仅来自系统环境变量** `API_X_TOKEN`（不接受命令行参数或本地配置兜底）

## webhook 配置

支持五种方式，优先级从高到低：

1. `WEWORK_WEBHOOK_URL` / `WEWORK_WEBHOOK_KEY`：全局环境兜底机器人
2. `--robot` + `--config`：从配置文件的 `robots` 中选择机器人（配置只提供 *_env 字段）
3. `--agent` + `--config`：从配置文件的 `agents` 映射到机器人
4. 配置文件中的 `default_robot`

```text
https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key=<key>
```

## 多机器人配置

配置文件结构：

```json
{
  "default_robot": "general",
  "robots": {
    "general": {
      "webhook_key_env": "WEWORK_WEBHOOK_KEY_GENERAL"
    },
    "security": {
      "webhook_key_env": "WEWORK_WEBHOOK_KEY_SECURITY"
    }
  },
  "agents": {
    "generate-document": "docs",
    "import-docs": "docs",
    "wework-bot": "general",
    "implement-code": "code",
    "security-reviewer": "security",
    "code-reviewer": "general"
  }
}
```

机器人支持字段：

| 字段 | 说明 |
|------|------|
| `webhook_url_env` | 从环境变量读取完整 webhook |
| `webhook_key_env` | 从环境变量读取 webhook key |
| `api_url` | 可选，覆盖默认发送 API |

推荐只提交 `config.json`（或示例配置），真实凭证只放在系统环境变量中。配置文件不得包含 `webhook_url` / `webhook_key` / token 等任何明文密钥。

## 消息格式

### 阶段通知（通用格式）

```text
📣 <流程名> <状态>
━━━━━━━━━━━━━━━━━
🎯 结论：<一句话说明结果和影响>
🛠️ 流程：<流程名>
📌 功能：<功能名或文档路径>
📝 描述：<100字以内描述>
📍 阶段：<阶段名>
📊 状态：<开始 / 通过 / 阻断 / 失败 / 需要人工介入>
🔎 关键：<1-2 个关键事实或原因>
🤝 AI 调用：skills <N> / agents <N> / mcp <N> / tools <N>
🔗 调用链：<find-skills→...→wework-bot>
📁 测试路径：<tests/ 内 spec/page/fixture 数量与逸出情况>
🧫 测试统计：<spec/case/通过/失败>（如有）
🔁 修复轮次：<阶段2: N 轮 / 阶段6: N 轮>（如有）
📦 产物：<代码 / 测试 / 文档>（如有）
📈 指标：<lint / 影响链 / 冒烟时间>（如有）
🌐 影响：<影响范围、交付范围或风险范围>
📎 证据：<报告路径 / 命令 / MCP 调用序列 / 结果摘要>
📂 报告：<docs/<功能名>/06_实施总结.md 或对应文档路径>
☁️ 文档同步：docs → YiAi（创建 N，覆盖 N，失败 N）
⏱️ 用时：<本次会话总耗时>
🪙 会话用量：<本次会话 Token 或用量摘要，与 Cursor 可核对>
🟢 开始时间：<YYYY-MM-DD HH:mm:ss>（如有）
🌿 分支：<git-branch>
🔖 提交：<short-sha>
🤖 模型：<模型名>
🧰 工具：<主要工具清单>
🕒 最后更新：<YYYY-MM-DD HH:mm:ss>
👉 下一步：<需要接收者执行的动作；无需行动时写"无需人工介入">
```

状态值建议：`开始` / `通过` / `阻断` / `失败` / `需要人工介入`

电梯法则要求：每条推送必须让接收者在 10 秒内读懂"发生了什么、影响是什么、证据在哪里、下一步做什么"。正文必须分行显示，禁止把结论、描述、关键事实和下一步塞进同一行。

描述要求：每条推送必须包含 `描述：` 或 `📝 描述：` 行，内容不超过 100 字，用于说明本次推送的背景、结果或下一步动作。

业务度量要求（适用于完成、阻断、门禁异常通知）：

- 必须根据本次执行的真实情况补齐 `🤝 AI 调用` / `🔗 调用链`，二者数据应与 `06_实施总结.md` 中的「AI 调用记录」「§1 AI 调用流程图 / §2 AI 调用时序图」一致
- 若已生成 `06_实施总结.md`，应补齐 `📐 实施总结图表`（§1 节点数、§2 参与者数或等价摘要）、`🧩 MCP 明细`（各 browser_* 次数或汇总）、`🧾 待办与风险`（P1/P2 与 §9 对应）、`🗂️ 状态回写`（`01/02/.../07` 与 §5.5 结论）；可通过 `send-message.js` 的 `--diagram-summary`、`--mcp-breakdown`、`--backlog`、`--status-rewrite` 传入
- 必须基于 `artifact-contracts.md §2.2 测试路径门禁扫描命令` 真实结果填写 `📁 测试路径`，未执行扫描时不得发送
- 凡是涉及到测试文件的产物，必须在数字中区分 `tests/` 内与 `tests/` 外（`tests/` 外应为 0）
- `📦 产物` / `🧫 测试统计` / `🔁 修复轮次` / `📈 指标` 数字必须来自实际命令、MCP 操作或 git diff
- `📂 报告` 行必须指向实际生成的报告路径（固定为 `docs/<功能名>/06_实施总结.md` 或兜底 `docs/99_agent-runs/`）

元信息要求：每条推送必须包含 `🤖 模型：...`、`🧰 工具：...`、`🕒 最后更新：YYYY-MM-DD HH:mm:ss` 三行。时间必须精确到秒；若调用方未传入，发送脚本必须使用本地当前时间自动补齐。`🌿 分支` / `🔖 提交` 由脚本自动从 git 检测，可通过 `--no-auto-git` 关闭。

**会话成本（必含）**：`⏱️ 用时`（本次会话耗时）与 `🪙 会话用量`（本次会话 Token/用量）必须在长流程完成、阻断、门禁异常及建议含度量的阶段通知中出现；若正文未写且未传 `--duration` / `--token-usage`，`send-message.js` 会注入与 Cursor 核对的缺省说明行，**禁止**留空。

会话与复盘（长流程完成/阻断类推送须**可核对**）：

- `⏱️ 用时` / `🟢 开始时间`：与本次会话/执行真实起止一致；未记时可使用脚本缺省句或写「未记录」，**禁止**猜测秒数冒充精确值。
- `🪙 会话用量`：须与 Cursor 用量面板、账单或导出一致；无法取得时使用脚本缺省句或「未提供，请从 Cursor 用量核对」，**禁止**编造。
- `💡 改进建议`：只写与**本次已发生**问题相关的可执行项（如证据链缺口、可自动化步骤），≤500 字；避免空泛「加强测试」类套话。

正文转义：`content` 中不得向用户展示字面量 `\n`（反斜杠加字母 n）。多行正文应使用真实换行（`--content-file` 推荐），或由 `send-message.js` 在加载后对误写 `\\n` / `\\t` 做归一化。

## generate-document 观测点

- 开始生成时发送阶段通知（可使用 `--agent generate-document`）
- 开始生成全文档或关键单文档
- P0 自检失败并写入目标文档
- 缺参或缺少上游材料，已写入 `docs/99_agent-runs/`
- `import-docs` 同步完成后发送完成通知，`☁️ 文档同步` 行必须使用真实创建 / 覆盖 / 失败数字

## import-docs 观测点

- `docs` 标准导入开始
- 导入完成并产生 created / overwritten / failed 汇总
- 导入失败但主流程继续交付时，向 `wework-bot` 提供失败摘要

## implement-code 观测点

- 阶段 0 文档预检完成
- 阶段 2 动态检查门禁失败或通过
- 阶段 4 模块实施完成
- 阶段 6 冒烟测试失败或通过
- 阶段 7 实施总结已生成
- 任意阻断需要人工介入
- 任意应执行门禁未执行、无证据、被跳过或降级未记录

## 安全约束

- 不得提交真实 `X-Token`、完整 webhook URL 或 webhook key。
- 不得把任何密钥写入仓库文件（含 `config.json`、示例配置或脚本参数）。
- 日志和最终回复必须脱敏。
- 缺少 token 或 webhook 时必须停止，不尝试匿名发送。
- 使用 `--dry-run` 验证参数时不得实际发送消息。
- 若门禁异常通知发送失败，必须在实施总结或 `docs/99_agent-runs/` 兜底记录中写明发送失败原因、脱敏路由、模型、工具和最后更新时间。
