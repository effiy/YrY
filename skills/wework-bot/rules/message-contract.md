# wework-bot 消息契约

## API 契约

默认请求：

```text
POST https://api.effiy.cn/wework/send-message
```

请求体：

```json
{
  "webhook_url": "<企业微信机器人 webhook>",
  "content": "<消息正文>"
}
```

认证：

- Header 使用 `X-Token`
- `X-Token` **仅来自系统环境变量** `API_X_TOKEN`（不接受命令行参数或本地配置兜底）

客户端请求须携带与浏览器访问 `effiy.cn` 一致的 Origin/Referer；脚本使用与 Chrome 一致的 `User-Agent`、`Sec-Fetch-*`、`sec-ch-ua` 等头，避免网关拒绝非浏览器 UA。实现上与浏览器侧 `curl 'https://api.effiy.cn/wework/send-message' ...` 的请求头对齐，并使用 keep-alive Agent。

## webhook 配置

优先级从高到低：

1. `--robot`（若提供），否则 `--agent` 映射到 robot，否则使用 `default_robot`
2. `config.json` 中选中 robot 的 `webhook_url` 或 `webhook_key`（顶层同名字段可作兜底）

```text
https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key=<key>
```

## 多机器人配置

配置文件结构：

```json
{
  "default_robot": "general",
  "api_url": "https://api.effiy.cn/wework/send-message",
  "robots": {
    "general": {
      "webhook_url": "https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key=<key>"
    },
    "security": {
      "webhook_url": "https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key=<另一 key>"
    }
  },
  "agents": {
    "generate-document": "general",
    "security-reviewer": "security",
    "implement-code": "general"
  }
}
```

机器人支持字段：

| 字段 | 说明 |
|------|------|
| `webhook_url` | **推荐**：完整企业微信 webhook URL（与 API 请求体字段同名） |
| `webhook_key` | 可选：仅 key；脚本会拼接为 `https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key=...` |
| `api_url` | 可选，覆盖默认发送 API |

仓库默认配置使用明文 **`webhook_url`** 与单一 `general` 路由；若需多群分流，可增设多个 robot 并在 `agents` 中映射。

`X-Token` 仍只来自环境变量 `API_X_TOKEN`，不得写入配置文件。

## 消息格式

### 管理层阅读优先（摘要 / 明细分层）

推送面向群内 **决策者（老板）与执行者**，必须避免「一行一个 emoji、重复同义信息、技术栈刷屏」。

排版规则：

1. **摘要段（分隔线之上）**：控制在 **约 10 行内**，让读者先读完 `🎯 结论`、`🌐 影响`、`👉 下一步`。用词短句；**禁止**在摘要里重复粘贴完整调用链、长 MCP 清单。
2. **明细段（分隔线之下，标题建议固定）**：`────────────【以下为核对明细】────────────` 之后，再放 `🔗 调用链`、`🤝 AI 调用`、`📐/🧩`、`🤖/🧰/🕒`、`🌿/🔖`、测试与产物统计等。
3. **合并同类项**：
   - 用 **`📌 范围：`** 一行替代原 `🏷️ 项目名` + `🧾 功能描述`（格式：`项目 / 功能 — 一句话范围`）。
   - 用 **`⏱️ 会话：`** 一行合并耗时与 Token/用量：`耗时 <…> · 用量 <…>`；无可靠数据时写可核对说明（如「未记录，请核对账单」）。**不再要求**必须拆成 🟢 开始 / 🔴 结束 / ⏱️ / 🪙 四行；若审计需要，可在 **明细段** 再补拆分。
4. **`📎 证据：`** 摘要里 **一行为主**（主路径或最关键一条）；路径罗列放入明细段。
5. **`💡 改进建议`**：仅在确有 actionable 项时追加，**≤120 字、≤2 条**，禁止空话（如泛泛「加强测试」）。

电梯法则：**摘要段**须让人在约 10 秒内读懂「发生了什么、业务影响、要不要处理」。每条信息独占一行；禁止把结论与下一步塞进同一行。

描述：`📝 描述：` 仍 ≤100 字。

### 推送正文（不提供固定模板）

推送由 `message-pusher` 基于事实整理；契约约束 **分层必含** 与 **可核对**，不提供填充式模板句。

#### A. 摘要段·必含（所有类型）

- **标题行**：事件类型 + 流程名（如 `📄 文档生成完成`、`⛔ 代码实施阻断`）
- `🎯 结论：`
- `📝 描述：`
- `📌 范围：`（合并项目与功能一句话）
- `👉 下一步：`

#### B. 摘要段·完成 / 阻断 / 门禁异常须补充

- `🌐 影响：`
- `📎 证据：`（单行摘要或「主路径 + 详见明细」）
- `⏱️ 会话：`（合并耗时 + 用量，见上）
- 阻断类还须：`❌ 原因：`（≤2 条）、`🧭 恢复点：`
- 门禁类还须：`🔍 门禁：`、`📊 结果：`、`❌ 原因：`（可与门禁结论合并表述，避免重复三行同义话）

#### C. 明细段·核对与审计（长流程完成 / 阻断 / 门禁）

与 `06_实施总结.md`、`artifact-contracts.md`、真实命令输出 **一致时** 写入以下行；**无来源则省略该行**，禁止假数字：

- `🔗 调用链`、`🤝 AI 调用`（可与 `06` 摘要一致）
- `🤖 模型`、`🧰 工具`、`🕒 最后更新`（精确到秒）
- `🌿 分支`、`🔖 提交`
- 实施总结相关：`📐 实施总结图表`、`🧩 MCP 明细`、`🧾 待办与风险`、`🗂️ 状态回写`、`📁 测试路径`（须基于真实扫描；未执行则写明「未执行扫描」而非编造）、`📦 产物`、`☁️ 文档同步`、`📂 报告`
- `📂 报告` 须指向真实路径：`docs/<功能名>/06_实施总结.md` 或 `docs/99_agent-runs/` 兜底

### 完成通知·精简模板（摘要 + 明细）

> `generate-document` / `implement-code` 流程结束须按下列结构发送：**先摘要、后明细**。明细字段按上节 C 从真源补齐，可删减无数据的行。

### 格式强制要求（摘要优先）

- 分隔线 **至多两条**：标题下一条 `━━━`，摘要与明细之间一条「核对明细」。
- `🎯 结论`、`📝 描述`、`👉 下一步` 为硬性必含（摘要段）。
- 完成 / 阻断 / 门禁类：`🌐 影响`、`📎 证据`、`⏱️ 会话` 为摘要段必含（会话合并一行即可）。
- **明细段** 才要求补齐（在有来源时）`🤖` `🧰` `🕒`；勿在摘要重复粘贴。
- 数字须来自执行结果，禁止 `<占位符>` 直接发送。
- 全文长度：**摘要段建议 ≤600 字**；含明细 **全文建议 ≤2000 字**。门禁 / 阻断类 **不得** 因压缩而删掉摘要中的原因、影响、恢复点；冗长技术细节移到明细段。
- `☁️ 文档同步`：若 `docs` 不存在：`☁️ 文档同步：docs 不存在，跳过导入`

正文转义：`content` 中不得出现字面量 `\n`。多行用 `--content-file` 或脚本 `normalizeMessageText`。

## generate-document 观测点

- 开始生成时发送阶段通知（可使用 `--agent generate-document`）
- 开始生成全文档或关键单文档
- P0 自检失败并写入目标文档
- 缺参或缺少上游材料，已写入 `docs/99_agent-runs/`
- `import-docs` 同步完成后发送完成通知，`☁️ 文档同步` 行必须使用真实创建 / 覆盖 / 失败数字

## import-docs 观测点

- `docs` 标准导入开始
- 导入完成并产生 created / overwritten / failed 汇总
- 导入失败但主流程继续交付时，向 `wework-bot` 提供失败摘要

## implement-code 观测点

- 阶段 0 文档预检完成
- 阶段 2 动态检查门禁失败或通过
- 阶段 4 模块实施完成
- 阶段 6 冒烟测试失败或通过
- 阶段 7 实施总结已生成
- 任意阻断需要人工介入
- 任意应执行门禁未执行、无证据、被跳过或降级未记录

## 安全约束

- 不得提交真实 `X-Token`、完整 webhook URL 或 webhook key。
- 不得把任何密钥写入仓库文件（含 `config.json`、示例配置或脚本参数）。
- 日志和最终回复必须脱敏。
- 缺少 token 或 webhook 时必须停止，不尝试匿名发送。
- 若门禁异常通知发送失败，必须在实施总结或 `docs/99_agent-runs/` 兜底记录中写明发送失败原因、脱敏路由、模型、工具和最后更新时间。
