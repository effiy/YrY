# Improve Codebase Architecture

> 发现架构摩擦，提出**深化机会**——把浅模块转为深模块的重构。目标是可测试性和 AI 可导航性。

## 词汇

整个技能使用下列术语，不要漂移到 "组件"、"服务"、"API"、"边界"：

- **模块** — 有接口与实现的任何东西（函数、class、包、切片）。
- **接口** — 调用者需知的一切：类型、不变式、错误模式、顺序、配置。不止类型签名。
- **实现** — 接口后面的代码。
- **深度** — 接口处的杠杆：大量行为被小接口隐藏。**深** = 高杠杆。**浅** = 接口几乎与实现一样复杂。
- **接缝** — 接口所在之处；不改原地就能改变行为的地方。（用这个，不用 "边界"）
- **适配器** — 在接缝处满足接口的具体东西。
- **杠杆** — 调用者从深度中获得的东西。
- **局部性** — 维护者从深度中获得的东西：变更、bug、知识集中在一处。

关键原则：
- **删除测试**：想象删除这个模块。复杂度消失 = 透传。复杂度回到 N 个调用方 = 它在赚位置。
- **接口即测试面。**
- **一个适配器 = 假设接缝。两个适配器 = 真实接缝。**

本技能受项目领域模型*约束*。领域语言给出好接缝的命名；ADR 记录本技能不应重新讨论的决定。

## 流程

### 1. 探索

先读项目的 [CONTEXT.md](../../../CONTEXT.md) 和涉及区域的 [ADR](../../../docs/adr/)。

然后用 Agent 工具 `subagent_type=Explore` 走代码库。不遵循死板启发式——有机探索，注意摩擦：

- 理解一个概念需要来回跳转多个小模块？
- 哪些模块**浅**——接口几乎与实现一样复杂？
- 哪些纯函数被提取出来只为了可测试性，但真正的 bug 藏在调用方式里（没有**局部性**）？
- 哪些紧密耦合的模块在接缝处泄漏？
- 代码库的哪些部分未测试，或难以通过当前接口测试？

对任何怀疑是浅模块的应用**删除测试**：删除它，复杂度是集中还是只是转移？"是的，集中" 就是你要的信号。

### 2. 呈现候选

呈现一个编号的深化机会列表。每个候选：

- **文件** — 涉及哪些文件/模块
- **问题** — 当前架构为何造成摩擦
- **方案** — 用简明语言描述如何改变
- **收益** — 用局部性和杠杆解释，以及测试如何改善

**用 CONTEXT.md 词汇描述领域，用本文词汇描述架构。** 如果 CONTEXT.md 定义了 "Order"，就说 "Order intake 模块"——不说 "FooBarHandler"，也不说 "Order service"。

**ADR 冲撞**：如果候选与现有 ADR 矛盾，仅在摩擦大到值得重新讨论 ADR 时才呈现。明确标记（如 "与 ADR-0007 矛盾——但因…值得重新打开"）。不要列出 ADR 禁止的所有理论重构。

不要提议接口。问用户："**这些里面哪个想深入？**"

### 3. 烧烤循环

用户选了候选后，进入烧烤对话。和他们走设计树——约束、依赖、深化后模块的形状、接缝后藏什么、哪些测试存活。

边做边发生副作用：

- **给深化后模块命名的概念不在 CONTEXT.md 里？** 添加术语到 CONTEXT.md——与 `/grill-with-docs` 相同的纪律。如果文件不存在则惰性创建。
- **对话中锐化了模糊术语？** 当场更新 CONTEXT.md。
- **用户有支撑性理由拒绝候选？** 提议 ADR："希望我记载为 ADR 吗，让未来架构评审不再重新建议它？" 仅在理由确实会被未来探索者需要避免重新建议时提议——跳过短暂的（"不值得现在做"）和自明的。
- **想探索深化后模块的替代接口？** 用 §接口设计。

## 接口设计

> 设计/评估替代接口。在用户确定深化候选后使用。

对于每个替代方案：
- **形状** — 类型签名 + 不变式 + 错误模式
- **调用者的代价** — 使用这个接口需要知道/做多少东西
- **实现的深度** — 接口后面现在和未来能藏多少
- **哪些测试存活** — 现有测试哪些不需要改就能通过
- **一致性与惊喜** — 这个接口是否像项目里已有的类似模块的接口

接口的核心要求：**从调用者的角度看，不变式最小也最不难理解。**

### ADR 级决定

如果用户在替代接口间做出的选择是：
- 难逆转
- 缺上下文会奇怪
- 真实权衡的结果

→ 主动提议："你想把这个写为 ADR 吗？这样以后我们不会在替代方案间反复。"

## Red Flags — 暂停回到 Iron Law

- "这个模块太小不值得重构"
- "加个抽象层就解决问题了" — 没有第二个调用方 = 浅模块
- "我会同时重构这几个模块" — 一次一个模块，观察效果
- "不需要查 CONTEXT.md，我知道领域叫什么"
- "这个重构不影响测试" — 运行确认
