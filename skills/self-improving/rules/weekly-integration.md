# Weekly Integration

> Defines how the agent harness performance system (`self-improving`) feeds consolidated performance intelligence into the weekly report, and the contract between `self-improving` and `weekly-analyzer`.

## 1. Producer / Consumer Split

| Role | Skill / Agent | Responsibility |
|------|---------------|----------------|
| Producer | `self-improving` | Extract, normalize, and aggregate per-document reflection sections. |
| Consumer | `weekly-analyzer` | Interpret aggregated data, prioritize, and write weekly report §5.2 and §5.3. |

`weekly-analyzer` MUST NOT fabricate reflection data. If `self-improvement-aggregate.md` is missing or empty, the analyzer writes "no reflection data collected this week" and continues.

## 2. Aggregation Output Format

`self-improving weekly` produces:

```
docs/weekly/<YYYY-MM-DD~YYYY-MM-DD>/self-improvement-aggregate.md
```

### 2.1 File Structure

```markdown
# Self-Improvement Aggregate (<week-range>)

> Generated by `self-improving` — do not edit manually.

## Workflow Standardization Review Summary

### Per-Feature Table

| Feature | Doc | Q1 | Q2 | Q3 | Q4 |
|---------|-----|----|----|----|----|
| ... | ... | ... | ... | ... | ... |

### Cross-Feature Patterns

- **Repeated manual operations**: list features and operations
- **Missing decision criteria**: list fuzzy decisions that recurred
- **Information silos**: list silos that spanned multiple features
- **Feedback loop gaps**: list features without clear owner/acceptance

## System Architecture Evolution Thinking Summary

### Per-Feature Table

| Feature | Doc | A1 Bottleneck | A2 Evolution Node | A3 Risk & Rollback |
|---------|-----|---------------|-------------------|--------------------|
| ... | ... | ... | ... | ... |

### Cross-Feature Patterns

- **Shared bottlenecks**: group features by bottleneck type
- **Common evolution nodes**: highlight repeated architectural proposals
- **Risk clusters**: flag high-risk patterns appearing in ≥2 features

## Data Quality

- Total documents scanned: N
- Documents with missing sections: N
- Coverage rate: N%
```

## 3. Integration Points in Weekly Report

### 3.1 Section 5.2 — Workflow Standardization Review

`weekly-analyzer` reads the **Cross-Feature Patterns** from the aggregate and produces the four-question review. It MUST:

1. Use the aggregate counts (e.g., "3 out of 5 features reported manual repetitive operations").
2. Cite specific features as evidence (e.g., "`user-login` and `order-checkout` both reported fuzzy boundary decisions").
3. Map findings to concrete rule/checklist/file paths for improvement.

### 3.2 Section 5.3 — System Architecture Evolution Thinking

`weekly-analyzer` reads the **Per-Feature Table** and **Cross-Feature Patterns** from the aggregate. It MUST:

1. List architecture bottlenecks that appeared in ≥1 feature delivery this week.
2. Propose next evolution nodes, referencing the aggregate evidence.
3. Include risk and rollback plans copied or summarized from the aggregate.

## 4. Timing and Orchestration

In `/generate-document weekly` workflow:

1. **Before** `weekly-analyzer` is invoked, the orchestrator MUST run:
   ```bash
   node skills/self-improving/scripts/collect-self-improvement.js --week <week-start>
   ```
2. The script writes `self-improvement-aggregate.md`.
3. `weekly-analyzer` receives the aggregate path as part of its context (e.g., via `--aggregate` or file read).
4. If the script fails, `weekly-analyzer` falls back to "aggregate unavailable" and continues without blocking.

## 5. Backwards Compatibility

- Existing `self-improve.js` (execution-memory based) continues to run after `weekly` command Stage 6, producing `self-improve-proposal.md`. Its output is appended to the weekly report as a separate section ("System Self-Improvement Proposals").
- The new `self-improving` aggregate is a **distinct** input stream focused on per-document reflection, not execution-memory statistics. Both may coexist in the weekly report.
