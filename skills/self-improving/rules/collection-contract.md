# Collection Contract

> Agent harness performance data contract. Defines the standardized telemetry format for `Workflow Standardization Review` and `System Architecture Evolution Thinking` sections that must appear in every document generated by `generate-document` and `implement-code`.
>
> These sections are the **primary performance signals** of the agent harness: they measure process friction, decision quality, information flow, and architectural health per delivery.

## 1. Scope

Applies to all documents under `docs/<feature-name>/` produced by the following skills:

- `generate-document` → `01_`, `02_`, `03_`, `04_`, `05_`, `07_`
- `implement-code` → `06_process-summary.md`

These sections are **mandatory** regardless of document type or change level (T1/T2/T3).

## 2. Placement

Append the two sections **after** the existing postscript `## Postscript: Future Planning & Improvements`. Order:

1. `Postscript: Future Planning & Improvements`
2. `## Workflow Standardization Review`
3. `## System Architecture Evolution Thinking`

If a document has no postscript, append directly at the end of the document.

## 3. Workflow Standardization Review

### 3.1 Purpose

Force a standardized four-question retrospective on the process used to produce **this specific document** (or the implementation summarized in it).

### 3.2 Format

```markdown
## Workflow Standardization Review

1. **Repetitive labor identification**: Were there more than 2 manual repetitive operations while producing this document / feature? Can they be scripted or templated?
   - Answer: Yes / No / Partial
   - Evidence: (specific operations, e.g., "manually copied interface definitions into 03 and 04")

2. **Decision criteria missing**: Were there fuzzy decision points relying on personal experience? Can they be沉淀 as checklist or rules?
   - Answer: Yes / No / Partial
   - Evidence: (specific decisions, e.g., "module boundary between auth and session was decided ad-hoc")

3. **Information silos**: Did this feature rely on oral transmission or temporary docs? Can they be unified into a single source?
   - Answer: Yes / No / Partial
   - Evidence: (specific information gaps, e.g., "API rate limit was communicated in Slack, not in 01")

4. **Feedback loop**: Do problems found during this document / implementation have a clear follow-up owner and acceptance node?
   - Answer: Yes / No / Partial
   - Evidence: (owner and acceptance criteria, or gap description)
```

### 3.3 Constraints

- **Answer must be one of**: `Yes`, `No`, `Partial`.
- **Evidence is mandatory**: every answer must include at least one concrete sentence.
- **Prohibit fabrication**: if truly none, answer `No` with evidence "none observed during this delivery".
- **Per-document scope**: review the process of producing **this** document, not the entire project history.

## 4. System Architecture Evolution Thinking

### 4.1 Purpose

Capture architecture-level reflection for the feature covered by this document. If the document has no architecture relevance (e.g., pure copy changes), answer "N/A" with justification.

### 4.2 Format

```markdown
## System Architecture Evolution Thinking

- **A1. Current architecture bottleneck**: (performance / maintainability / scalability / security / none / N/A)
  - Evidence: (specific code path, design tension, or metric)

- **A2. Next natural evolution node**: (concrete next step, or "none" if stable)
  - Evidence: (which module/interface would change, and why)

- **A3. Risks and rollback plans for evolution**: (risk description + rollback strategy)
  - Evidence: (data migration, dual-write, feature flag, etc.)
```

### 4.3 Constraints

- **Bottleneck type is mandatory**: must select from the enumerated list or write `N/A`.
- **N/A justification**: if architecture is untouched by this feature, state "This document does not introduce or modify architecture deliverables".
- **Evidence is mandatory**: even for "none" answers, explain why no bottleneck/evolution node is expected.
- **Rollback plan is mandatory** whenever an evolution node is proposed.

## 5. Extraction Rules (for `collect-self-improvement.js`)

### 5.1 Section Boundaries

Use regex anchors (case-insensitive):

- Start: `^##\s+Workflow Standardization Review\s*$`
- End: `^##\s+System Architecture Evolution Thinking\s*$` or next `^##\s+` or EOF
- Start: `^##\s+System Architecture Evolution Thinking\s*$`
- End: next `^##\s+` or EOF

### 5.2 Field Parsing

For each `Workflow Standardization Review`:

- Extract `Q1`–`Q4` answers via pattern: `Answer:\s*(Yes|No|Partial)`
- Extract `Evidence` via pattern: `Evidence:\s*(.+?)(?=\n\s*\d+\.|\n##|$)`
- Default to `"missing"` if not found.

For each `System Architecture Evolution Thinking`:

- Extract `A1` bottleneck via pattern: `\*\*A1[.\s]*Current architecture bottleneck.*?\n\s*-\s*(.+?)(?=\n\s*-\s*\*\*A2|$)`
- Extract `A2` evolution node via pattern after `A2` header.
- Extract `A3` risk/rollback via pattern after `A3` header.
- Default to `"missing"` if not found.

### 5.3 Output Record

Each extracted record is a single JSON line:

```json
{
  "feature": "user-login",
  "doc": "03_design-document.md",
  "week": "2026-04-27~2026-05-03",
  "workflow": {
    "q1": { "answer": "Yes", "evidence": "..." },
    "q2": { "answer": "No", "evidence": "..." },
    "q3": { "answer": "Partial", "evidence": "..." },
    "q4": { "answer": "Yes", "evidence": "..." }
  },
  "architecture": {
    "bottleneck": "scalability",
    "evolution_node": "shard session store",
    "risk_rollback": "risk: data migration; rollback: dual-write"
  },
  "extracted_at": "2026-05-03T12:00:00Z"
}
```

## 6. Missing Section Handling

If a document lacks either section:

1. Log a P1 gap: `"missing Workflow Standardization Review in docs/<feature>/<doc>"`.
2. Emit a stub record with all fields set to `"missing"`.
3. Append a reminder to the document header (do not auto-edit the document body unless in update mode).
