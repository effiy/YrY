# rui-analysis 分析方法论

> 代码与架构静态分析的规则和方法，独立于实现细节。

## 五维分析

### 1. 复杂度分析

McCabe 圈复杂度: M = E − N + 2P，简化为 `分支点数 + 1`。

每个 `if`/`else`/`switch`/`case`/`for`/`while`/`do`/`catch`/三元运算符/`&&`/`||` 计 1 个分支点。

| 复杂度等级 | M 范围 | 风险 | 建议 |
|-----------|--------|------|------|
| Simple | 1-5 | 低 | 健康 |
| Moderate | 6-10 | 中 | 可接受 |
| Complex | 11-20 | 高 | 考虑拆分 |
| Very Complex | 21-50 | 很高 | 需要重构 |
| Extreme | 50+ | 极高 | 必须拆分 |

### 2. 耦合分析

- import/export 依赖图
- 循环依赖检测
- fan-in/fan-out 比
- 稳定性指标 I = Ce/(Ca+Ce)

### 3. 文件膨胀

- >500 行告警
- >1000 行严重
- 文件增长趋势
- God Module 检测

### 4. 依赖健康

- 版本新鲜度
- 依赖树深度
- 未使用依赖
- 安全漏洞扫描

### 5. 架构边界

- 层次违规检测
- 跨边界依赖
- 循环依赖群组
- 模块边界穿越

## 严重级别

| 级别 | 标识 | 含义 | 处置 |
|------|------|------|------|
| Critical | 🚫 | 阻断性 | 必须修复 |
| High | 🔴 | 高风险 | 本次迭代修复 |
| Medium | 🟡 | 中风险 | 计划修复 |
| Low | 🟢 | 低风险 | 跟踪观察 |
| Info | 🔵 | 信息 | 无需动作 |

## 核心规则

| # | 规则 | 设计理由 |
|---|------|---------|
| 1 | 规约驱动，由 implementing agent 执行 | 分析本身不修改代码 |
| 2 | 只读操作，不修改源码 | 分析安全 |
| 3 | 可量化指标 > 主观判断 | 验现实 |
| 4 | 热点 Top-5% 优先报告 | 惜注意 |
| 5 | 趋势对比需基线数据 | 无基线无法判断退化 |

## 输出格式

### 五维分析报告

每个维度的发现必须包含：

| 字段 | 说明 | 示例 |
|------|------|------|
| 文件路径 | 发现所在的文件 | `skills/rui/SKILL.md` |
| 严重级别 | Critical/High/Medium/Low/Info | `High` |
| 指标值 | 量化的分析结果 | `M=23` |
| 阈值 | 触发该级别的阈值 | `>20` |
| 建议 | 可操作的修复建议 | `拆分为 3 个函数，每个 M≤10` |

## 边界场景

| 场景 | 处置 |
|------|------|
| 分析范围为空目录 | 输出空报告，标注 `empty-scope` |
| 文件不可读 | 跳过，记录 `unreadable` |
| 超大文件 (>10K 行) | 采样分析前 5000 行，标注 `truncated` |
| 无基线数据 | 仅输出当前值，标注 `no-baseline` |
| 二进制文件误入 | 跳过，标注 `binary-skipped` |
| 符号链接 | 跟随，标注 `symlink` |
