# 文档规范

## 文档分层

```
docs/
├── 组件文档/                  ← --from-code 前端探索生成（项目级参考文档）
│   └── {project}/
│       └── {component-name}/  ← 单个组件的完整文档目录
│           ├── 00-索引.md     ← 阅读地图：按角色/场景推荐阅读路径，文件间导航
│           ├── 01-组件概述.md  ← 组件身份 + Props/Events/Expose 完整契约
│           ├── 02-状态与依赖.md ← 状态定义 + 状态流图 + 依赖图 + 外部引用
│           ├── 03-样式与交互.md ← 样式隔离 + 视觉状态矩阵 + 动画 + DOM/事件
│           └── 04-操作场景.md  ← 主要操作场景（Given→When→Then）+ 边界异常
├── 接口文档/                  ← --from-code 后端探索生成（项目级参考文档）
│   └── {project}/
│       └── {resource-name}/   ← 单个资源的完整文档目录
│           ├── 00-索引.md     ← 阅读地图：按角色/场景推荐阅读路径，文件间导航
│           ├── 01-接口概述.md  ← 端点清单 + 契约表 + 序列图 + 版本
│           ├── 02-数据模型.md  ← DTO/ORM + 存储结构 + 迁移 + 容量
│           ├── 03-中间件与安全.md ← 中间件链 + 威胁模型 + 校验 + 错误码
│           └── 04-操作场景.md  ← 调用场景 + 异常场景 + 性能约束
└── 故事任务面板/              ← rui doc/code 生成（故事级执行文档）
    └── <project>-<name>/
        ├── 01-故事任务.md      ← 引用 → ../../组件文档/{project}/{component-name}/00-索引.md
        └── ...
```

**项目级参考文档**（`docs/组件文档/{project}/{component-name}/`、`docs/接口文档/{project}/{resource-name}/`）：由 `--from-code` 空 req 探索生成。每个资源独立一子目录，含 00 索引（阅读入口）+ 01~04 四份分层文档。描述代码的当前状态（what is），不包含任务/AC/设计决策。供故事目录下的 01~08 引用。

**故事级执行文档**（`docs/故事任务面板/<project>-<name>/`）：由 `rui doc`/`rui code` 生成。描述要做什么、怎么做、做了什么（what to do / what was done）。引用项目级参考文档作为事实基线。

### 文件级导航约定

每个 01~04 文件首尾包含标准导航块，阅读者无需退回目录即可在文件间跳转：

```markdown
> **导航**: [← 00-索引](./00-索引.md) · [↑ 组件文档](../) · [02-状态与依赖 →](./02-状态与依赖.md)
```

文件模板自带导航占位符，生成时自动填充前驱/后继文件。00-索引.md 作为阅读入口，提供按角色和场景的推荐路径。

### 引用格式

故事文档引用项目级参考文档时优先指向 00-索引，按需深入具体文件：

```markdown
全套文档见 [UserTable 组件](../../组件文档/YiWeb/user-table/00-索引.md)。
接口契约详见 [UserTable §2](../../组件文档/YiWeb/user-table/01-组件概述.md#2-接口契约)。
API 调用场景参见 [用户 API §4](../../接口文档/YiWeb/user-api/04-操作场景.md#4-创建用户)。
```

## 故事拆分

pm 收到需求后按决策树判断是否拆分：

```
需求 → 单一场景/单一角色? → 是 → 一个故事
                    → 否 → 可独立验证的子需求? → 逐个子需求拆为独立故事
                    → 否 → 一个故事 + 明确范围边界
```

### 拆分信号

| 信号 | 说明 | 处理 |
|------|------|------|
| 多角色 | 需求涉及 ≥2 个独立角色（管理员/终端用户/开发者） | 按角色拆分 |
| 多入口 | 需求涉及 ≥2 个独立入口（Web/API/CLI/后台） | 按入口拆分 |
| 可独立交付 | 子需求可独立上线并产生用户价值 | 拆为独立故事 |
| 跨边界 | 需求跨越前后端且任一端复杂度 > 3 模块 | 前端故事 + 后端故事 |
| 无拆分信号 | 单一场景、单一角色、不可再分 | 一个故事 |

### 拆分约束

- 每个故事必须有独立的 AC（可独立验证）
- 故事间依赖显式标注在 §1 依赖字段
- 拆分后逐故事串行，先完成的故事先验证
- 禁止过度拆分：一个函数/一个 API 不构成独立故事

### 故事目录命名

| 故事类型 | 命名格式 | 示例 |
|---------|---------|------|
| 功能故事 | `<project>-<feature>` | `YiWeb-user-login`、`YiAdmin-role-mgmt` |
| 重构故事 | `<project>-<scope>-refactor` | `YiWeb-auth-refactor` |
| 修复故事 | `<project>-<scope>-fix` | `YiWeb-login-timeout-fix` |

规则：project 为大驼峰（如 `YiWeb`），story 为 kebab-case（如 `user-login`）。总长度 ≤ 64 字符。

## 故事目录结构

按项目类型自动选择目录骨架。pm 在文档生成阶段根据项目类型和故事内容决定创建哪些文件。

### 全栈故事

```
docs/故事任务面板/<project>-<name>/
├── 01-故事任务.md              ← 唯一真相源
├── 02-后端技术评审.md           ← 服务架构/API/数据模型/安全
├── 03-前端技术评审.md           ← 组件树/状态流/样式/交互
├── 04-测试用例评审.md           ← 覆盖矩阵/Gate 映射/用例
├── 05-后端实施报告.md           ← 交付文件/偏差/审查/存储
├── 06-前端实施报告.md           ← 交付文件/偏差/审查/性能
├── 07-测试用例报告.md           ← 执行结果/覆盖率/Gate 结论
├── 08-自改进复盘.md             ← 六维评估/改进项/经验沉淀
├── 00-消息通知列表.md           ← wework-bot 自动生成（不入库审查）
├── 10-{领域}.md                 ← 按需：补充文档（领域建模、专项设计等）
├── 1x-{专题}.md                 ← 按需：专题文档（迁移方案、集成方案等）
├── .improvement/proposals.jsonl  ← 自改进提案（追加写入）
└── .memory/
    ├── execution-memory.jsonl    ← 执行记忆（追加写入）
    └── rui-state.json            ← 管线状态（单对象覆盖写入）
```

### 纯前端故事

后端不涉及任何服务端改动时（纯 UI/组件/样式/交互），省略后端文件：

```
docs/故事任务面板/<project>-<name>/
├── 01-故事任务.md
├── 03-前端技术评审.md           ← 必选
├── 04-测试用例评审.md
├── 06-前端实施报告.md           ← 必选（前端项目必选前端实施报告）

```
docs/故事任务面板/<project>-<name>/
├── 01-故事任务.md
├── 02-后端技术评审.md           ← 必选
├── 04-测试用例评审.md
├── 05-后端实施报告.md           ← 必选

```

省略：`03-前端技术评审.md`、`06-前端实施报告.md`。

## 文件创建生命周期

```
需求解析 → 自适应规划 → 影响分析 → 架构设计 → 文档生成 → 预检 → 测试先行 → 实现 → 验证 → 自改进 → 交付
                                                      │                │         │         │        │
                                                      ▼                ▼         ▼         ▼        ▼
                                                   01 创建       02/03/04   05/06/07   08    00(通知)
                                                   故事任务        创建       创建      创建    创建
```

| 阶段 | 创建文件 | 负责人 | 条件 |
|------|---------|--------|------|
| 文档生成 | `01-故事任务.md` | pm | 必创建 |
| 文档生成(架构设计后) | `02-后端技术评审.md` | coder + security | 涉及后端 |
| 文档生成(架构设计后) | `03-前端技术评审.md` | coder | 涉及前端 |
| 文档生成(架构设计后) | `04-测试用例评审.md` | tester | 必创建 |
| 文档生成(补充) | `10-{领域}.md` / `1x-{专题}.md` | pm 决策 | 见补充文档规则 |
| 验证 | `05-后端实施报告.md` | coder | 涉及后端且有 02 |
| 验证 | `06-前端实施报告.md` | coder | 涉及前端且有 03 |
| 验证 | `07-测试用例报告.md` | tester | 必创建 |
| 自改进 | `08-自改进复盘.md` | pm + reporter | 必创建 |
| 交付 | `00-消息通知列表.md` | wework-bot | 自动生成 |
| 实现→验证 | `.memory/execution-memory.jsonl` | 自动 | 追加写入 |
| 全阶段 | `.memory/rui-state.json` | 自动 | 每次阶段变更覆盖写入 |
| 自改进 | `.improvement/proposals.jsonl` | self-improve | 有提案时追加写入 |

> **关键约束**：
> - 01 是唯一真相源 — 所有其他文件的引用必须最终追溯到 01
> - 02/03/04 在文档生成阶段创建，05/06/07 在验证阶段创建 — 不可提前
> - `.memory/` 和 `.improvement/` 由脚本自动管理，人工不编辑
> - 补充文档（10/1x）在文档生成阶段由 pm 按决策表判断是否创建

## 完整度判定

`list.js` 按文件存在性判定故事状态。判定逻辑以项目类型为基准 — 纯前端故事不要求后端文件，反之亦然：

| 状态 | 条件 |
|------|------|
| `not_started` | 01 不存在 |
| `docs_in_progress` | 01 存在，必选文件有缺失 |
| `docs_done` | 所有必选文档文件存在 |
| `code_in_progress` | 文档齐全，存在部分实施报告 |
| `code_done` | 所有必选文件存在 + 08 存在 |
| `blocked` | rui-state.json 中 blocked=true |

### 项目级参考文档完整度

`list.js` 同样判定 `docs/组件文档/` 和 `docs/接口文档/` 下的目录完整度：

| 状态 | 条件 |
|------|------|
| `complete` | 00~04 五份文件全部存在 |
| `partial` | 目录存在，00 存在，01~04 有缺失 |
| `stale` | 目录存在，00 不存在（旧格式，需迁移） |
| `empty` | 项目目录存在但无组件/接口子目录 |

> `recommend.js` 优先推荐 `stale` 和 `partial` 的文档补齐，其次推荐源码存在但尚无文档的组件/接口。

## --from-code 自主探索模式

`/rui doc --from-code` 不传 req 时，pm 按项目类型差异化探索源码：

### 通用流程

1. 检测项目类型（前端/后端/全栈），选择对应探索策略
2. 扫描项目目录，识别可独立文档化的单元
3. 每个候选输出：推荐名称(kebab-case)、覆盖范围、源码证据（Level A 路径列表）、优先级
4. 按优先级排序（有源码无文档 > 有文档过时 > 其他）
5. 用户选择后，生成**项目级参考文档**写入 `docs/组件文档/{project}/{component-name}/` 或 `docs/接口文档/{project}/{resource-name}/`，每个资源一个独立子目录，含 00 索引 + 01~04 五份文件
6. 随后自动推荐可基于参考文档创建的故事

### 前端项目：组件发现 → docs/组件文档/{project}/{component-name}/

| 步骤 | 操作 | 产出 |
|------|------|------|
| 1. 组件扫描 | 搜索 `.vue` / `.jsx` / `.tsx` / `.svelte` 文件，排除 `node_modules` | 组件文件列表 |
| 2. 接口提取 | 解析每个组件的 Props/Events/Expose（Vue）或 Props/回调（React） | 组件契约表（参数名、类型、必要性） |
| 3. 依赖分析 | 识别组件子组件、composables/hooks、store 引用 | 组件依赖图 |
| 4. 状态追踪 | 提取 data/reactive/ref/useState/useReducer，标注状态变更路径 | 状态流图 |
| 5. 样式检查 | 识别样式隔离方案（Scoped/CSS Modules/Shadow DOM/Tailwind） | 样式策略标注 |
| 6. 生成参考文档 | 每个选中组件创建 `docs/组件文档/{project}/{component-name}/`，含 00~04 五份文件 | 推荐列表（P0: 核心业务组件无文档 > P1: 普通组件无文档 > P2: 已有过时文档） |

### 组件文档目录结构

每个组件一个子目录 `{component-name}`（kebab-case），含 00 索引 + 01~04 分层文档：

```
docs/组件文档/{project}/{component-name}/
├── 00-索引.md         ← 阅读入口：身份卡片 + 按角色/场景的阅读路径
├── 01-组件概述.md      ← 组件身份 + Props/Events/Expose 完整契约
├── 02-状态与依赖.md     ← 状态定义 + 状态流图 + 依赖图 + 外部引用
├── 03-样式与交互.md     ← 样式隔离 + 视觉状态矩阵 + 动画 + DOM/事件
└── 04-操作场景.md      ← 主要操作场景（Given→When→Then）+ 边界异常
```

#### 00-索引.md 章节结构

00 是阅读入口，让任何角色在 30 秒内定位到自己需要的文件。不包含具体技术内容，仅做导航。

| 章节 | 内容 | 必选 |
|------|------|------|
| 元信息 | version/timestamp/model/branch | ✓ |
| 身份卡片 | 组件名、文件路径、一句话职责、所属项目、技术栈 | ✓ |
| 角色阅读路径 | "我是开发者/测试者/产品"三条路径，每条 ≤3 步 | ✓ |
| 场景阅读路径 | "我要调用此组件 / 我要修改此组件 / 我要排查Bug"三条路径 | ✓ |
| 文件清单 | 01~04 文件列表，每行：文件名 + 一句话概括 + 预计阅读时间 | ✓ |
| 关联资源 | 关联的接口文档、故事目录、上游组件链接 | 有则填 |

**角色阅读路径示例：**

| 角色 | 路径 | 预计时间 |
|------|------|---------|
| 调用方开发者 | 00 → 01(§2 接口契约) → 04(场景) | 5 min |
| 维护者 | 00 → 01(全) → 02(全) → 03(全) | 15 min |
| 测试者 | 00 → 04(全场景) → 01(§2 Props 必填/可选) | 8 min |
| 设计审查 | 00 → 03(§1 样式策略) → 04(场景过渡图) | 5 min |

#### 01-组件概述.md 章节结构

| 章节 | 内容 | 必选 |
|------|------|------|
| 元信息 | version/timestamp/model/branch + 导航块 | ✓ |
| §1 组件身份 | 文件路径、注册路径/命名空间、挂载容器、组件类型（展示/容器/布局） | ✓ |
| §2 接口契约 | Props(名称/类型/默认值/必填/说明) + Events(名称/参数/触发时机) + Expose(方法/参数/返回值) | ✓ |
| §3 使用示例 | 最小可用示例代码 + 典型用法代码 | ✓ |
| §4 版本兼容 | 当前版本 + 已知 Breaking Changes + 迁移指引 | 有则填 |
| §5 关联索引 | 引用状态文档(→02)、样式文档(→03)、场景文档(→04) | ✓ |

> 01 是「API 参考手册」——查阅型文档。开发者想查某个 prop 的类型，30 秒内定位。

#### 02-状态与依赖.md 章节结构

| 章节 | 内容 | 必选 |
|------|------|------|
| 元信息 | version/timestamp/model/branch + 导航块 | ✓ |
| §1 状态全景 | 所有状态字段(名称/类型/初始值/来源/持久化) + 响应式/派生/方法分类 | ✓ |
| §2 状态流图 | mermaid flowchart：用户操作→状态变更→UI 更新 + 外部调用路径 | ✓ |
| §3 子组件依赖 | mermaid graph：组件→子组件树 + 每个子组件的 Props 透传标注 | 有子组件时 |
| §4 外部引用 | composables/hooks(路径/函数/用途) + store 引用(store名/字段/操作) + 第三方库 | ✓ |
| §5 生命周期 | 挂载/更新/卸载各阶段的初始化、订阅、清理逻辑 | ✓ |

> 02 是「架构蓝图」——理解型文档。改一个状态字段前，先看这张图知道会波及谁。

#### 03-样式与交互.md 章节结构

| 章节 | 内容 | 必选 |
|------|------|------|
| 元信息 | version/timestamp/model/branch + 导航块 | ✓ |
| §1 样式策略 | 隔离方案(Scoped/CSS Modules/Shadow DOM/Tailwind) + 设计 token 路径 + 禁止事项 | ✓ |
| §2 视觉状态矩阵 | 正常/加载/空/错误/禁用 五状态 × 每个视图区域的表现 | ✓ |
| §3 动画过渡 | 动画元素/类型/时长/缓动/触发条件 | 有动画时 |
| §4 DOM 与事件 | 挂载容器/创建方式/生命周期 + 事件监听/处理/清理时机 | ✓ |
| §5 命名空间 | 注册命名空间 + 样式前缀约定 + 加载顺序位置 | ✓ |

> 03 是「视觉规范」——还原型文档。设计师核对视觉状态，开发者查样式隔离规则。

#### 04-操作场景.md 章节结构

| 章节 | 内容 | 必选 |
|------|------|------|
| 元信息 | version/timestamp/model/branch + 导航块 | ✓ |
| §1 场景总览 | 场景列表(名称/触发角色/前置条件/覆盖的 Props/Events) + 场景过渡图(mermaid) | ✓ |
| §2 正常场景 | 每个场景：Given(前置状态)→When(用户操作)→Then(UI结果/Event输出) | ✓ |
| §3 边界场景 | 空数据、极限值、并发操作、权限不足各场景的处理行为 | ✓ |
| §4 异常场景 | 加载失败、超时、数据校验失败等异常态的处理与恢复 | ✓ |

> 04 是「用户手册」——验证型文档。测试者按场景逐条验证，产品核对交互是否符合预期。

#### 组件文档阅读路径

```
调用方开发者          维护/修改者            测试者              设计/产品审查
