# 架构深化

> 发现架构摩擦，把浅模块转为深模块。架构深化的目标是提高模块的"深度"——接口简单但实现强大。

[核心概念](#核心概念) · [深化流程](#深化流程) · [识别信号](#识别信号) · [Red Flags](#red-flags)

## 核心概念

| 概念 | 定义 | 示例 |
|------|------|------|
| **模块** | 有接口与实现的任何东西（函数/class/包/切片） | `lib/fs.mjs` 的 `readFile` 函数 |
| **接口** | 调用者需知的一切：类型、不变式、错误模式、顺序。不止类型签名 | 函数签名 + 可能抛出的异常 + 副作用 |
| **深度** | 接口后的行为量 / 接口复杂度。深 = 高杠杆 | `JSON.parse()` — 接口简单(1个参数)，实现强大(解析引擎) |
| **浅模块** | 接口≈实现，透传多过处理 | 仅转发参数的 wrapper 函数 |
| **接缝** | 不改原地就能改行为的地方 | 依赖注入点、回调函数、配置项 |
| **删除测试** | 想象删除它：复杂度消失 = 透传；回到 N 个调用方 = 它有价值 | 删除后需要 N 处修改 = 它在赚位置 |

## 深化流程

```
1. 探索 — 读 ADR，注意架构摩擦信号
2. 识别 — 定位浅模块（接口≈实现、透传、多余 getter）
3. 分析 — 计算深度：实现复杂度 / 接口复杂度
4. 呈现 — 候选模块 + 涉及文件 + 方案 + 收益
5. 执行 — 用户选定后走设计树，一次一个模块
```

## 识别信号

### 浅模块信号

| 信号 | 示例 | 深化方向 |
|------|------|---------|
| 透传函数 | `function getX() { return this.x; }` | 合并到调用方或删除 |
| 拆分过细 | 3 个函数各 5 行，总有一个调用方 | 合并为 1 个函数 |
| 接口比实现复杂 | 5 个参数，内部只做一次转发 | 简化接口或内联实现 |
| 单调用方抽象 | 只有一个调用方的 wrapper | 展开到调用方 |

### 深模块信号

| 信号 | 示例 | 价值 |
|------|------|------|
| 小接口大实现 | `JSON.parse()` — 1 个参数 | 高杠杆 |
| 多调用方 | `lib/constants.mjs` — 被 15+ 模块导入 | 高复用 |
| 隐藏复杂度 | `lib/fs.mjs` — 封装文件操作细节 | 降低调用方认知负担 |

## 架构摩擦检测

| 摩擦类型 | 表现 | 诊断 |
|------|------|------|
| **循环依赖** | A → B → A | 提取共享接口到独立模块 |
| **跨层依赖** | 底层 import 上层 | 反转依赖方向或提取共享抽象 |
| **God Module** | 单文件 500+ 行，20+ 函数 | 按职责拆分为多个模块 |
| **散弹式修改** | 改一个功能需改 5+ 文件 | 相关代码不够内聚，合并目录 |
| **紧耦合** | 改 A 必改 B | 引入接口/抽象层解耦 |

## Red Flags

- "加个抽象层就行" — 无第二调用方 = 浅模块，不抽象
- "同时重构几个模块" — 一次一个，避免范围蔓延
- "这个接口以后会用到" — YAGNI，等有第二个调用方再抽象
- "拆细一点更清晰" — 拆分过细 = 浅模块，增加认知负担
