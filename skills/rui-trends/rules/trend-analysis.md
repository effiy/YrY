# rui-trends 趋势分析规则

> 技术趋势发现和分析的规则，独立于实现细节。

## 数据源

| # | 数据源 | URL | 提供信号 | 主要消费方 |
|---|--------|-----|---------|----------|
| 1 | GitHub Trending | `github.com/trending` | 当前热门仓库 · 语言趋势 | D0 基线偏离 · D3 复杂度 · D6 文档过时 |
| 2 | OSS Insight | `ossinsight.io` | 仓库排名 · 指标对比 | D5 依赖退化 · 计划选型验证 |
| 3 | TrendShift | `trendshift.io` | star 增长量/率 · 排名变化 · 快速上升/下降 | D3 架构简化 · D5 依赖健康 |
| 4 | Top-Starred | `github.com/search` | 社区验证的高星项目 | D5 选型参照 · 计划参考实现 |

## 消费方

| 诊断 | 数据源 | 用途 | 触发条件 |
|------|--------|------|---------|
| D0 基线偏离 | GitHub Trending | 检测技术栈是否落后于社区主流 | 当前技术栈连续 3 个月未出现在 trending |
| D3 复杂度 | GitHub Trending + TrendShift | 识别更简洁的替代方案 | 当前方案复杂度 > 同类 2x |
| D5 依赖退化 | OSS Insight + Top-Starred | 对比依赖版本健康度 | 依赖版本落后 ≥ 2 major |
| D6 文档过时 | GitHub Trending | 对比文档更新频率 | 文档 mtime < 源码 commit 超过 30 天 |

## 数据新鲜度

| 频率 | 数据源 | 缓存策略 | 过期判定 |
|------|--------|---------|---------|
| 每日 | GitHub Trending | 24h 缓存 | 超过 24h 重新抓取 |
| 每周 | OSS Insight | 7d 缓存 | 超过 7d 重新查询 |
| 每周 | TrendShift | 7d 缓存 | 超过 7d 重新查询 |
| 按需 | Top-Starred | 30d 缓存 | 按需刷新 |

## 降级策略

| 情况 | 降级行为 | 恢复方式 |
|------|---------|---------|
| GitHub Trending 不可达 | 使用本地缓存数据，标注 `stale` | 下次自动重试 |
| OSS Insight API 限流 | 跳过 OSS Insight，仅用其他源 | 等待限流窗口过期 |
| TrendShift 不可达 | 降级为 GitHub Trending + Top-Starred | 下次自动重试 |
| 全部数据源不可达 | 输出空报告，标注 `data-unavailable` | 提示用户手动检查网络 |
| 缓存过期且无法刷新 | 使用过期缓存，标注 `expired-cache` | 下次自动重试 |

## 输出格式

### 结构化报告

趋势报告必须包含以下区块：

| 区块 | 内容 | 格式 |
|------|------|------|
| 摘要 | 当前技术趋势概述 (≤3 句) | 文本 |
| 热门仓库 Top-10 | 排名 + 仓库名 + stars + 趋势方向 | 表格 |
| 语言趋势 | 语言 + 排名变化 + 占比 | 表格 |
| 快速上升项目 | 项目名 + 增长率 + 潜在影响 | 表格 |
| 快速下降项目 | 项目名 + 下降率 + 风险提示 | 表格 |
| 诊断建议 | 基于趋势的 D0/D3/D5/D6 诊断建议 | 列表 |

## 集成点

| 消费方 | 消费方式 | 数据格式 |
|--------|---------|---------|
| rui-yry (自改进) | 读取趋势报告 → D0/D3/D5/D6 诊断 | Markdown 报告 |
| rui-plan (计划) | 技术选型验证 → 参考实现推荐 | Markdown 报告 |
| rui-skills (技能发现) | 委托 — 趋势中的技能发现转交 rui-skills | 直接委托 |
| 通知面板 | 趋势摘要展示 | 结构化 JSON |

## 核心规则

| # | 规则 | 设计理由 | 违反后果 |
|---|------|---------|---------|
| 1 | 技能发现委托 rui-skills | SRP 分离 — 趋势是数据，技能是工具 | 职责混淆，两个技能都变弱 |
| 2 | 趋势数据仅作参考，不自动决策 | 人为判断不可替代 | 自动跟风引入不成熟技术 |
| 3 | 数据源不可达时降级为缓存 | 外部不可达对策 — L1 自包含 | 趋势数据缺失，诊断降级 |
| 4 | 趋势报告结构化输出 | 可被管线消费 | 下游无法解析，需人工提取 |
| 5 | 不推荐未经验证的新技术 | 社区热度 ≠ 生产就绪 | 引入不稳定依赖 |
| 6 | 趋势分析结果标注数据新鲜度 | 可追溯 | 过期数据误导决策 |

## 趋势指标

| 指标 | 计算方式 | 阈值 |
|------|---------|------|
| 项目热度 | GitHub stars 增长率 (周) | > 5% 为上升 |
| 社区活跃度 | Issues/PRs 关闭率 (月) | > 70% 为健康 |
| 技术采纳率 | npm 下载量增长率 (月) | > 10% 为增长 |
| 维护响应 | 平均 Issue 响应时间 | < 7 天为健康 |
| 破坏性变更频率 | MAJOR 版本发布间隔 | > 6 个月为稳定 |

## 边界场景

| 场景 | 处置 |
|------|------|
| 全部数据源不可达 | 输出空报告，标注 `data-unavailable` |
| 单一数据源不可达 | 使用其他源，标注缺失源 |
| 缓存过期 | 使用过期缓存，标注 `expired-cache` |
| 查询无结果 | 告知用户，建议调整关键词 |
| API 返回格式变更 | 降级为原始文本，标注 `format-changed` |
| 跨数据源结论矛盾 | 并列展示，标注冲突，不仲裁 |
