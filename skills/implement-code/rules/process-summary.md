# 过程总结规范

> implement-code 阶段 4 生成的总结文档内容与格式。

## 1. 核心约束

| 编号 | 约束 |
|------|------|
| S0-1 | 总结文档必须保存到 `docs/<功能名>/06_实施总结.md` |
| S0-2 | 必须包含 AI 调用流程图和时序图（实际调用路径，不画理想路径） |
| S0-3 | 必须包含变更文件清单（路径 + 变更类型 + 关联模块） |
| S0-4 | 阻断时必须生成阻断版总结，记录已产生产物和阻断原因 |
| S0-5 | 必须回写 `01/02/03/04/05/07` 的实施状态 |
| S0-6 | 正常完成时必须包含动态检查清单最终复查（P0/P1/P2 统计） |

---

## 2. 文档结构（章节顺序不可变）

| 节号 | 标题 | 说明 |
|------|------|------|
| §0 | 任务概览 | 时间、模型、分支、最终状态 |
| §1 | AI 调用流程图 | Mermaid flowchart，含实际调用节点 |
| §2 | AI 调用时序图 | Mermaid sequenceDiagram，真实消息往返 |
| §3 | 变更文件清单 | 所有变更文件路径、类型、模块、是否在 tests/ 下 |
| §4 | 验证结果 | 门禁报告 + 动态检查清单复查 |
| §5 | 状态回写记录 | 01/02/03/04/05/07 回写结果 |
| §6 | 未解决问题与后续建议 | P1/P2 + 自我改进 + 可执行下一步 |
| §7 | 通知记录 | wework-bot、import-docs（若有） |

---

## 3. 变更文件清单要求

| 字段 | 说明 |
|------|------|
| 文件路径 | 完整相对路径 |
| 变更类型 | 新增 / 修改 / 删除 |
| 关联模块 | Store / Component / Route / E2E 等 |
| 说明 | 变更内容摘要 |
| 是否在 tests/ 下 | 测试文件必须为"是" |

所有 spec、原型页、fixture、截图必须在 `tests/` 下。

---

## 4. 自我改进（证据驱动）

每次实施结束必须产出改进 `.claude/skills/`、`.claude/agents/`、`rules/` 的建议：

| # | 分类 | 问题 | 证据 | 建议路径 | 最小改动点 | 验证方式 |
|---|------|------|------|----------|------------|----------|

只能引用本总结中已出现的证据位置，不得虚构。

---

## 5. 可执行下一步

每条须含：依据（引用 §/文件/检查项）+ 验证方式（命令/清单号/锚点），缺一不可。

---

## 6. 阻断版总结

阻断时使用简化结构：阻断概要 + 已产生产物 + 阻断详情 + 建议恢复操作。
阻断版总结保存到同一文件路径；后续恢复成功时由完整版覆盖。

---

## 7. 禁止事项

- 流程图添加未实际调用的节点
- 变更清单遗漏文件
- 测试文件在 `tests/` 外不阻断
- 阻断时未生成总结直接终止
- 动态检查清单有未完成项时写成"已完成"
