# E2E 测试规范

> CDN SPA 测试策略：手动浏览器验证 + createBaseView 集成检查。  
> **`implement-code` 准入与证据标准**（编码前 Gate A / 编码后 Gate B）以 **[`implement-code-testing.md`](./implement-code-testing.md)** 为真源；本文件约定目录结构、`data-testid`、Mock 与自动化偏好。

---

## 1. 核心约束

| 编号 | 约束 |
|------|------|
| E0-1 | 每个场景有明确的验证步骤和预期结果（来自动态检查清单） |
| E0-2 | 所有可交互 UI 元素必须标记 `data-testid`，格式：`<功能名>-<元素名>` |
| E0-3 | 断言必须来自动态检查清单的预期结果 |
| E0-4 | API mock 通过 hooks/store 层隔离，不依赖真实后端 |
| E0-5 | 测试文件路径：`tests/e2e/<功能名>/` |
| E0-6 | Mock 仅限 `tests/` 目录，生产代码禁止 mock 逻辑 |
| E0-7 | 写代码前必须在真实入口完成主场景 MVP 最小可用测试并留存证据 |
| E0-8 | 写代码后必须由 AI 自动执行主流程全链路冒烟测试并通过 |

---

## 2. 验证方式

推荐验证方式：

| 方式 | 用途 |
|------|------|
| 写代码前真实场景 MVP 验证（强制） | 启动真实运行环境 → 从主流程入口执行最小可用路径 → 记录通过证据 |
| 手动浏览器验证 | 启动 dev server → 按检查清单操作 → 截图记录 |
| 代码审查 | 确认 data-testid 完整、createBaseView 调用正确、CDN 组件注册正确 |
| 构建验证 | 确认页面无 JS 控制台错误、组件正常渲染 |
| AI 自动主流程冒烟（强制） | 由 AI 自动串联主流程步骤执行端到端冒烟，输出通过/失败与证据路径 |

若已安装 Playwright，应优先用于 AI 自动主流程冒烟；未安装时，需通过可脚本化命令 + 可复核日志完成等价自动化执行，不得以“手工点点看”替代。

### 2.1 与 [`implement-code-testing.md`](./implement-code-testing.md) 的对照

| 门禁 | 本文件承载的细节 | `implement-code-testing` 定义的底线 |
|------|------------------|-------------------------------------|
| Gate A（编码前） | checklist 路径、`tests/e2e/` 布局、原型页与 `data-testid` | 真实入口、主路径 MVP、可追溯证据；无证据不得编码 |
| Gate B（编码后） | Playwright / 静态检查优先顺序 | AI 必须真实执行；失败阻断交付 |

---

## 3. 文件结构

```text
tests/
└── e2e/<功能名>/
    ├── <场景名>-checklist.md    # 验证清单（操作步骤 + 预期结果 + 截图）
    └── fixtures/                 # mock 数据（可选）
```

---

## 4. 禁止事项

- 在 E2E 测试中 import 项目源码
- 只测试成功路径不覆盖失败分支
- Mock 数据使用无意义占位符
- 在 `tests/` 外生成测试文件
- 假设 Playwright 可用（当前未安装）
- 未完成真实场景 MVP 测试就进入编码
- 未执行 AI 自动主流程冒烟就宣告完成
