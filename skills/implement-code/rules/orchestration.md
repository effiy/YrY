# 编排与阶段契约规范

> 本规范约束 `implement-code` 的阶段状态机、输入前提、skill/agent 分派、MCP 探针与降级策略。  
> **权威原则**：阶段编号、阶段名称、进入条件、退出条件以本文件和 `../SKILL.md` 为准，其他规则文件不得重新定义阶段映射。

---

## 1. 阶段状态机（权威定义）

| 阶段 | 名称 | 目标 | 解锁条件 |
|------|------|------|---------|
| 阶段 0 | 文档驱动 | 解析功能名，预检文档，完成 Grounding 与探针检查 | P0 文档齐全，MCP 可用性已声明 |
| 阶段 1 | 测试先行 | 生成测试原型页与 Playwright 骨架 | 场景锚点完整，`data-testid` 覆盖齐全 |
| 阶段 2 | 动态检查门禁 | 在原型页面上验证全部 P0 项 | P0 全部通过并回写 `05_动态检查清单.md` |
| 阶段 3 | 模块预检 | 完成环境预检与全项目影响链闭合分析 | 预检清单全部通过，上游依赖、反向依赖、传递依赖和外部依赖已闭环 |
| 阶段 4 | 写项目代码 | 逐模块实施并逐模块验证 | 模块实现完成，Lint / 回归 / 影响链回归记录齐全，真实 diff 已复核 |
| 阶段 5 | 代码评审 | 执行 code-review 与 Mock 泄漏检查 | 无 P0 评审问题 |
| 阶段 6 | 冒烟测试 | 在真实页面上验收全部 P0 项并回写状态 | P0 全部通过并回写 `05_动态检查清单.md` |
| 阶段 7 | 过程总结 | 生成 `06_实施总结.md` 并回写功能文档状态 | 总结写入完成，`01/02/03/04/05/07` 状态回写完成并记录，**动态检查清单最终完成门禁通过**（按 `artifact-contracts.md §3.4` 与 `verification-gate.md §9.8`） |
| 阶段 8 | 文档同步与通知 | 同步 `docs` 并发送完成 / 阻断 / 门禁异常通知 | 阶段 7 三项退出条件全部满足，`import-docs` 已执行或记录跳过原因，wework-bot 已按本次结束类型（成功 / 阻断 / 门禁异常）模板发送或记录发送失败原因 |

---

## 2. 输入前提

### 2.1 默认输入

- `{功能名}`：对应 `docs/<功能名>/`
- `{文档集路径}`：默认 `docs/<功能名>/`

若无法解析 `{功能名}` 或定位 `{文档集路径}`，必须先写入 `docs/99_agent-runs/<YYYYMMDD-HHMMSS>_implement-code.md`，记录原始请求、路径解析失败原因和建议恢复步骤，再向用户请求补充信息。

## 2.4 一次执行到底（默认策略）

`implement-code` 默认应“一次执行到底”，减少人工干预：

- 除非命中本文件「阻断点」或门禁要求，否则不得频繁向用户追问确认；缺失信息写“待确认”并继续推进（由阶段规则约束证据链与降级记录）。
- 任何需要人工介入的情况（阻断 / 门禁异常 / 证据缺失 / 无降级方案）都必须先落地兜底记录（`06_实施总结.md` 或 `docs/99_agent-runs/`），再**必须调用 `wework-bot`** 推送消息，包含原因、证据与恢复点。

### 2.2 最低文档要求

| 文件 | 级别 | 用途 |
|------|------|------|
| `01_需求文档.md` | P1 | 背景与目标补充 |
| `02_需求任务.md` | P0 | 提取用户故事、场景与前置条件 |
| `03_设计文档.md` | P0 | 提取模块、接口、文件路径和实现约束 |
| `04_使用文档.md` | P2 | 辅助 UI 文案和用户提示 |
| `05_动态检查清单.md` | P0 | 提取全部待验证检查项 |

### 2.3 缺失处理

任一 P0 文档缺失时：

1. 记录缺失文档名称和级别。
2. 停止进入阶段 1。
3. 按 `./process-summary.md` 先生成阻断总结。
4. 提示用户先运行 `generate-document` 补齐文档。

## 3. 阶段 0 细则

### 3.1 文档 Grounding 输出

必须至少提取以下结构化信息：

```text
用户故事：
  US-{N}
    场景名
    前置条件
    操作步骤
    预期结果

实现约束：
  模块列表（名称 + 文件路径）
  接口规范（输入 / 输出 / 错误）
  状态管理约束
  已有代码路径
  全项目影响链（搜索词与改动点清单 / 上游依赖 / 反向依赖 / 传递依赖 / 依赖闭合摘要 / 未覆盖风险）

检查项映射：
  场景 -> P0 / P1 / P2 检查项列表
```

### 3.2 Skill / Agent 分派

默认分派：

| 类型 | 名称 | 用途 |
|------|------|------|
| Skill | `find-skills` | 声明和发现要用到的 skill |
| Skill | `find-agents` | 发现和分派代理 |
| Skill | `e2e-testing` | 阶段 1 产出测试骨架 |
| Skill | `verification-loop` | 阶段 3 / 5 的静态预检与测试执行 |
| Skill | `code-review` | 阶段 5 真实代码审查 |
| Skill | `search-first` | 外部依赖选型（按需） |
| Agent | `test-page-builder` | 阶段 1 生成测试原型页 |
| Agent | `architect` | 架构不确定时咨询 |
| Agent | `security-reviewer` | 安全相关场景 |
| Agent | `docs-lookup` | 文档歧义定位 |
| Agent | `impl-reporter` | 阶段 7 生成实施总结 |

### 3.3 MCP 探针与降级

必须在进入阶段 1 之前完成：

| MCP 工具 | 最小探针 | 降级方案 | 是否可阻断 |
|---------|---------|---------|-----------|
| `playwright` | 打开 `about:blank` | `npx playwright test` + 人工确认 | 否 |
| 文件系统能力 | 复用文档读取结果 | 改用本地读写工具 | 否 |
| 无降级方案的必要工具 | 最小可行探针 | 无 | 是 |

禁止静默降级。所有降级都必须记录到 `06_实施总结.md`。

若探针、降级记录或门禁证据缺失，按"门禁失效"处理：停止进入下一阶段，写入实施总结或兜底运行记录，并调用 `wework-bot` 发送门禁异常通知。

### 3.4 场景—检查项覆盖预检（提升 `05` 与 `02` 一致性）

> 目的：在写测试与过门禁前，**显式**核对「需求任务中的场景、操作」是否都有 `05_动态检查清单.md` 中可执行的 P0 检查项，降低遗漏率与无依据跳步。  
> 依据：`../../shared/evidence-and-uncertainty.md`（不虚构已覆盖关系）。

1. 从 `02_需求任务.md` 列出全部用户故事/场景行（可抄表或编号 US-x.S-y）。
2. 从 `05_动态检查清单.md` 列出全部 **P0** 检查项行号或表内唯一标识。
3. 建表（可写在实施笔记或本阶段对话顶部，**最终**在 `06_实施总结.md` 的 `§3` 或 `§5` 用一句话概括结论）：

| 场景/故事（摘自 `02`） | 覆盖的 `05` P0 项（编号或摘要） | 缺口说明 |
|------------------------|--------------------------------|----------|
| ... | ... | 无 / 无对应用例—待补 `generate-document` 或本阶段补写 `05` |

4. **进入阶段 1 的最低要求**：上表可存在「待补」**仅当**在 `05` 或 `docs/99_agent-runs/` 记录原因与补全方式；**否则**应先用 `generate-document` 或人工补链再实施。
5. **进入阶段 2 的最低要求**：所有 **P0 场景**在表中均有至少一项 `05` 映射，或己在 `05` 中显式将该项标为 N/A 并写原因（禁止静默跳过）。

## 4. 阶段进入 / 退出条件

### 阶段 1

- 进入：阶段 0 完成
- 退出：原型页面、测试骨架、场景锚点、`data-testid` 覆盖率齐全；**全部测试产物路径位于仓库 `tests/` 下**（见 `e2e-testing.md`、`artifact-contracts.md §2`），否则不得进入阶段 2

### 阶段 2

- 进入：阶段 1 完成
- 退出：全部 P0 通过，且状态已回写 `05_动态检查清单.md`

### 阶段 3

- 进入：阶段 2 完成
- 退出：环境预检、按 `../../../shared/impact-analysis-contract.md` 完成的全项目影响链闭合分析、编码预检全部通过

### 阶段 4

- 进入：阶段 3 完成
- 退出：模块实现、ReadLints、逐模块验证、基于真实 diff 的全项目影响链回归搜索完成

### 阶段 5

- 进入：阶段 4 完成
- 退出：无 P0 评审问题，Mock 泄漏检查通过

### 阶段 6

- 进入：阶段 5 完成
- 退出：真实页面全部 P0 通过并回写状态

### 阶段 7

- 进入：阶段 6 完成，或任意前置阶段阻断
- 退出（按顺序全部满足才放行；阻断版总结只需满足 1 / 2 / 3）：
  1. `06_实施总结.md` 写入完成
  2. 按 `artifact-contracts.md §4 功能文档状态回写契约` 更新同功能目录内 `01/02/03/04/05/07` 文档状态并记录结果
  3. **动态检查清单最终完成门禁通过**（按 `artifact-contracts.md §3.4` 与 `verification-gate.md §9.8` 执行"先回写最终状态 → 再复查 → 写入总结"序列；不通过则改走阻断版总结并切到门禁异常通知）

### 阶段 8

- 进入：阶段 7 完成，或任意停止点完成兜底运行记录
- 执行：先运行 `import-docs` 标准导入并记录真实结果，再调用 `wework-bot` 发送完成、阻断或门禁异常通知
- 退出：
  1. 通知已发送（模板与本次结束类型一致：成功 / 阻断 / 门禁异常）
  2. 发送失败时已把 API 状态 / 错误摘要、脱敏机器人路由、模型、工具、最后更新时间（精确到秒）写回 `06_实施总结.md` 或 `docs/99_agent-runs/`
  3. 若动态检查清单最终完成门禁未通过却发出了"完成（成功）"通知，必须立即追发"门禁异常"通知并在总结中标注修正记录

## 5. 阻断点

以下情况必须停止当前推进：

- `{功能名}` 或文档集路径无法定位
- P0 文档缺失
- 无降级方案的必要 MCP 不可用
- 阶段 2 达到修复上限仍未通过
- 阶段 6 达到修复上限仍未通过
- 任一阶段门禁应执行但未执行、缺少证据、被跳过或降级未记录
- 所有模块均阻断且无法继续

停止时必须：

1. 输出阻断原因。
2. 记录当前阶段和已产生产物。
3. 功能目录可定位时，先生成 `06_实施总结.md`。
4. 功能目录不可定位时，先生成 `docs/99_agent-runs/<YYYYMMDD-HHMMSS>_implement-code.md`。
5. 功能目录可定位时，回写同功能目录内文档状态；若回写失败，记录失败原因。
6. 执行 `import-docs` 标准导入；目录不存在、导入失败或部分失败都必须记录真实数字和错误摘要。
7. 调用 `wework-bot` 发送阻断或门禁异常通知，正文必须包含阻断阶段、证据路径、P0 统计、模型、工具、最后更新时间（精确到秒）。
8. 通知发送失败时，把失败原因写入 `06_实施总结.md` 或 `docs/99_agent-runs/`，再结束流程。
