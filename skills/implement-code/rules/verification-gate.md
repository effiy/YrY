# 验证门禁规范

> CDN SPA（静态托管、无构建链路）验证决策逻辑。通过手动浏览器验证 + createBaseView 集成检查。

---

## 1. 门禁原则

| 编号 | 原则 |
|------|------|
| G0-1 | P0 检查项未全部通过，禁止交付 |
| G0-2 | 验证失败时必须修复代码，不得修改检查清单降级 P0 |
| G0-3 | 自修复上限 2 轮 |
| G0-4 | 人工确认类检查项不计入 P0 通过/失败判定 |
| G0-5 | 门禁未执行、被跳过或缺少报告证据，视为门禁失效 |

---

## 2. 检查项分类

| 优先级 | 验证方式 | 通过条件 |
|--------|---------|---------|
| P0 | 手动浏览器验证 + 代码审查 | UI 可交互、data-testid 完整、createBaseView 调用正确 |
| P1 | 手动验证 + Jest | 错误状态展示、边界值、组件渲染 |
| P2 | 人工 | 视觉一致性、响应式布局 |

---

## 3. 验证执行方式

```
1. 启动 dev server（或 HTTP 服务器）
2. 在浏览器中按检查清单逐步操作
3. 截图记录操作结果
4. 检查 JS 控制台无错误
5. 记录：✅ 通过 / ❌ 失败
```

---

## 4. 修复循环

```
轮次 1：分析根因 → 修复代码 → 重新验证
轮次 2：同上
轮次 3：输出阻断报告，停止执行
```

---

## 5. 状态回写

验证完成后将结果回写到 `05_动态检查清单.md`：
- P0 通过 → ✅ + 验证日期
- P0 未通过 → ❌ + 失败原因

---

## 6. 禁止事项

- 将 P0 降级为 P1 以"通过"门禁
- 修改检查清单预期结果
- 修复超过 2 轮后仍继续
- 未实际验证就改状态为 ✅
