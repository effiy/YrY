# implement-code 测试门禁（真源）

> **适用范围**：`implement-code` 全流程中与「编码前验证」「编码后验证」相关的强制步骤。  
> **UI 细节**：交互方式、`data-testid`、目录骨架仍以 [`e2e-testing.md`](./e2e-testing.md)、[`test-page.md`](./test-page.md) 为准；本文件定义 **何时做、做到什么算通过、证据是什么**。

---

## 1. 两道门禁（必须）

| 门禁 | 对应编排阶段 | 含义 |
|------|----------------|------|
| **Gate A：编码前真实场景 MVP** | 阶段 1「测试先行」 | 在**真实入口**上对「主路径」做一次**最小可用**验证，证明方案在当前仓库环境下可走通；**留可追溯证据**后再允许进入动态清单门禁与写项目代码。 |
| **Gate B：编码后主流程冒烟** | 阶段 6「冒烟测试」 | 代码已在仓库落地后，由 **AI 自动执行**（或可脚本化、可复核的命令链路）跑通 **P0 主流程**：起点 → 关键步骤 → 预期结果 → 必要时状态回写；**失败即阻断**，不得进入阶段 7。 |

**缺一不可**：未满足 Gate A 不得写项目代码（见 [`code-implementation.md`](./code-implementation.md) C0-1）；未满足 Gate B 不得宣告实施完成。

---

## 2. Gate A：编码前——何为「真实场景」「最小可用」

### 2.1 真实场景

- **必须**贴合 `02_需求任务.md` 中与本次改动相关的 **主用户故事 / P0 场景**，以及 `05_动态检查清单.md` 中对应 **P0 验收表述**。
- **入口真实**：使用本项目约定的真实启动方式（例如本地静态服务打开的页面、`chrome://extensions` 加载扩展后的页面、或文档中指明的 CLI / API 调用路径），**禁止**仅用口头推演或未落地的伪页面冒充。

### 2.2 最小可用（MVP）

- 只覆盖 **一条主路径**（最短闭环）：能回答「在当前环境下，核心行为是否可触发、核心结果是否可见或可观测」。
- 不要求覆盖全部分支或边界；**但必须**与交付边界一致：**若无 MVP 证据，后续编码视为无效投入**。

### 2.3 证据（缺一不可）

下列任一组合可作为「可追溯证据」，须写入实施笔记、`06_实施总结.md` 或 `tests/` 下可追溯路径：

| 证据类型 | 示例 |
|----------|------|
| 命令与退出码 | 终端完整命令、`exit code 0`、关键 stderr 为空 |
| 日志摘录 | 可复核的文件路径 + 行号或截取段落 |
| 截图 / 录屏 | 存放于 `tests/screenshots/<功能名>/` 或同契约路径 |
| 清单勾选 | `tests/e2e/<功能名>/…-checklist.md` 中步骤与预期结果已勾选并与 MVP 一致 |

**禁止**：仅凭「曾经试过」「应该可以」作为通过依据。

### 2.4 与 TDD / 自动化测试的关系

- **已有自动化测试基建**（如 `node:test`、Playwright、项目脚本）：在 Gate A 内优先体现为 **先写失败用例或最小复现脚本**，再进入实现；与仓库 [`behavioral-guidelines`](../../../shared/behavioral-guidelines.md) 一致。
- **零构建 / 无 runner**：Gate A 仍以 **真实入口 + MVP 证据链** 为准；可将最小复现步骤固化到 `tests/` 下的 checklist 或一键脚本，避免口头步骤漂移。

### 2.5 按形态的快照指引（非穷尽）

| 形态 | Gate A 典型做法 |
|------|------------------|
| 静态网页/前端页面 | 本地起服务打开入口页 → 主按钮/主流程一步可走通 → checklist + 可选截图 |
| 浏览器扩展 | 加载未打包扩展 → content/popup 一侧可见路径走通 → 证据路径写明 |
| Node / CLI | `node script.js` 或约定命令对主参数跑通 → 日志落 `tests/` 或实施记录 |

## 3. Gate B：编码后——何为「主流程冒烟」

### 3.1 范围

- **必须**覆盖：`05_动态检查清单.md` 中 **全部 P0** 项里与「主路径」相关的条目；若某 P0 非主路径，须在实施记录中标注 **N/A 与原因**，不得静默跳过。
- **执行者**：由当前会话中的 **AI 实际运行**命令或自动化（Playwright 优先；未安装则等价可脚本化链路），不得以「请你本地点点」替代自动化结论。

### 3.2 通过条件

- 主路径 **端到端** 一次跑通；必要时允许 **≤2 轮**修复循环（见 [`verification-gate.md`](./verification-gate.md)）。
- 产出 **可复核产物**：终端输出、HTML 报告、`tests/traces/` 等路径写入实施总结或 `05` 备注。

### 3.3 失败处理

- **阻断**进入阶段 7；按 `SKILL.md` 触发 **门禁异常**类 `wework-bot`（若流程要求），并在 `06_实施总结.md` 记录失败步骤与日志路径。

## 4. 与 `e2e-testing` 技能的分工

| 角色 | 职责 |
|------|------|
| [`../../e2e-testing/SKILL.md`](../../e2e-testing/SKILL.md) | 输出场景级策略、用例骨架、选择器与 mock 建议 |
| **本文件** | 将上述产出 **钉死在** Gate A / Gate B 的准入与证据标准上 |

调用 `e2e-testing` 时，输出物必须能直接服务于 Gate A（清单、路径）与 Gate B（可执行步骤或可自动化 spec）。

---

## 5. 禁止事项

- 未完成 Gate A 即修改项目源码（测试骨架与原型页除外，且须在 `tests/` 约定路径内）。
- Gate B 仅以「代码审查代替执行」或编造执行结果。
- 将测试产物放在 `tests/` 以外却宣称满足门禁（见 [`artifact-contracts.md`](./artifact-contracts.md) §2）。
