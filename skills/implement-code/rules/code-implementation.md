# 代码实施规范

> 本规范约束 implement-code 技能在**阶段 2（代码实施）**的全部编码行为。
> 项目特有架构约束、编码规范和实施顺序见 `../../../docs/architecture.md`，本规范只承载通用流程约束。

---

## 1. 核心约束（P0）

| 编号 | 约束 |
|------|------|
| C0-1 | **不得在阶段 3 验证门禁通过之前交付代码**。 |
| C0-2 | 实施的每一行代码必须可追溯到设计文档的某个模块、接口规范或影响链闭合记录。 |
| C0-3 | 不得新增设计文档中未提及的文件或目录（新增前必须标注原因）。 |
| C0-4 | 实施后 **必须** 检查语法，P0 语法错误不得遗留。 |
| C0-5 | 实施后 **必须** 补充 `data-testid` 到真实 UI 组件（与测试页面中的 testid 完全一致）。 |
| C0-6 | 实施后 **必须** 通过阶段 3 验证门禁，才能进入阶段 4。 |
| C0-7 | 删除、重命名或修改公共接口前，必须完成全项目影响链闭合分析。 |
| C0-8 | 共享组件和模块必须放在项目约定目录下，不得违反项目架构约定。 |

---

## 2. 项目架构约定

项目特有架构模式、实施顺序、编码规范和目录结构见 `../../../docs/architecture.md`（实施前必须读取）。

---

## 3. data-testid 移植规则

阶段 2 测试页面中定义的所有 `data-testid` **必须** 原样出现在真实组件中，不得更名或缺省。

---

## 4. 静态预检清单（实施前必须全部通过）

- [ ] 设计文档中所有模块的文件路径已确认存在（或为新建）
- [ ] 项目架构约定已读取并确认遵循
- [ ] 已读取 `../../../shared/document-contracts.md`
- [ ] 每个拟改动点已完成全项目影响链闭合分析

---

## 5. 每模块完成后自检

每完成一个模块的实施，必须：

1. 检查语法，消除 P0 语法错误
2. 确认 `data-testid` 完整性
3. 确认项目架构约定满足（对照项目约定文件）
4. **全项目范围回归验证**：按 `../../../shared/document-contracts.md` 对真实 diff 重建搜索词，搜索影响链

---

## 6. 禁止事项

- ❌ 违反项目架构约定放置共享/应用特有代码
- ❌ 在未读取现有相关代码前直接写新代码
- ❌ 跳过 data-testid 移植
- ❌ 在 P0 语法错误未消除时进入下一阶段
