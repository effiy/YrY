# 全项目影响分析契约

本文档定义 `generate-document` 与 `implement-code` 共用的影响分析口径。凡涉及需求拆解、设计生成、代码实施、修复回归或交付总结，都必须以本契约为准。

## 适用阶段

| 阶段 | 使用方 | 目标 | 必须产物 |
|------|--------|------|----------|
| 规划期 | `generate-document` 生成需求任务 / 设计文档 | 从上游文档和现有代码识别可能受影响的模块、依赖关系和验证范围 | 写入需求任务第 6 章或设计文档第 5 章 |
| 实施前 | `implement-code` 阶段 3 | 把设计文档中的拟改动点复核为可实施清单，确认无遗漏依赖 | 写入实施记录或 `06_实施总结.md` 的阶段 3 记录 |
| 实施后 | `implement-code` 阶段 4 / 6 | 对真实 diff 做回归搜索，确认同步修改项已处理、间接影响已验证 | 写入逐模块记录、动态检查清单回写依据和实施总结 |

## 核心原则

1. **全项目范围**：默认搜索整个仓库，而不是只搜索 `src/`、当前模块目录或当前文档目录。
2. **依赖图闭合**：每个拟改动点都必须追踪“它依赖谁、谁依赖它、调用方是否继续导出 / 注册 / 转发 / 封装”，直到影响链闭合或明确停止原因。
3. **反向依赖优先**：删除、重命名、修改公共接口、修改 payload 或改变默认行为前，必须先证明所有调用方、消费方、监听方和测试方已覆盖。
4. **传递影响必查**：发现直接命中点后，继续检查该命中点是否作为二级 API、barrel export、全局注册、路由入口、Store 聚合或文档示例被再次依赖。
5. **事实来自工具**：引用关系必须来自实际搜索、文件读取、结构化工具或测试结果，不得凭记忆填写。
6. **未命中也要记录**：每个搜索词都必须记录结果；没有命中时写“未找到引用”，并说明这代表“可能是新能力”还是“可能遗漏动态引用”。
7. **规划与实施互相校验**：`generate-document` 给出影响预判，`implement-code` 必须在真实代码变更前后复核，不得把规划期结论直接当作实施期事实。

## 搜索范围

默认包含：

- 项目源码：`src/`、页面、组件、Store、composables、services、utils、styles。
- 测试资产：`tests/`、fixture、mock、测试原型页、截图 / 下载目录约定。
- 文档资产：`docs/`、`README.md`、使用文档、动态检查清单、项目报告、实施总结。
- Agent / Skill 规则：`.claude/` 下的 skills、agents、shared 契约、命令脚本。
- 配置与构建：`package.json`、构建配置、路由配置、环境变量示例、CI / 脚本入口、别名配置。
- 项目根目录下与运行、构建、发布相关的 Markdown / JSON / JS / TS / Vue / CSS 文件。

默认排除：

- `node_modules/`
- `dist/`
- `build/`
- `.git/`
- lock 文件
- 二进制资源
- 与当前分析无关的自动生成快照；若快照可能作为验收依据，必须在未覆盖风险中说明是否排除。

## 必查维度

| 维度 | 必查内容 | 闭合标准 |
|------|----------|----------|
| 直接引用 | import、模板标签、函数调用、字符串引用、路由跳转、CSS 类名和变量 | 每个搜索词都有命中记录或“未找到引用”记录 |
| 上游依赖 | 当前改动点依赖的组件、Store、工具函数、常量、配置、第三方包 | 上游变更风险已标注为同步修改 / 保持兼容 / 仅验证 |
| 反向依赖 | 当前模块 / 函数 / 组件 / Store / 常量的所有调用方、消费方、监听方 | 所有调用方处置方式明确，无“待判断” |
| 传递依赖 | 调用方继续被谁调用、导出后被谁复用、注册后被谁消费 | 二级及以上影响已闭合，或写明停止原因 |
| 导出链 | index.js、barrel export、default export、named export、re-export | 导出入口和所有 import 方一致 |
| 注册链 | app.use、app.component、provide/inject、router、store index、插件注册 | 注册点、消费点、注册顺序影响已核对 |
| 数据流 | props、emit、事件总线、Store state/action/getter、service 返回结构 | payload / 返回字段与所有消费方一致 |
| 类型与契约 | TS 类型、JSDoc、接口字段、枚举、常量、API payload、错误码 | 类型、文档示例、mock / fixture 与实现一致 |
| 样式影响 | 全局 class、CSS 变量、mixin、`@apply`、动态 class 字符串 | 模板、`:class`、classList、文档示例均已覆盖 |
| 测试与文档 | 单元测试、E2E、fixture、mock、使用文档、动态检查清单、项目报告、实施总结 | 需新增 / 更新 / 回归的测试与文档已列入处置 |
| 配置与构建 | package scripts、vite/webpack 配置、别名、环境变量、CI 脚本 | 本地运行、构建、CI 和文档说明无断链 |
| 外部依赖 | npm 包、远端脚本/静态资源、浏览器 API、MCP / 脚本工具、运行时权限 | 新增或升级依赖有来源、用途、风险和降级策略 |

## 分析步骤

1. 从需求、设计、现有代码或变更 diff 中列出所有拟改动点：文件、导出、函数、组件、Store、路由、事件、CSS、配置、依赖包。
2. 为每个改动点建立搜索词集合：名称、别名、路径、标签名、事件名、路由名、字符串 key、CSS token、环境变量名、脚本名、包名。
3. 先搜索改动点本身，再搜索导出入口、注册入口和公共聚合入口，避免只命中实现文件。
4. 在全项目范围搜索每个搜索词，记录命中文件、行号、引用方式、影响层级和证据来源。
5. 对每个直接命中点判断是否存在二级影响：该命中点是否继续导出、注册、转发、封装、被测试依赖或被文档示例依赖。
6. 对存在二级影响的命中点继续搜索，直到影响链闭合；停止追踪时必须写明原因，例如动态字符串无法静态确认、第三方运行时约定、已到文档示例边界。
7. 标注处置方式：同步修改、保持兼容、补充验证、人工复核、无需处理。
8. 实施后以真实 diff 重跑第 2-7 步，新增、删除、重命名、移动文件都必须重新建立搜索词。

## 输出格式

影响分析必须至少包含四个部分。需求任务和设计文档可将其合并排版，但字段不得缺失。

### 1. 搜索词与改动点清单

| 改动点 | 类型 | 搜索词 | 来源 | 备注 |
|--------|------|--------|------|------|
| `<模块/函数/组件>` | component / store / route / event / css / config / dependency | `<关键词1>, <关键词2>` | 需求文档 / 需求任务 / 设计文档 / diff / 代码路径 | `<为什么纳入分析>` |

### 2. 改动点影响链

| 改动点 | 搜索词 | 命中文件 | 引用方式 | 影响层级 | 依赖方向 | 处置方式 | 闭合状态 | 说明 |
|--------|--------|----------|----------|----------|----------|----------|----------|------|
| `<模块/函数/组件>` | `<关键词>` | `<path>:<line>` | import / call / route / css / test / docs | 直接 / 二级 / 传递 | 上游依赖 / 反向依赖 / 传递依赖 | 同步修改 / 保持兼容 / 补充验证 / 人工复核 / 无需处理 | 已闭合 / 阻断 / 待人工确认 | `<原因>` |

### 4. 未覆盖风险

| 风险来源 | 原因 | 影响 | 缓解方式 |

## P0 门禁

- 未完成全项目搜索，不得生成设计结论或开始实施代码。
- 未输出搜索词与改动点清单，不得声称影响分析完整。
- 未分析上游依赖、反向依赖和传递依赖，不得删除、重命名或修改公共接口。
- 未分析导出链、注册链、测试、文档和配置影响，不得修改 Store、事件、路由、全局组件、构建配置或外部依赖。
- 发现“同步修改”或“阻断”项但未处理时，必须写入阻断原因或待处理清单。
- 实施后未用真实 diff 回归搜索，不得进入最终总结或宣称影响链闭合。
