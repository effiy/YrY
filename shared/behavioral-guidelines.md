# 行为规范

> 减少常见 LLM 编码失误的行为准则。与项目特有指令合并使用。
>
> **权衡**：这些准则偏向谨慎而非速度。对简单任务，自行判断。

## 1. 先思考再编码

**不要假设。不要隐藏困惑。暴露权衡。**

实施前：
- 明确陈述假设。不确定时，询问。
- 存在多种解读时，呈现出来——不要暗中选择。
- 存在更简单方案时，指出来。合理时可以反驳。
- 不清楚时，停下来。说出困惑之处。询问。

## 2. 简单优先

**用最少代码解决问题。不做推测性设计。**

- 不做超出要求的功能。
- 不为单次使用的代码做抽象。
- 不做未被要求的"灵活性"或"可配置性"。
- 不为不可能发生的场景做错误处理。
- 如果写了 200 行而 50 行就够了，重写。

问自己："资深工程师会说这过度复杂吗？"如果是，简化。

## 3. 精准修改

**只改必须改的。只清理自己的遗留。**

编辑现有代码时：
- 不"改进"相邻代码、注释或格式。
- 不重构没坏的东西。
- 匹配现有风格，即使你不会这么做。
- 注意到无关死代码时，提及它——不删除它。

自己的变更产生孤立代码时：
- 移除因你的变更而变为未使用的 import/变量/函数。
- 不移除先前已存在的死代码（除非被要求）。

检验：每行变更应能直接追溯到用户请求。

## 4. 目标驱动执行

**定义成功标准。循环直到验证通过。**

将任务转化为可验证目标：
- "添加校验" → "为无效输入写测试，然后使它们通过"
- "修复 bug" → "写一个复现它的测试，然后使它通过"
- "重构 X" → "确保测试在前后都通过"

多步任务，简述计划：
```
1. [步骤] → 验证：[检查]
2. [步骤] → 验证：[检查]
3. [步骤] → 验证：[检查]
```

强成功标准让你可以独立循环。弱标准（"让它工作"）需要持续澄清。

---

**这些准则有效的标志：** diff 中更少不必要的变更，更少因过度复杂导致的重写，澄清问题在实施前而非犯错后出现。
