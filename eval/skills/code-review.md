# code-review 评测示例

真源：`.claude/skills/code-review/SKILL.md`。

**评测约束**：审查结果必须基于实际读取到的代码，不得推断未看到的文件内容；输出须按 P0/P1/P2 分级，并给出具体文件路径与修复建议。

---

## 用户故事 A：单文件规范审查

**故事**：作为开发者，希望对某个组件进行规范性审查，确保符合项目约定。

**示例输入（对话）**

- 「审查 `src/components/Button.vue`，重点看状态管理和组件导出是否符合项目约定。」
- 「review `src/stores/auth.ts`，关注安全风险和边界处理。」

---

## 用户故事 B：多文件批量审查

**故事**：作为代码审查者，希望一次性审查多个相关文件，输出统一的分级问题列表。

**示例输入（对话）**

- 「帮我 review 这几个文件：`src/utils/api.ts`、`src/hooks/useAuth.ts`、`src/views/Login.vue`。」
- 「按项目规范检查 `src/composables/` 下的所有文件。」

---

## 成功判据（可观测）

| 维度 | 预期 |
|------|------|
| 真实性 | 只审查实际读取到的代码，未读取文件明确说明"无法读取文件 <路径>，跳过" |
| 分级 | P0（必须修复）、P1（建议修复）、P2（可选优化）清晰区分 |
| 定位 | 每项问题须给出文件路径与行号（或代码片段位置） |
| 建议 | P0 项须给出修复建议；P1/P2 至少给出问题描述 |
| 覆盖 | 审查维度覆盖项目专项（入口初始化、状态管理、组件注册/导出、代码结构、编码规范）和通用质量（可读性、边界处理、安全、性能） |

---

## 负例（应判不达标）

- 审查未读取的文件并给出具体行号问题（幻觉）。
- 输出未分级或仅有笼统描述，无文件路径和修复建议。
- 以"看起来没问题"概括全部维度，未逐一说明无问题项。
