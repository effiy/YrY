# verification-loop 评测示例

真源：`.claude/skills/verification-loop/SKILL.md`。

**评测约束**：执行前必须进行 MCP 可用性探针，禁止静默降级；阶段 1-3 有阻断项时禁止进入阶段 4；阶段 4 失败后不自动重试，须给出一次性修复清单。

---

## 用户故事 A：构建验证

**故事**：作为开发者，希望在提交前验证构建是否通过。

**示例输入（对话）**

- 「验证一下 `npm run build` 能不能通过。」
- 「跑一下类型检查和 lint，看看有没有阻断项。」

---

## 用户故事 B：UI/E2E 验证（含 MCP 探针）

**故事**：作为开发者，希望在验证流程中自动检测 Playwright 可用性，并据此决定执行方式。

**示例输入（对话）**

- 「验证 E2E 测试能不能跑通，先检查 Playwright MCP 是否可用。」
- 「跑一下 playwright 测试，如果 MCP 不可用就降级为 npx playwright test。」

---

## 成功判据（可观测）

| 维度 | 预期 |
|------|------|
| 探针 | 执行前进行 MCP 可用性探针，状态明确标注 ✅/⚠️/❌ |
| 阻断控制 | 阶段 1-3 发现阻断项时停止，不进入阶段 4 |
| 命令来源 | 验证命令名称取自 `package.json scripts`，无硬编码假设 |
| 单次执行 | 阶段 4 只执行一次，失败后不重试 |
| 修复清单 | 失败后给出按优先级排序的"一次性修复清单"，并说明修复后应从哪个阶段重入 |
| 降级标注 | playwright 降级时输出中须含 ⚠️ 标注，不得以全通过呈现 |

---

## 负例（应判不达标）

- 静默跳过 MCP 探针或必要工具不可用时未提示即降级。
- 阻断项未解决即执行构建命令。
- 失败后自动重试而不给出根因定位和修复清单。
- 降级执行后未标注 ⚠️，呈现为全部通过。
