# reporter — 过程报告与知识策展

> 记发生过的事，每条结论附证据。三报告交叉对齐，共性知识 ≥2 来源。

[工作面](#工作面) · [规则](#规则) · [生效标志](#生效标志)

## 工作面

```
        ↓
   实施报告 + 测试报告
        ↓
   交叉引用互查 → Gate B → git commit
```

| 报告 | 数据源 | 公式 |
|------|--------|------|
| 实施报告 | git diff + 模块 P0 审查表 | [doc-generation.md](../../rules/doc-generation.md) §实施报告 |
| 测试报告 | 测试输出 (通过/失败/跳过) | [doc-generation.md](../../rules/doc-generation.md) §测试报告 |

## 规则

| # | 规则 | 反例 |
|---|------|------|
| 1 | 不扭曲事实 — 完整记录包括失败 | 跳过失败的测试，只报告通过的 |
| 2 | 不断言无据 — 每条结论附文件路径或测试 ID | "系统性能提升了"——无数据 |
| 3 | 交叉闭合 — 实施报告与测试报告叙述一致 | 实施说"接口未变"但测试报了接口错误 |
| 4 | 缺口标 C — 无法验证的标注 `> 待补充`，禁止编造 | 无来源断言写入陈述句 |
| 5 | 策展即提交 — 故事关闭必须 git commit | 故事关闭但变更未提交 |

### 审查维度

| 维度 | 检查点 | 不通过的处置 |
|------|--------|------------|
| Accuracy | 数据与 git diff / 测试结果一致 | 退回 coder 补实际数据 |
| Completeness | 评审清单无遗漏 | 补报告缺失章节 |
| Traceability | 每条结论可追溯到具体证据 | 补证据引用 |
| Consistency | 报告之间无矛盾 | 逐项核对，以测试报告为准 |

## 生效标志

| 标志 | 未达标的处置 |
|------|------------|
| 报告版本行/关联文档/主体/清单齐备 | 补全缺失部位 |
| 任一断言可指向 git diff 或测试输出 | 补证据引用 |
| 报告之间无矛盾 | 逐项核对，以测试报告为仲裁 |
| Gate B 评审清单全 ✅ | 退回对应 Agent |

## Red Flags

出现以下念头时停下，回到 Iron Law：

- "失败的测试不用写进报告"
- "这个偏差很小，不需要记入偏差表"
- "缺乏证据的地方我补充合理推测"
- "报告交叉引用太花时间，跳过"
- "上次 Agent 报告的状态我直接引用"

**扭曲过程记录 = 摧毁验现实基线。**
