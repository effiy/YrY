图: 7 项自测 → npm publish — 8 阶段发布流水线
架构洞察: 这是一个 门禁流水线 (Gated Pipeline) 模式 — 每个阶段都是后续阶段的前置条件，通过边标签 (全 PASS / 链接有效 / files[] 齐全 / 字段完整 / 复用通过 / 版本合规 / tag 就绪) 明确门禁标准。关键设计特征：NPM 扇出为两条并行验证路径（PKG 字段完整性 + CLASS 类名复用验证），两条路径在 SEMVER 处汇聚收敛，体现了 并行门禁 → 汇聚决策 的发布前多重校验策略。

概念视图

✅
7 项自测清单
入口阶段
发布流水线的第一道关卡：node --check 语法检查、3 demo 目视验证、console 无错误、硬刷新验证、mount 挂载验证、Tab 键盘导航、SR 屏幕阅读器测试。全部 PASS 才能进入下一阶段。
→ 跨页面验证
📦
npm 集成 (扇出点)
并行验证枢纽
流水线的关键分叉点：从 NPM 节点分叉为两条并行验证路径 — left: package.json 字段完整性检查 (name/version/description/files/publishConfig)，right: 类名复用验证 (.checklist/.badge 跨模块一致性)。
← 跨页面验证 → PKG 字段 · CLASS 类名
🔖
版本号与 Tag
汇聚决策
PKG 和 CLASS 两条并行路径在 SEMVER 处汇聚：验证版本号语义化合规 (v1.1.0 minor，无破坏性变更)，通过后打 git tag 附发布说明，作为 npm publish 的前置条件。
← PKG · CLASS → git tag → npm publish
🚀
npm publish 发布
出口阶段
流水线终点：灰度发布 → 稳定发布。所有前置门禁 (7 项自测 → 跨页面验证 → npm 集成 → 字段完整 → 版本合规 → tag 就绪) 均通过后执行 npm publish，将组件推送到 npm registry。
← git tag (tag 就绪)
架构完整性: 场景 5 的发布流水线展现了 线性管道 + 并行验证 + 汇聚发布 的复合拓扑：前 3 阶段为严格线性 (SELF→CROSS→NPM)，中段 NPM→(PKG||CLASS)→SEMVER 为并行扇出/扇入，后 2 阶段 (SEMVER→TAG→PUB) 再次收敛为严格线性。每条边标注门禁条件 (如 "全 PASS" / "链接有效" / "files[] 齐全")，使流水线的每个决策点都可自动验证。8 阶段 8 门禁形成完整的 CI/CD 发布前置信链。
