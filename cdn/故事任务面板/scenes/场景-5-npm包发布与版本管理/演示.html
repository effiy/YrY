💡 演示 1 · 完整发布流程

发布 v1.3.0 的完整流程：

# 1. 更新版本号
sed -i '' 's/"version": "1.2.0"/"version": "1.3.0"/' package.json
node -e "console.log(require('./package.json').version)"
# 输出: 1.3.0

# 2. 发布前检查 (6 项)
./publish-prep.sh
# 输出:
# 📦 当前版本: 1.3.0
# 📋 files 白名单预览: ... (107 组件 + shared + theme + fonts + *.md)
# 📏 包体积: 3.1MB (目标 < 5MB) ✓
# 🧪 运行测试... 全部通过
# ✅ 发布前检查全部通过

# 3. 创建 git tag 并推送
git tag v1.3.0
git push origin main --tags

# 4. 发布到 npm
npm publish
# 输出: + yry-cdn@1.3.0

💡 演示 2 · jsDelivr 同步验证

# 发布后 5 分钟内验证 jsDelivr 同步
V=1.3.0

# 验证 5 步加载链
for path in shared/index.css theme/index.css shared/index.js; do
  code=$(curl -sI "https://cdn.jsdelivr.net/npm/yry-cdn@$V/$path" \
          | head -1 | grep -oE '[0-9]{3}')
  echo "$path: HTTP $code"
done

# 期望输出:
# shared/index.css: HTTP 200
# theme/index.css: HTTP 200
# shared/index.js: HTTP 200

# 浏览器验证
open "https://cdn.jsdelivr.net/npm/yry-cdn@$V/"

💡 演示 3 · SRI 哈希生成与对比

# 生成本地 SRI 哈希
LOCAL_HASH=$(openssl dgst -sha384 -binary shared/index.js | openssl base64 -A)
echo "本地: sha384-$LOCAL_HASH"

# 拉取 CDN 上的 SRI 哈希 (通过 Subresource Integrity API)
CDN_HASH=$(curl -s "https://cdn.jsdelivr.net/npm/yry-cdn@1.3.0/shared/index.js" \
           | openssl dgst -sha384 -binary | openssl base64 -A)
echo "CDN:  sha384-$CDN_HASH"

# 对比
if [ "$LOCAL_HASH" = "$CDN_HASH" ]; then
  echo "✅ SRI 哈希一致"
else
  echo "❌ SRI 哈希不一致 (可能 CDN 尚未同步完成)"
fi

# 在 HTML 中引用 SRI
echo '<script src="https://cdn.jsdelivr.net/npm/yry-cdn@1.3.0/shared/index.js"
        integrity="sha384-'$LOCAL_HASH'"
        crossorigin="anonymous"></script>'

💡 演示 4 · 回滚流程

当 v1.3.0 发现严重 bug 需回滚到 v1.2.0：

# 1. 通知用户回滚到 v1.2.0
npm dist-tag add yry-cdn@1.2.0 latest
# npm 默认安装会拉到 v1.2.0

# 2. 保留 v1.3.0 tag 便于回溯 (不要 unpublish)
# npm unpublish 仅在发布 72h 内可用

# 3. 紧急发布修复版本 v1.3.1
sed -i '' 's/"version": "1.3.0"/"version": "1.3.1"/' package.json
# 修复 bug ... 提交 ...
git tag v1.3.1
git push origin main --tags
npm publish

# 4. 将 latest 指回 v1.3.1
npm dist-tag add yry-cdn@1.3.1 latest

# 5. 企微通知
./notify.sh "回滚完成: v1.3.0 → v1.2.0 → v1.3.1"
