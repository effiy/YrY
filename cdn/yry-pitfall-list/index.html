⚠️ YryPitfallList · 常见陷阱列表

自定义元素 <yry-pitfall-list> — 展示常见错误/陷阱/提示。 Light DOM 子元素直接放入列表，支持 high/med/low 三个严重级别和 warn/tip 变体。 每条可包含标题、正文和修复建议。

Demo — YrY 开发常见陷阱

🔥 跨分支共享管线状态 P0 · HIGH

记忆/缓存/本地状态文件禁止跨分支共享。每个 feat/<name> 分支有独立的管线状态，混用会导致 Gate 检查结果不可靠。

常见错误：在一个分支的 worktree 中运行 health 检查后，切换到另一分支复用同一份 .memory/health-trend.jsonl。

✅ 正确做法：每次切分支后运行 node skills/rui-bot/send.mjs health --diff 重新生成基线，确保状态隔离。
⚡ 魔法数字硬编码 P1 · MED

健康评分阈值（如 60、80）直接写死在函数体内，后续调整需要多处修改。违反「禁止魔法数字」铁律。

✅ 正确做法：所有阈值统一定义在 lib/constants.mjs 中，各脚本通过 import { HEALTH_THRESHOLD } from '../lib/constants.mjs' 引用。
💡 过度使用 Workflow 工具 P2 · TIP

Workflow 工具适合大规模并行审计/迁移场景。简单的 2-3 文件修改直接用 Agent 工具即可，Workflow 的隔离开销（worktree setup ~500ms/agent）在小任务上得不偿失。

💡 建议：任务涉及 <5 个文件时使用 Agent，5-20 个文件使用 parallel Agent，20+ 文件或需要隔离时才使用 Workflow。
🐛 未验证就声称修复 P1 · MED

修复代码后未运行验证命令就声称"已完成"，违反「验先于称」铁律。\"上次通过了\"、\"应该没问题\"都是违反信号。

✅ 正确做法：每个修复后执行验证五步法：①运行相关测试 ②运行 arch-check ③运行 health --short ④确认输出符合预期 ⑤记录验证结果。

📋 使用方式

使用 <yry-pitfall-list> 包裹多个 .pitfall 子元素。 每个 pitfall 支持变体类名 .warn（橙色左边框）和 .tip（青色左边框），默认红色。 严重度标签用 .pf-sev.s-high / .s-med / .s-low。
